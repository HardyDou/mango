# 后端开发流程

## 1. 适用范围

- 本文件只约束后端开发。
- 通用流程以 [00-dev-flow.md](../00-dev-flow.md) 为准。

## 2. 开发前

必须明确：

- 目标
- 范围
- 验收标准
- 影响模块
- 接口变化
- 数据变化
- 测试范围

必须先拆解需求并提取：

- 业务对象：聚合根、实体、值对象、关联对象和跨模块引用。
- 字典和枚举：状态、类型、来源、级别、动作、开关等有限选项。
- 数据模型：表、字段、索引、唯一约束、审计字段、租户字段和历史数据影响。
- 接口模型：Command、Query、VO、分页结果、批量操作入参和错误返回。
- 业务规则：状态流转、权限边界、数据权限、幂等、并发、校验和异常分支。
- 辅助数据：组织、用户、角色、租户、部门、分类、标签、上级对象等依赖数据。
- 设计模式：策略、模板方法、工厂、状态机、规格校验、事件发布等是否有真实复用价值。
- 复用能力：已有 API、service、provider、checker、validator、converter、mapper 和测试物料。

拆解结果必须决定：

- 哪些模型进入 `api` 契约，哪些模型只留在 `core`。
- 哪些能力放在当前模块，哪些能力通过其它模块 API 调用。
- 哪些字典和枚举需要后端统一定义，哪些只属于页面筛选条件。
- 哪些设计模式能降低复杂度，哪些只是无意义包装。
- 哪些规则需要测试覆盖，哪些规则需要集成验证。

## 3. 开发中

- 只改本次需求相关代码。
- 新增代码必须遵守最新规范。
- 不把 `PO`、`Entity` 直接暴露到 API。
- 不新增跨模块错误依赖。
- 不在本次任务中顺手扩大改动范围。

## 4. 数据库变更

- DDL 变更必须使用 Flyway migration。
- 禁止直接改线上库。
- 每次变更新增 migration 文件。
- 不修改已执行的历史 migration。
- migration 路径使用 `db/migration/{module}/V{version}__{description}.sql`。
- migration 按模块隔离。

## 5. 提交前验证

至少执行与改动范围对应的检查：

- `mvn test`
- `mvn verify`
- `mvn pmd:check`
- `mvn checkstyle:check`
- `mvn mango:check`

## 6. 提交要求

提交说明必须写清：

- 改动范围
- 验证结果
- 未完成项
- 风险

## 7. 禁止事项

- 没有设计就直接改跨模块边界
- 没有完成需求拆解就直接建表、写接口或写 service
- 没有验证就提交
- 直接修改生产数据结构
- 用一次性脚本替代正式 migration
