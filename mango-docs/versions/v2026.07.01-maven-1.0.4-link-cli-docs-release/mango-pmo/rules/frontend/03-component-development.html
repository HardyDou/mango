前端组件开发规范 ​

1. 适用范围 ​

  • 适用于 Mango 前端公共组件、业务组件、页面内可复用组件。
  • 适用于 Mango 单体部署、微前端部署和基于业务 starter 创建的业务项目。
  • 组件开发必须同时遵守 Vue 代码规范、Element Plus UI 规范和前端 Monorepo 架构规范。

2. Mango 特性 ​

Mango 是业务开发脚手架，组件会被发布到私有或公开 npm 仓库，并被其它业务项目作为独立组件消费。组件必须支持：

  • 单体部署：组件可被宿主 app 直接注册和使用。
  • 微前端部署：组件可在独立业务包中运行，不依赖宿主私有实现。
  • npm 独立消费：组件包必须在脱离 Mango 当前仓库源码后仍可被业务项目安装、构建、类型检查和运行。
  • 业务复用：组件通过稳定 props、emits、slots、类型、样式变量和文档对外提供能力。

3. 组件拆分 ​

开发组件前必须先拆解：

  • 组件职责：展示、输入、选择、表格、状态、操作、布局或业务聚合。
  • 依赖数据：字典、枚举、辅助数据、远程接口、上下文和权限。
  • 输入输出：props、emits、v-model、slots、expose 方法和错误状态。
  • 部署边界：是否会在单体和微前端同时使用。
  • 发布边界：作为 npm 包独立消费时的公共 API、依赖、样式和资源边界。
  • 复用层级：页面私有、业务包复用、Mango 公共复用。
  • 独立消费：业务项目只通过 npm 包、公开类型和文档使用组件时是否完整可用。

拆解结果必须决定组件放置位置：

  • 页面私有组件放当前页面目录。
  • 业务域复用组件放对应业务 packages/*。
  • 跨业务复用组件放公共包。
  • 能独立发布的组件必须从一开始按包边界设计。

4. 包边界 ​

  • 组件包禁止依赖 apps/*。
  • 公共组件禁止依赖宿主 store、router、菜单、权限装配、启动逻辑。
  • 业务组件需要接口时，必须依赖对应 API 包或通过 props/callback 注入。
  • 组件包不得直接读取宿主环境变量、全局对象或运行时菜单。
  • 微前端和单体共用能力必须放在 packages/*，禁止通过宿主 app 私有文件中转。
  • 组件包导出必须稳定，新增公共组件必须在包入口统一导出。

5. API 设计 ​

  • 组件对外 API 必须显式声明 TypeScript 类型。
  • props 只表达组件输入，不承载隐式全局配置。
  • emits 只表达组件事件，不直接触发宿主业务流程。
  • 支持双向绑定时使用标准 v-model 命名。
  • slots 必须有明确语义，禁止用一个万能 slot 承载多种不相关结构。
  • 组件错误、加载、空状态必须是可控状态，不能只靠内部临时变量。
  • ID、业务主键和后端 Long 字段按前端 ID 规则使用字符串语义。

6. 样式与主题 ​

  • 组件样式必须可被宿主主题覆盖。
  • 优先使用 Element Plus 变量、Mango 设计变量和 CSS 变量。
  • 禁止在组件内硬编码品牌色、状态色、页面宽度和宿主布局尺寸。
  • 组件 class 必须有稳定前缀，避免污染宿主或其它微前端。
  • 组件不得覆盖 Element Plus 全局 class 作为自身实现。
  • 组件必须在不同容器宽度下保持可用，不能假设固定页面宽度。
  • 样式必须跟随组件或页面所属 package 发布，package 应提供唯一的 ./style.css 样式入口。
  • admin 类宿主或业务管理应用可以通过聚合入口引入各 package 的 style.css，但聚合入口不能成为样式的唯一来源。
  • admin 类宿主或业务管理应用的 package 样式聚合必须由本应用 package 声明文件生成，业务开发者只维护声明，不手工追加 @import 清单。
  • admin 类宿主或业务管理应用的 dev 和 build 命令必须自动执行样式聚合生成；手动生成命令只作为调试、修复和 CI 差异定位入口。
  • package 样式聚合必须通过通用生成工具校验消费方依赖声明和 package ./style.css 公开导出，必须同时支持 workspace package 和已发布到 npm/Nexus 后安装到 node_modules 的 package。
  • 官方业务模块接入 admin/full 或 CLI 可选模块时，必须同时接入页面注册、package 依赖、样式聚合、CLI 模块清单和样式治理检查；禁止只在 apps/* 私有入口手工 import 样式兜底。
  • 微前端 app 必须显式引入自身运行页面依赖的 package 样式，禁止依赖宿主或 shell 的聚合样式穿透隔离容器。
  • 业务项目按需消费 package 时，必须同时按 package 公开入口引入对应样式，禁止复制 Mango 页面样式或依赖仓库源码路径。
  • 开发中心、组件示例和文档示例页只能维护示例页布局样式；组件运行所需样式必须归属组件自身 package。
  • 组件示例必须通过组件公开 API 和组件 package 的 style.css 展示效果，禁止要求业务项目额外依赖开发中心或示例中心 package 才能获得组件样式。
  • 示例页使用其它 package 组件时，运行环境必须引入被展示组件 package 的 style.css；示例页自身样式只允许覆盖文档排版、示例容器和代码块。

7. 资源与副作用 ​

  • 组件不得在 import 阶段发起请求、注册全局事件、修改全局样式或读取运行时上下文。
  • 组件创建的定时器、事件监听、观察器和外部实例必须在卸载时释放。
  • 组件不得把数据持久化到宿主 store；需要缓存时通过参数或回调交给宿主决定。
  • 组件不得直接操作全局路由跳转；需要导航时通过事件通知宿主。
  • 文件预览、下载、上传必须遵守 Mango 文件 ID 规则，禁止把访问地址作为业务数据。

8. 发布要求 ​

组件包必须按发布到私有或公开 npm 仓库的标准开发，并满足：

  • package.json 声明清晰的 name、version、type、exports、types 和 peerDependencies。
  • 发布页面或组件样式的 package 必须在 exports 中声明 ./style.css，并确保构建产物包含该文件。
  • Vue、Element Plus、Mango 公共包应作为 peer 或外部依赖处理，不打进组件包私有副本。
  • 包入口只导出公共 API，不导出页面私有文件。
  • 构建产物包含 ESM 和类型声明。
  • 变更必须有版本策略和迁移说明。
  • 发布前必须验证包在单体宿主和微前端业务包中都能使用。
  • 发布影响变更必须执行 pnpm -C mango-ui release:impact --base=origin/main --head=HEAD，确保受影响 npm 包、固定版本依赖、CLI release lock 和 starter 消费版本同步。
  • 发布后必须执行 pnpm -C mango-ui release:verify-npm <package> --version=<version> 回查公司内网 Nexus tarball；关键产物应登记到 mango-ui/release-contracts.json，防止源码已改但发布包缺文件。

8.1 独立消费要求 ​

组件作为 npm 包被独立使用时必须满足：

  • 安装可用：业务项目只安装 npm 包和声明的 peer 依赖即可使用，不需要复制 Mango 仓库源码。
  • 构建可用：不得依赖仓库内未导出的别名、相对路径、构建脚本或 Vite 私有配置。
  • 类型可用：所有公开 props、emits、slots、expose、工具类型必须从包入口或明确子路径导出。
  • 样式可用：组件样式、主题变量和必要资源必须随包发布或有明确引入路径。
  • 运行可靠：缺少可选 provider、远程数据失败、空数据、权限不足时必须有明确降级或错误状态。
  • 版本可靠：破坏性变更必须升级主版本或提供迁移说明，禁止静默改变公共 API 语义。
  • 环境可靠：不得假设宿主一定存在 Mango admin shell、特定 router、特定 store、特定全局变量或特定部署 base path。
  • 依赖可靠：不得把 Vue、Element Plus 打包成组件私有副本；不得依赖未声明的 transitive dependency。
  • 资源可靠：图标、样式、worker、wasm、字体等资源必须通过包内路径或明确 peer 方案提供。
  • 安全可靠：不得在组件内保存业务 token、租户上下文或文件访问地址；敏感上下文只能由宿主显式传入。

8.2 发布账号 ​

  • 发布到 Mango Nexus npm hosted 仓库时，必须复用 Maven/Nexus 发布账号。
  • npm 发布凭据只能配置在发布机用户级 npm 配置、CI secret 或临时环境变量中。
  • 禁止把 npm token、用户名、密码或 Maven server 密码写入仓库 .npmrc、package.json、脚本或文档示例。
  • Maven/Nexus 账号轮换时，npm 发布凭据必须同步轮换。

9. 示例与文档 ​

  • 公共组件必须提供示例和 API 文档。
  • 示例必须覆盖基础用法、禁用或只读、加载、空状态、错误状态、远程数据和业务集成。
  • 示例不得依赖宿主私有接口；需要数据时使用可替换的示例 provider。
  • 文档必须说明适用场景、不适用场景、props、emits、slots、类型和样式变量。

10. 测试与检查 ​

  • 公共组件必须有组件测试或等价验证。
  • 可交互组件必须覆盖输入、输出、事件、加载、空状态和错误状态。
  • 支持远程数据的组件必须覆盖请求成功、失败、空结果和检索。
  • npm 包组件必须执行包构建和类型检查。
  • 单体和微前端共用组件必须至少验证两种部署形态的引用路径和样式生效结果。
  • 微前端样式验证必须检查隔离容器内目标元素的 computed style，不能只检查宿主 document 是否存在 CSS 文本。
  • npm 包组件必须用独立消费示例验证安装、构建、类型检查、样式引入、空状态和错误状态。
  • 官方 admin 模块集成变更必须执行 pnpm admin:styles:check 和 pnpm admin:module-styles:check。
  • 模板检查必须确保业务 starter baseline 携带本规范。

11. 禁止事项 ​

  • 禁止组件直接依赖宿主 app 私有 store、router、菜单、权限或启动逻辑。
  • 禁止为单体部署写死路径、服务名、baseUrl 或运行时上下文。
  • 禁止复制组件到多个业务包后长期分别维护。
  • 禁止发布包含业务项目私有接口、私有样式或私有环境变量的公共组件。
  • 禁止发布后仍要求消费方通过源码相对路径、仓库别名或宿主私有配置才能使用。
  • 禁止为了短期页面交付破坏组件包边界。
  • 禁止用宿主全局样式或 shell 样式兜底修复 package 内页面样式。
  • 禁止把组件运行样式放在开发中心、组件示例或文档示例页中。
  • 禁止把通用组件样式集中塞进开发中心 package，导致业务项目为了使用单个组件被迫依赖开发中心。
  • 禁止手工维护 admin 类宿主或业务管理应用的 package 样式聚合清单而绕过生成校验。
