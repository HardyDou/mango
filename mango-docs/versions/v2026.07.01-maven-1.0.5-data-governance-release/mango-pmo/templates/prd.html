<需求名称> PRD ​

适用规范：mango-pmo/rules/product/01-prd-template.md。 首先 For AI：让 AI 可理解、可开发、可验收。 其次 For Human：帮助人把需求写成 AI 能理解的无歧义输入，降低 AI 落地偏差。 本文只写业务需求和页面原型。禁止写接口、权限码、数据库、路由、component key、模块归属、初始化方式、测试命令和交付台账。

0. 编写前检查 ​

0.1 AI 动作判定 ​

动作使用条件当前结论依据
STOP/ASK/WRITE/NEXT阻断则 STOP；需业务确认则 ASK；可生成正文则 WRITE；自检全 PASS 且可进设计则 NEXT

0.2 待确认问题 ​

顺序问题类型待确认问题影响范围是否阻断AI动作(STOP/ASK/WRITE)用户回答后落点
1目标用户/关键对象/生命周期/关键流程/业务规则/菜单页面/验收标准/不处理范围是/否

0.3 输出顺序 ​

存在阻断项时，只输出编写前检查和待确认问题；无阻断项时再输出 PRD 正文、自检和进入详细设计判定。

0.4 编号规则 ​

  • 关键对象：&lt;BO-ID&gt;，必须替换为真实稳定编号。
  • 关键业务流程：&lt;BF-ID&gt;，必须替换为真实稳定编号。
  • 业务规则：&lt;BR-ID&gt;，必须替换为真实稳定编号。
  • 页面：&lt;PG-ID&gt;，必须替换为真实稳定编号。
  • 验收项：&lt;AC-ID&gt;，必须替换为真实稳定编号。

16. AI 输出自检 ​

自检结果只能填写 PASS、FAIL、BLOCKED 或 EXCEPTION。任一 FAIL/BLOCKED 时，最终动作只能是 STOP/ASK；EXCEPTION 必须引用用户确认来源；全部 PASS 且进入详细设计判定为 READY 时，最终动作才可为 NEXT。

检查项结果说明
每个关键对象是否至少关联一个流程、页面或验收项
每个关键流程是否包含规则、状态变化、异常限制和验收标准
每条业务规则是否至少被一个流程、页面或验收项引用
每个页面按钮是否能追溯到关键流程或业务规则
每个验收项是否有明确用户操作和可见结果
是否覆盖正常态、空态、加载态、失败态、不可操作态
PRD 中是否没有接口、权限码、数据库、路由、组件实现和测试命令
