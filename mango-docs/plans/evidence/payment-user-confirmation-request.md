# 支付中心用户确认单

## 1. 目的

本确认单用于关闭设计文档第 21 节和交付台账中的用户确认项。确认前，Agent 不得自行假设线下转账、外部通道清单、开票触发或租户定义。

## 2. 确认范围

| 台账 ID | 设计来源 | 待确认问题 | 推荐确认口径 | 原因 | 确认后动作 |
|---|---|---|---|---|---|
| PAY-CONFIRM-001 | 设计文档 21、Sprint Q-001、用户最新确认 | 线下收款通道范围确认 | 用户已确认线下收款作为独立完整支付通道，新增“线下收款”菜单，具备确认到账、支付凭证、退款处理、银行流水 Excel 导入和对账匹配能力 | 线下收款是支付通道，不是支付订单列表人工确认；需要独立通道能力、数据模型、权限、审计和 E2E | 更新设计和台账，关闭确认项，并新增 `PAY-OFFLINE-001` 至 `PAY-OFFLINE-007` 实现项 |
| PAY-CONFIRM-002 / PAY-CHANNEL-006 | 设计文档 21、Sprint Q-002 | 除芒果支付、通联、华夏银行外，微信、支付宝、连连等通道清单和优先级 | 本期已实现芒果支付内置虚拟通道；真实外部通道按资料到位顺序推进，通联、华夏、微信、支付宝、连连均不伪实现 | 当前缺各机构商户参数、证书、回调、账单和联调资料；不能用种子配置或芒果支付替代真实通道 | 若确认推荐口径，通道清单项记录为用户确认的阻塞/后续范围；资料到位后逐通道研发 |
| PAY-CONFIRM-003 | 设计文档 21、Sprint Q-003 | 支付成功后是否通知业务系统触发开票 | 支付系统只发送支付结果通知，不直接处理开票；开票由业务系统收到支付成功通知后自行触发 | 支付系统边界是收单、支付结果通知、退款、对账和结算；开票属于业务财税系统，不应耦合进支付模块 | 若确认推荐口径，通知协议不新增开票字段；如需开票字段，另行定义接口契约和业务联调 |
| PAY-CONFIRM-004 | 设计文档 12、21、Sprint Q-004 | 租户是公司内部事业部、外部客户，还是两者都支持 | 保持通用租户模型，两者都支持；权限、菜单、应用、主体、订单、报表继续按租户隔离 | 当前代码和验收已按通用租户隔离建设，过早收窄为单一租户定义会破坏平台复用边界 | 若确认推荐口径，更新设计为通用租户模型；如只支持内部或外部，需要重新评估权限、菜单授权和数据隔离 |

## 3. 可直接回复模板

请按下面格式回复，未确认项可以保留“暂不确认”。

```text
确认 PAY-CONFIRM-001：
线下收款作为独立完整支付通道，新增“线下收款”菜单；收银台对公转账通过该通道生成随机对账码和转账物料，用户可上传支付凭证；财务在“线下收款”菜单确认到账、导入银行流水 Excel 自动匹配并处理线下退款。

确认 PAY-CONFIRM-002 / PAY-CHANNEL-006：
本期已实现芒果支付内置虚拟通道；真实外部通道按资料到位顺序推进，通联、华夏、微信、支付宝、连连均不伪实现。

确认 PAY-CONFIRM-003：
支付系统只发送支付结果通知，不直接处理开票；开票由业务系统收到支付成功通知后自行触发。

确认 PAY-CONFIRM-004：
保持通用租户模型，内部事业部和外部客户两者都支持；权限、菜单、应用、主体、订单、报表继续按租户隔离。
```

## 4. 用户未按推荐口径确认时的处理

| 场景 | 处理 |
|---|---|
| 要求线下收款接入银行实时入账通知 | 另行补充银行入账通知来源、验签规则、匹配规则、异常处理、数据模型、接口和 E2E；当前已确认范围优先交付银行流水 Excel 导入匹配和财务确认到账 |
| 要求本期接入某真实外部通道 | 等待该通道商户参数、证书、公钥私钥、回调地址、账单样例和联调账号，然后按通道适配层实现 |
| 要求支付通知携带开票触发字段 | 补充通知协议字段、兼容性说明、业务系统联调和 E2E，不把开票逻辑写入支付核心 |
| 要求只支持单一租户定义 | 重新评估权限、菜单授权、数据隔离、默认角色和历史数据影响 |
