# 系统管理菜单规划

更新时间：2026-05-09

## 核心原则

系统管理不按技术模块堆菜单，按管理员的工作对象分组。用户可见术语统一使用“机构”；技术字段仍可沿用 `tenant/tenantId`。机构是数据隔离主体，组织是机构内部结构，角色是权限模型，三者不能混放。

四个概念必须分开：

| 概念 | 解决的问题 | 是否机构隔离 | 不能混用的点 |
|---|---|---:|---|
| 机构 | 数据隔离边界，例如芒果集团、A 公司、B 公司 | 是 | 机构不是组织节点，普通机构不能新增其它机构 |
| 组织 | 机构内部的集团、公司、部门、小组结构 | 是 | 组织不决定接口权限，只作为成员归属和业务范围 |
| 成员 | 机构内可登录账号关系 | 账号可复用，成员关系按机构隔离 | 成员不等于组织节点，权限来自角色绑定 |
| 角色 | 权限集合与授权载体 | 是 | 角色只绑定菜单/API 权限，不承载组织层级 |

因此“成员 + 组织”不能合并成“权限”。正确关系是：账号在某个机构、某个登录上下文下成为机构成员，并绑定机构内角色；成员可再关联组织/岗位作为归属和数据范围。权限系统只使用角色判断能否访问接口和菜单。

平台运营方和普通机构管理员的权限边界必须分开：

| 能力 | 平台管理员 | 机构管理员 | 说明 |
|---|---:|---:|---|
| 新增/停用机构 | 是 | 否 | 机构管理员不能创建其它机构 |
| 应用定义 | 是 | 否 | 应用是平台元数据 |
| 菜单定义 | 是 | 否 | 菜单是平台元数据 |
| 基础字典 | 是 | 默认否 | 字典先作为平台基础数据，后续如需机构自定义另建覆盖模型 |
| 参数配置 | 是 | 视参数范围开放 | 系统级参数归平台，机构级参数后续单独建模 |
| 角色管理 | 是 | 是 | 角色数据按机构隔离，机构管理员只能管理本机构角色 |
| 成员管理 | 是 | 是 | 成员是账号在机构内的身份关系，机构内分配角色/组织/岗位 |
| 组织架构 | 否 | 是 | 组织是机构内结构 |
| 岗位管理 | 否 | 是 | 岗位是机构内组织人事数据 |
| 审计日志 | 是 | 是 | 平台可看全局，机构只能看本机构 |

## 权限落库策略

平台管理员，也就是机构 `1` 的 `ROLE_ADMIN`，可以拥有完整系统管理菜单。

A 公司、B 公司、C 公司等普通机构的 `ROLE_ADMIN`，默认只能拥有：

- 系统管理
- 账号权限
- 成员管理
- 角色管理
- 角色查询、新增、修改、删除、授权按钮权限
- 组织人事
- 组织架构
- 岗位管理
- 审计日志
- 登录日志
- 操作日志
- 登录日志与操作日志查询权限

普通机构默认不能拥有：

- 机构管理
- 新增/修改/删除/停用机构权限
- 应用管理
- 菜单管理
- 菜单新增/修改/删除权限
- 字典管理写权限
- 参数配置写权限
- `*:*`

## 当前待补齐项

- 保持菜单规划与权限数据同步：新增非按钮级菜单必须配置图标，避免顶部和左侧导航出现空图标。
- 普通机构侧只开放账号权限、组织人事、审计日志；平台运营和基础数据维护入口默认不开放。
- 字典、参数、行政区划属于平台基础数据，普通机构默认不开放维护入口；业务选择器可开放只读读取接口。
- 后续新增业务系统菜单时，按“系统入口 / 业务工作台 / 配置管理 / 审计查询”分组，不把业务角色硬塞进系统管理。
- 参考 Pigx 菜单组合方式重新优化 Mango 后台菜单信息架构。重点不是复制菜单名，而是吸收它按平台能力、系统配置、流程任务、业务管理分层组织的方式，解决当前菜单组合不够清晰的问题。
- 参考 Pigx 租户新增页的“套餐”交互，把套餐建模为可维护的菜单授权集合。新增/编辑机构时选择套餐并可查看套餐菜单树；不要把套餐做成单纯字典值，否则无法表达菜单范围、按钮权限、授权变更和套餐版本。
- 路由管理与菜单管理职责重复，系统入口和动态路由统一由“菜单管理”维护；不再保留路由管理页面、直达入口和维护接口。

## Pigx 菜单参考后的信息架构建议

Mango 后续顶部/一级能力建议逐步收敛为：

- 系统管理：账号权限、组织人事、平台运营、基础数据、审计日志。
- 业务平台：保函业务、支付/订单等真实业务模块，必须有页面和接口后再落菜单。
- 基础工具：文件管理、消息通知、数据字典只读选择器等通用工具能力。
- 协同办公：任务管理、发起流程、流程管理、业务表单。
- AI 大模型：知识库、AI 问答、AI 办公、AI 视觉等后续 AI 能力。
- 开发平台：代码生成、数据源、元数据等平台研发工具。

协同办公首批菜单建议：

- 任务管理：待办任务、我的发起、抄送给我、我的已办。
- 发起流程。
- 流程管理：流程定义、流程分组。
- 业务表单。

工作流表单设计器组件分组已落地：

- 辅助组件：提示、文字、HTML、分割线、按钮、标签、图片展示。
- 子表单组件：对象容器、子表单、表格子表单。
- 基础组件：输入框、选择器、日期、上传等 form-create 标准控件。
- 业务组件：人员、部门、岗位、角色、上传、图片、地区、签名、字典、流水号。业务组件输出标准 form-create 规则，后续再按真实组织/岗位/成员/角色 API 扩展数据联动。

落地原则：只对已经有真实页面、真实 API、真实 E2E 的菜单落库授权；规划中的菜单先写计划，不提前造不可用入口。

## 结论

系统管理菜单按“账号权限 / 组织人事 / 平台运营 / 基础数据 / 审计日志”组织。机构与组织不放在一起；成员与组织也不合并到权限里。机构管理员不能新增机构，平台元数据只由平台管理员维护。
