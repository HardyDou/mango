# 通知中心消息入口与接收设置需求说明

## 1. 目标

优化管理后台通知中心的用户消息体验和接收控制能力。

本需求要解决：

- 右上角消息提示入口不友好。
- 系统消息没有完整消息中心页面。
- 实时推送链路没有真正接入管理后台。
- 接收设置只是一组简单开关，不能表达用户对不同业务域、不同消息、不同渠道的接收偏好。
- 发送链路没有根据用户接收偏好和接收账户进行完整判断。

最终目标：

> 用户可以配置自己是否接收某个业务域、某个消息，以及通过哪些渠道接收；系统发送时根据消息定义、用户偏好和接收账户决定是否发送、发到哪里、为什么不发。

## 2. 范围

### 2.1 本次范围

- 管理后台右上角消息提示入口优化。
- 新增或完善用户侧消息中心页面。
- 接收设置页面重构。
- 用户接收账户管理能力设计。
- 发送链路接入接收设置判断。
- 系统消息实时推送接入 `mango-infra-realtime`。
- 未读数、消息弹窗、消息详情、已读状态联动。
- 消息级别定义和展示规则。

### 2.2 不做范围

- 不在 `mango-infra-realtime` 中持久化业务消息。
- 不把接收设置做成系统配置页。
- 不把发送系统消息表单放在消息中心里。
- 第一版不实现微信、钉钉、企微、飞书的完整 OAuth 或绑定认证流程，只预留账户类型和发送判断字段。
- 不用 Mock 或前端临时事件替代真实接口和真实 realtime 链路。

## 3. 角色与使用场景

### 3.1 普通管理后台用户

- 查看自己的系统消息。
- 配置自己接收哪些业务消息。
- 配置短信、邮件、微信、钉钉等接收账户。
- 对某类消息选择接收或不接收。
- 对某类消息选择通过哪些渠道接收。

### 3.2 管理员

- 查看用户接收设置。
- 维护消息定义是否允许用户关闭。
- 维护消息级别。
- 排查消息没有发送的原因。

### 3.3 通知发送链路

- 根据消息定义、渠道模板、通知渠道、用户偏好、接收账户，决定生成哪些发送记录。
- 对被用户设置拦截或缺少接收账户的渠道，记录明确原因。

## 4. 概念定义

## 4.1 消息级别

消息级别表达消息本身的重要程度，不表示渠道优先级。

枚举：

| 值 | 中文名 | 说明 |
|---|---|---|
| NORMAL | 普通 | 常规业务通知 |
| IMPORTANT | 重要 | 需要用户尽快关注 |
| URGENT | 紧急 | 安全、风控、验证码、失败告警等强提示消息 |

消息级别影响：

- 消息中心排序。
- 小铃铛预览优先展示。
- toast、桌面通知、TTS 等提示强度。
- 是否建议允许用户关闭。

渠道顺序和通道选择不使用“优先级”字段表达，应使用“通道路由”或“渠道策略”表达。

## 4.2 接收渠道

第一版支持：

- 系统消息
- 短信
- 邮件

预留：

- 微信公众号
- 企业微信
- 钉钉
- 飞书
- Webhook

## 4.3 接收账户

接收账户是用户用于接收外部渠道消息的地址或平台身份。

| 渠道 | 接收账户 |
|---|---|
| 系统消息 | 系统用户 ID，不需要额外账户 |
| 短信 | 手机号 |
| 邮件 | 邮箱 |
| 微信公众号 | openId / unionId |
| 企业微信 | 企业微信用户 ID |
| 钉钉 | 钉钉用户 ID |
| 飞书 | 飞书用户 ID |

账户状态：

| 状态 | 说明 |
|---|---|
| UNBOUND | 未绑定 |
| PENDING_VERIFY | 待验证 |
| VERIFIED | 已验证 |
| DISABLED | 已禁用 |

发送时只使用 `VERIFIED` 状态账户。

系统消息不需要外部账户，默认使用当前系统用户身份。

## 5. 菜单与页面规划

通知中心菜单保持后台管理职责清晰：

```text
通知中心
├── 消息配置
├── 渠道配置
├── 接收设置
├── 发送任务
├── 发送记录
├── 失败重试
└── 消息中心
```

说明：

- `消息配置`：配置业务消息、参数、消息类型、版本。
- `渠道配置`：配置系统可用通道。
- `接收设置`：配置用户对业务消息和渠道的接收偏好。
- `发送任务`：管理端人工触发发送任务。
- `发送记录`：查看每个渠道、每个接收人的发送明细。
- `失败重试`：处理失败发送记录。
- `消息中心`：用户查看自己收到的系统消息。

右上角小铃铛是 `消息中心` 的快捷入口，不替代完整页面。

## 6. 右上角消息提示入口

## 6.1 入口要求

- 使用无背景小铃铛图标，和顶部其它图标风格一致。
- 显示未读角标。
- 未读数超过 99 显示 `99+`。
- 点击后显示轻量消息面板。

## 6.2 消息面板

面板内容：

- 标题：`我的消息`
- 最新未读消息列表，默认最多 5 条。
- 每条显示：
  - 消息标题
  - 业务域
  - 消息级别
  - 接收时间
- 操作：
  - 全部已读
  - 查看全部
  - 接收设置

交互：

- 点击消息项打开消息详情。
- 点击 `查看全部` 进入消息中心。
- 点击 `接收设置` 进入接收设置页面。
- 全部已读后刷新角标和列表。

## 6.3 新消息提示

收到实时消息后：

1. 刷新未读数。
2. 拉取消息详情。
3. 根据用户设置和消息级别决定提示方式。

提示方式：

| 消息级别 | 默认提示 |
|---|---|
| 普通 | 刷新角标 |
| 重要 | 刷新角标 + 站内 toast |
| 紧急 | 刷新角标 + 站内 toast + 桌面通知，用户开启 TTS 时播报 |

桌面通知和 TTS 必须尊重用户接收设置。

## 7. 消息中心页面

## 7.1 页面定位

消息中心用于用户查看自己收到的系统消息。

消息中心不是发送页面，不提供手工发送表单。

## 7.2 列表字段

| 字段 | 说明 |
|---|---|
| 标题 | 消息标题 |
| 业务域 | 消息归属业务域 |
| 消息类型 | 消息定义名称 |
| 消息级别 | 普通、重要、紧急 |
| 状态 | 未读、已读 |
| 接收时间 | 系统消息落库时间 |

不在列表显示完整内容、发送参数 JSON、发送记录明细。

## 7.3 查询条件

- 关键字。
- 业务域。
- 消息类型。
- 消息级别。
- 是否未读。
- 接收时间范围。

## 7.4 页面功能

- 查看详情。
- 标记单条已读。
- 批量已读。
- 全部已读。
- 删除。

## 7.5 详情

详情展示：

- 标题。
- 内容。
- 业务域。
- 消息类型。
- 消息级别。
- 业务对象。
- 接收时间。
- 已读时间。

详情为只读。

## 8. 接收设置页面

## 8.1 页面定位

接收设置用于配置用户对不同业务消息、不同渠道、不同接收账户的接收偏好。

接收设置不维护系统级重试次数、消息保留天数、渠道运行参数。

## 8.2 页面结构

推荐布局：

```text
顶部：用户选择 / 当前用户

左侧：业务域列表
  基础
  保函
  订单
  财务
  风控
  营销
  系统

右侧：消息接收配置表
  消息名称 | 消息级别 | 是否接收 | 系统消息 | 短信 | 邮件 | 微信 | 钉钉 | 操作

底部或右侧抽屉：接收账户管理
```

## 8.3 用户选择

普通用户：

- 默认只能配置自己。

管理员：

- 可以选择用户查看或维护该用户接收设置。

## 8.4 业务域维度

用户可以配置某个业务域是否接收。

示例：

- 接收基础类消息。
- 不接收营销类消息。
- 接收保函类消息，但只接收系统消息和邮件。

## 8.5 单消息维度

用户可以配置具体消息是否接收。

示例：

- 接收 `登录提醒`。
- 不接收 `营销活动提醒`。
- 接收 `出函成功`，但只开启系统消息和邮件。

如果消息定义中 `是否允许用户关闭=false`：

- 是否接收开关禁用。
- 渠道关闭行为按消息定义强制策略执行。
- 页面展示原因：`该消息为强制接收消息`。

## 8.6 渠道维度

每个消息可配置渠道接收开关。

| 渠道 | 控件 | 账户要求 |
|---|---|---|
| 系统消息 | 开关 | 不需要外部账户 |
| 短信 | 开关 + 手机号选择 | 必须有已验证手机号 |
| 邮件 | 开关 + 邮箱选择 | 必须有已验证邮箱 |
| 微信公众号 | 开关 + 绑定状态 | 必须已绑定微信 |
| 钉钉 | 开关 + 绑定状态 | 必须已绑定钉钉 |

如果没有可用账户：

- 渠道开关禁用。
- 显示 `未绑定手机号`、`未绑定邮箱` 等提示。
- 提供 `去绑定` 操作。

## 8.7 接收账户管理

账户列表字段：

| 字段 | 说明 |
|---|---|
| 账户类型 | 手机号、邮箱、微信、钉钉等 |
| 账户标识 | 手机号、邮箱、平台 ID |
| 显示名称 | 用户可识别名称 |
| 是否默认 | 同类型默认账户 |
| 状态 | 未绑定、待验证、已验证、已禁用 |
| 更新时间 | 最后更新时间 |

账户功能：

- 新增手机号。
- 新增邮箱。
- 设置默认账户。
- 禁用账户。
- 删除未使用账户。

第一版：

- 手机号和邮箱可以从身份用户资料初始化。
- 手机号和邮箱支持用户新增。
- 验证流程可以先预留状态，不强制实现验证码校验。

## 9. 接收设置生效规则

## 9.1 优先级规则

接收设置按以下顺序生效：

```text
单消息设置 > 业务域设置 > 全局渠道设置 > 系统默认
```

说明：

- 单消息配置最具体，优先级最高。
- 业务域配置影响该业务域下所有消息。
- 全局渠道配置影响用户是否默认接收某渠道。
- 没有任何用户配置时，按系统默认接收。

## 9.2 强制消息规则

消息定义支持字段：

- `allowUserDisable`：是否允许用户关闭。
- `forceChannels`：强制渠道。

规则：

- `allowUserDisable=false` 时，用户不能关闭该消息。
- 强制渠道不受用户关闭影响。
- 强制渠道仍需要满足渠道启用、通道可用、接收账户存在且已验证。

## 9.3 系统消息兜底

系统消息是最低接收渠道。

规则：

- 如果消息定义启用了系统消息，优先按系统消息发送。
- 如果业务要求“最低发送系统消息”，则系统消息不需要外部账户。
- 如果用户关闭系统消息但消息为强制消息，仍发送系统消息。
- 如果消息允许关闭且用户关闭系统消息，则不发送系统消息。

## 10. 发送链路接入判断

发送前判断顺序：

1. 消息定义是否存在。
2. 消息定义版本是否生效。
3. 消息定义是否启用。
4. 消息类型渠道模板是否启用。
5. 通知渠道是否启用。
6. 用户接收设置是否允许接收该消息。
7. 用户接收设置是否允许该渠道。
8. 渠道对应接收账户是否存在。
9. 接收账户是否已验证且可用。
10. 通道路由是否有可用通道。

通过后生成发送记录并发送。

未通过时生成取消记录，用于排查原因。

## 11. 发送记录状态与原因

被接收设置或账户拦截时，发送记录状态为：

```text
CANCELED
```

取消原因：

| 原因码 | 说明 |
|---|---|
| USER_MESSAGE_DISABLED | 用户关闭该消息 |
| USER_BIZ_GROUP_DISABLED | 用户关闭该业务域 |
| USER_CHANNEL_DISABLED | 用户关闭该渠道 |
| RECIPIENT_ACCOUNT_MISSING | 缺少接收账户 |
| RECIPIENT_ACCOUNT_UNVERIFIED | 接收账户未验证 |
| CHANNEL_TEMPLATE_DISABLED | 渠道模板未启用 |
| CHANNEL_UNAVAILABLE | 无可用通道 |

失败重试只处理真实发送失败，不处理用户主动关闭或账户缺失导致的取消记录。

## 12. 实时推送方案

## 12.1 后端职责

系统消息发送成功后：

1. 写入系统消息表。
2. 发布 realtime 事件。
3. realtime 发布失败不影响系统消息落库。

实时事件 payload 必须结构化：

```json
{
  "messageId": "1001",
  "title": "登录提醒",
  "bizType": "auth.login_site_message",
  "priority": "IMPORTANT"
}
```

禁止后端发送 `id`，前端却读取 `messageId` 这类字段不一致。

## 12.2 前端职责

管理后台启动后初始化通知实时客户端：

- 首选 WebSocket。
- 自动降级 SSE。
- 再降级 Polling。

收到事件后：

1. 刷新未读数。
2. 拉取消息详情。
3. 根据用户偏好展示提示。

断线兜底：

- realtime 断开时每 30 秒轮询未读数。
- reconnect 成功后停止兜底轮询。

## 13. 数据模型建议

## 13.1 用户接收账户表

建议表名：

```text
notice_recipient_account
```

核心字段：

| 字段 | 说明 |
|---|---|
| id | 主键 |
| tenant_id | 租户 |
| user_id | 用户 ID |
| account_type | MOBILE、EMAIL、WECHAT、DINGTALK 等 |
| account_value | 手机号、邮箱、平台 ID |
| display_name | 展示名称 |
| verified_status | UNBOUND、PENDING_VERIFY、VERIFIED、DISABLED |
| default_account | 是否默认 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

## 13.2 用户接收偏好表

建议表名：

```text
notice_receive_preference
```

核心字段：

| 字段 | 说明 |
|---|---|
| id | 主键 |
| tenant_id | 租户 |
| user_id | 用户 ID |
| scope_type | GLOBAL、BIZ_GROUP、BIZ_TYPE |
| scope_value | 业务域或消息 key；GLOBAL 可为空 |
| channel_type | SITE、SMS、EMAIL 等；为空表示总开关 |
| enabled | 是否接收 |
| account_id | 指定接收账户，可为空 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

唯一约束建议：

```text
tenant_id + user_id + scope_type + scope_value + channel_type
```

## 15. 验收标准

## 15.1 消息入口

- 右上角小铃铛显示未读数。
- 收到新系统消息后未读数自动刷新。
- 点击小铃铛显示最新未读消息。
- 点击消息可以查看详情。
- 点击全部已读后角标归零。
- 点击查看全部进入消息中心。

## 15.2 消息中心

- 支持分页查询我的消息。
- 支持按未读、业务域、消息级别筛选。
- 支持详情、单条已读、批量已读、全部已读、删除。
- 页面不包含发送系统消息表单。

## 15.3 接收设置

- 用户可以按业务域配置是否接收。
- 用户可以按单消息配置是否接收。
- 用户可以按渠道配置是否接收。
- 短信渠道必须选择已验证手机号。
- 邮件渠道必须选择已验证邮箱。
- 没有账户时渠道开关不可用，并提示原因。
- 强制接收消息不可关闭。

## 15.4 发送判断

- 用户关闭某消息后，该消息不生成实际发送记录，生成 `CANCELED / USER_MESSAGE_DISABLED`。
- 用户关闭某渠道后，该渠道生成 `CANCELED / USER_CHANNEL_DISABLED`。
- 缺少手机号时短信生成 `CANCELED / RECIPIENT_ACCOUNT_MISSING`。
- 未验证邮箱时邮件生成 `CANCELED / RECIPIENT_ACCOUNT_UNVERIFIED`。
- 系统消息在允许发送时正常进入消息中心。
- 失败重试不处理 `CANCELED` 记录。

## 15.5 实时推送

- 系统消息发送后写入消息表。
- 在线用户通过 realtime 收到消息事件。
- 前端收到事件后刷新未读数。
- realtime 断开后轮询兜底仍能刷新未读数。
- payload 字段使用 `messageId`，前后端一致。

## 17. 风险与限制

- 浏览器桌面通知需要用户授权，不能保证默认可用。
- TTS 受浏览器自动播放策略影响，需要用户交互后更稳定。
- 微信、钉钉、企微等账户绑定涉及第三方平台授权，第一版只预留账户模型。
- realtime 多实例能力依赖 `mango-infra-realtime` 当前部署方式和网关策略。
- 接收设置会影响发送记录数量，需要在发送记录页面明确区分取消、失败、成功。
