# 机构模型改造与 E2E 任务计划

## 目标

将用户可见的“租户”收敛为“机构/企业空间”，同时保留底层 `tenant_id` 作为技术隔离字段。

本阶段不做大规模表名、字段名物理重命名，避免把业务概念调整和底层迁移风险混在一起。对前端用户、接口文档、菜单、页面、错误提示、E2E 用例，统一使用“机构”。对后端隔离、上下文、SQL 插件、JWT claim，继续使用 `tenantId`。

## 术语边界

| 层级 | 名称 | 说明 |
|---|---|---|
| 产品/用户可见 | 机构 | 元丰行、担保公司、银行、客户企业等拥有独立后台空间的业务主体 |
| 用户可见后台空间 | 机构空间/企业空间 | 机构自己的组织、成员、角色、流程、菜单权限边界 |
| 技术隔离字段 | tenantId / tenant_id | 后端用于行级隔离、权限隔离、审计隔离的技术字段 |
| 登录身份 | 账号 | 全局可登录身份 |
| 机构内身份 | 成员 | 某账号在某机构内的成员身份，对应 `tenant_member` |
| 机构内部结构 | 组织/岗位 | 机构内部部门、岗位，不等同于机构 |

## 改造原则

- 用户可见文本统一叫“机构”，不再显示“租户”。
- 接口路径本阶段保留 `/system/tenant`，避免前后端联调路径反复震荡；Swagger 描述改成“机构管理”。
- 数据库表、实体类、上下文字段本阶段保留 `tenant` 命名，只作为技术实现。
- 必要、将来一定要用的能力不留尾巴：机构成员、机构授权、机构初始化、机构禁用/删除边界必须逐步补齐。
- 每完成一个任务必须运行对应 E2E；失败时先修复再进入下一个任务。

### T8 追加：套餐与菜单关系

Pigx 的租户新增页里“套餐”本质上是菜单授权集合，不应该只落成一个 `dict` 编码。Mango 后续实现时按以下模型拆分：

- `authorization_menu_package`：套餐主档，维护套餐名称、编码、有效期、状态、备注。
- `authorization_menu_package_item`：套餐与菜单/按钮权限的多对多关系，保存菜单 ID，支持半选父级的子菜单授权。
- `sys_tenant_package` 或机构字段 `package_id`：机构当前绑定套餐。若需要历史追踪，使用独立绑定表保存生效时间、失效时间和操作人。
- 新增机构时根据套餐初始化机构管理员角色授权；切换套餐时生成授权变更预览，确认后同步机构默认角色菜单。
- 机构类型仍用于业务语义和模板初始化；套餐用于可见菜单和接口权限范围，二者不能混为一个字典。

## 菜单图标规范

- `menu_type = 3` 的按钮权限不需要图标。
- `menu_type = 1/2` 且 `visible = 1` 的目录、菜单必须配置图标。
- 顶部菜单和左侧菜单都必须显示稳定图标；新增迁移时一并修复历史缺失图标。
- 图标名称优先使用 Element Plus Icons 已存在的图标，避免自定义 SVG。
