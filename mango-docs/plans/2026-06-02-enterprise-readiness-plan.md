# Mango 企业业务开发框架正式推广计划

## 1. 目标

将 Mango 从“内部可继续开发验证”推进到“可以作为稳定企业业务开发框架正式推广”。

正式推广的含义：

- 企业不直接在 Mango 主仓做业务开发。
- 企业通过 Mango 发布物料初始化业务项目。
- 业务开发者可以在独立业务仓新增业务模块，并使用 Mango 平台能力完成真实业务交付。
- Mango 维护者可以通过固定验收门槛判断一次版本是否可推广。

## 2. 范围

本计划覆盖：

- Mango CLI 与业务 starter 初始化链路。
- 业务项目真实 CRUD 模块开发链路。
- 菜单、权限、租户、操作日志、文件、工作流、通知等平台能力接入。
- 单体和微前端两种运行形态。
- 后端、前端、CLI、模板、PMO baseline 和企业使用文档。
- 回归测试、截图验收和发布门槛。

## 3. 不做什么

- 不把单一业务项目逻辑写入 Mango 通用模块。
- 不恢复废弃保函业务模块。
- 不支持 CLI、前端包、后端 starter、业务模板之间任意版本组合。
- 不把 Mango Initializr Web UI 作为正式推广前置条件。
- 不以 mock、演示页面、接口 200 或页面能打开作为推广验收依据。

## 4. 当前判断

Mango 已具备企业业务开发起点，但尚未达到正式推广门槛。

已具备：

- `mango-cli` 和 `mango-business-starter` 可以生成业务项目。
- 业务项目可携带 `business-pmo/mango-baseline`。
- 业务 starter 已朝真实 CRUD 骨架收敛。
- 菜单、权限、租户、资源 manifest 和基础模块注册链路已具备。
- 废弃保函业务残留已单独清理。

未满足：

- 单体应用中仍有用户验收发现的文件上传组件 404、工作流组件 404。
- 表单设计器首个组件拖拽异常仍需确认和修复。
- 控制台 warning 仍需清理或登记为非阻塞。
- 企业业务项目全流程还未完成最终真实验收。
- E2E 验收证据需要覆盖功能结果、UI 细节、console/network 和截图。
- 发布物料版本一致性策略仍需固化。

## 5. 推广门槛

正式推广前必须全部满足：

1. `mango-cli` 能初始化独立业务项目，且生成项目不依赖 Mango 源码工作区。
2. 业务项目能新增至少一个真实业务模块，后端使用数据库、Service、MyBatis-Plus，前端使用真实 API。
3. 业务模块能接入菜单、权限、租户、操作日志、文件上传、工作流、通知。
4. 单体应用和微前端应用都能启动并完成浏览器验收。
5. 所有已知用户验收阻塞项修复或明确例外。
6. 全仓后端测试通过，前端关键构建和 E2E 通过。
7. 验收证据包含截图、功能断言、UI 细节、console/network 结果。
8. 企业使用文档覆盖环境准备、安装、初始化、开发、验证、升级。
9. 发布物料版本一致，禁止默认产生不兼容组合。

## 6. 阶段计划

### Sprint 1：推广阻塞项收口

目的：

- 先消除用户真实验收中阻塞正式推广的问题。
- 建立推广验收台账和证据格式。

范围：

- 单体文件上传组件 404。
- 单体工作流组件 404。
- 表单设计器首个组件拖拽异常。
- 控制台 warning 分类处理。
- E2E API baseURL 硬编码治理评估和接入。
- 保函残留清理分支合并后回归。

验收方式：

- 后端相关测试。
- 前端构建或受影响包测试。
- 单体真实应用浏览器验收。
- 微前端真实应用浏览器验收。
- 每个关键页面截图、console/network 记录和功能断言。

### Sprint 2：企业业务项目全流程仿真

目的：

- 验证业务开发者不改 Mango 主仓，也能完成业务模块开发。

范围：

- 使用 `mango-cli` 初始化业务项目。
- 新增中性业务模块，例如合同归档或订单管理。
- 业务模块使用 Mango 平台能力。
- 记录所有不通点。

验收方式：

- CLI 初始化命令通过。
- 业务项目 PMO preflight 通过。
- 后端 CRUD 单测和启动验证通过。
- 前端页面完成新增、查询、编辑、删除、上传、工作流发起或审批相关操作。
- 截图和 trace 留存。

### Sprint 3：规范和文档固化

目的：

- 让业务开发者、Mango 开发者和 Mango 维护者有清晰边界。
- 覆盖 AI 高频错误。

范围：

- `mango-pmo` 业务开发规则补强。
- 企业使用 README 补齐。
- 发布物料版本策略补齐。
- 验收证据模板和检查规则确认。

验收方式：

- 生成业务项目后能带出业务 PMO baseline。
- 文档中明确环境准备、初始化、开发、验证、升级。
- 规范不重复维护，长期规则只在 `mango-pmo`。

### Sprint 4：发布前全量回归

目的：

- 判断一个版本是否可以正式推广。

范围：

- 全仓后端测试。
- 前端构建和 E2E。
- 所有核心模块至少 3 个菜单页面、2 个真实功能抽查。
- 单体和微前端双形态验收。
- 发布说明和版本一致性检查。

验收方式：

- 自动化测试通过。
- 浏览器验收报告通过。
- 截图分析无明显 UI 错位、遮挡、空白、404、无限 loading。
- console/network 无阻塞错误。

## 7. 第一阶段执行顺序

1. 合并或确认保函残留清理分支，避免业务样例污染验证。
2. 拉取并审阅 open issue，标记推广阻塞项。
3. 修复单体文件上传组件和工作流组件 404。
4. 修复或隔离表单设计器首个组件拖拽异常。
5. 处理控制台 warning。
6. 跑真实单体与微前端浏览器验收。
7. 生成验收证据并提交 PR。

## 8. 风险与限制

- 真实浏览器验收依赖本地服务、数据库、前端依赖和端口状态。
- 部分历史 issue 可能不是正式推广前置项，需要逐项评估。
- 如果 CLI 版本、模板版本和后端 starter 版本没有统一发布策略，企业项目可能生成不兼容物料。
- 如果只做主链路验证，仍可能遗漏 UI 细节、权限和数据回显问题。
