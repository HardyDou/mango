| 模块 | 要求 |
|------|------|
| 上游对接 | 1个交易系统（固定） |

| 支付通道 | 1种（微信/支付宝） |
| 电子签章 | 第三方SDK |
| 架构 | 单体/微服务（预留） |

### 2.4 验收标准

- [ ] 投标人可在交易系统完成投保→支付→出函全流程
- [ ] 保函文件可正常下载
- [ ] 交易系统可验真
- [ ] 核心接口响应时间 < 3秒

### 2.5 独立可用性

> v1.0 MVP 独立可用，但仅支持单一渠道

## 8. MVP详细定义

### 8.1 MVP用户流程

```
┌─────────────────────────────────────────────────────────────────┐
│                   v1.0 MVP 用户流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   投标人                                                        │
│     │                                                          │
│     ▼                                                          │
│   1. 从交易系统跳转                                            │
│     │                                                          │
│     ▼                                                          │
│   2. 查看项目/选择保函                                         │
│     │                                                          │
│     ▼                                                          │
│   3. 提交投保申请                                              │
│     │                                                          │
│     ▼                                                          │
│   4. 支付保费（微信/支付宝）                                   │
│     │                                                          │
│     ▼                                                          │
│   5. 等待出函（1-5分钟）                                       │
│     │                                                          │
│     ▼                                                          │
│   6. 下载保函文件                                              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 8.2 MVP技术约束

| 约束项 | 说明 |
|--------|------|
| 交易系统 | 仅支持1个固定平台 |
| 出函机构 | 仅支持1家固定机构 |
| 支付方式 | 仅支持微信/支付宝 |
| 电子签章 | 仅用第三方SDK |
| 运营后台 | 简易版（后台代码） |

### 8.3 MVP数据模型（核心表）

```sql
-- 订单表（核心）
CREATE TABLE guarantee_order (
    id BIGINT PRIMARY KEY,
    order_no VARCHAR(50),        -- 订单号
    bidder_name VARCHAR(200),    -- 投标人
    guarantee_amount DECIMAL(18,2), -- 金额
    status VARCHAR(20),          -- 状态
    create_time DATETIME
);

-- 项目表
CREATE TABLE project (
    id BIGINT PRIMARY KEY,
    project_no VARCHAR(100),
    project_name VARCHAR(500),
    deadline DATETIME
);
```

## 9. 版本独立性

### 9.1 每个版本独立可用

| 版本 | 可独立运行 | 最小依赖 |
|------|-----------|----------|
| v1.0 | ✓ | 1交易系统+1机构 |
| v1.1 | ✓ | 1交易系统+1机构 |
| v1.2 | ✓ | 1交易系统+1机构 |
| v2.0 | ✓ | - |

> **原则**：每个版本发布后均可独立运行，不依赖后续版本

### 9.2 升级策略

- v1.0 → v1.1：平滑升级
- v1.1 → v1.2：平滑升级
- v1.2 → v2.0：平滑升级

## 10. 风险与预案

| 风险 | 影响 | 预案 |
|------|------|------|
| 第三方接口不稳定 | 出函失败 | 切换通道、重试机制 |
| 高并发性能不足 | 系统响应慢 | 限流、扩容 |
| 支付通道异常 | 无法支付 | 多通道备份 |
| 密钥泄露 | 安全风险 | 密钥轮换、审计 |
