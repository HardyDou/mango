# 上游平台接入对比分析报告

本文档对比分析各上游交易系统接入的相同与不同之处，为后续适配器设计方案提供参考。

---

## 一、平台对比总览

| 平台 | 模式 | 签名算法 | 加密方案 | 特殊要求 |
|------|------|----------|----------|----------|
| **数字泸州** | 数字信封 | SM3 | SM4+SM2数字信封 | 密文保函→明文保函两阶段 |
| **眉山4.0** | 数字信封 | SM3 | SM4+SM2数字信封 | 同数字泸州 |
| **江西省** | 平台密钥 | SM3 | SM4全局加密 | 支持见费/后费模式 |
| **蔓延** | 见费模式 | MD5 | 无 | 保费支付后出函 |

---

## 二、相同点分析

### 2.1 业务流程相同

所有平台的**核心业务流程**是一致的：

```
项目信息同步 → 投保申请 → 保函开具 → 保函验真 → 退保/注销
```

### 2.2 通讯协议相同

| 协议 | 说明 |
|------|------|
| HTTP/HTTPS | 必须支持 |
| JSON | 主流格式 |
| UTF-8 | 编码要求 |
| POST | 请求方式 |

### 2.3 核心接口相同

| 接口 | 说明 |
|------|------|
| 投保申请 | 平台→机构 |
| 投保结果通知 | 机构→平台 |
| 保函查询 | 平台→机构 |
| 退保通知 | 平台→机构 |

### 2.4 安全要求相同

| 要求 | 说明 |
|------|------|
| 三级等保 | 政务系统强制要求 |
| 日志留存 | ≥3年 |
| 签名验签 | 防篡改 |
| 加密传输 | HTTPS + 数据加密 |

## 三、不同点分析

### 3.1 加密方案差异

#### 方案一：数字信封模式（数字泸州、眉山）

```
┌─────────────────────────────────────────────────────────────────┐
│                    数字信封模式                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  投保申请：                                                    │
│  1. 用SM4密钥加密业务数据 → applydata                        │
│  2. 用投标人CA公钥加密SM4密钥 → digitalenvelope              │
│  3. 用appsecret对报文签名 → sign                             │
│                                                                  │
│  保函开具：                                                    │
│  1. 开标前：生成密文保函OF                                     │
│  2. 开标后：平台推送解密密钥                                  │
│  3. 生成明文保函，合并签章                                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**特点**：
- 敏感信息（投标人名称、金额等）用投标人CA公钥加密
- 开标前保密，开标后解密
- 安全性高

#### 方案二：平台密钥模式（江西省）

```
┌─────────────────────────────────────────────────────────────────┐
│                    平台密钥模式                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  投保申请：                                                    │
│  1. 用平台分配的appSecret作为SM4密钥加密整个请求体           │
│  2. 用appsecret对报文签名 → sign                             │
│                                                                  │
│  特点：                                                        │
│  - 无需投标人CA                                               │
│  - 平台统一管理密钥                                           │
│  - 解密简单                                                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**特点**：
- 统一密钥管理
- 实现简单
- 安全性依赖平台密钥安全

#### 方案三：无加密（蔓延见费模式）

```
┌─────────────────────────────────────────────────────────────────┐
│                    见费模式（无加密）                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  特点：                                                        │
│  - 无需加密敏感字段                                            │
│  - 保费支付成功后才出函                                       │
│  - 流程简单                                                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 签名算法差异

| 平台 | 签名算法 | 说明 |
|------|----------|------|
| 数字泸州 | SM3 | 国密算法 |
| 眉山4.0 | SM3 | 国密算法 |
| 江西省 | SM3 | 国密算法 |
| 蔓延 | MD5 | 普通算法 |

### 3.3 业务流程差异

#### 数字泸州/眉山：两阶段保函

```
阶段一（开标前）：
平台 → 机构：投保申请（密文）
机构 → 平台：密文保函

阶段二（开标后）：
平台 → 机构：解密密钥
机构 → 平台：明文保函+签章
```

#### 江西省：灵活模式

```
支持两种模式：
- 见费模式：保费支付后才出函
- 后费模式：先出函，后支付
```

#### 蔓延：纯见费模式

```
必须先支付保费：
投保申请 → 支付保费 → 出函
```

### 3.4 特殊字段差异

| 平台 | 特殊字段 | 说明 |
|------|----------|------|
| 国泰新点 | jbrname, jbrphone | 经办人信息 |
| 广联达 | project_category | 项目分类 |
| 福建 | biddercode | 统一社会信用代码 |
| 安徽 | deposit_amount, refund_amount | 保证金缴退分开 |

### 3.5 支付模式差异

| 模式 | 平台 | 说明 |
|------|------|------|
| 见费模式 | 蔓延 | 支付保费后才出函 |
| 后费模式 | 江西省 | 可以先出函后支付 |
| 混合模式 | 部分平台 | 根据配置选择 |

| sign | ✓ | ✓ | ✓ | ✓ |

## 五、适配器设计建议

│  └─────────────────────────────────────────────────────────────┘ │
│                            │                                    │
│          ┌───────────────┼───────────────┐                     │

│   ┌─────────────┐ ┌─────────────┐ ┌─────────────┐            │
│   │ 泸州适配器  │ │ 江西适配器  │ │ 蔓延适配器  │            │
│   └─────────────┘ └─────────────┘ └─────────────┘            │
│          │               │               │                     │
│          ▼               ▼               ▼                     │
│   ┌─────────────┐ ┌─────────────┐ ┌─────────────┐            │
│   │ 加密模块   │ │ 加密模块   │ │ 加密模块   │            │

每个通道的配置：

```json
{
  "channelCode": "SZ_LUZHOU",
  "channelName": "数字泸州",
  "signAlgorithm": "SM3",
  "encryptAlgorithm": "SM4",

  "secretKey": "encrypted_appsecret",

```

### 5.3 关键差异处理

| 差异点 | 处理方式 |
|--------|----------|
| 加密方案不同 | 抽象加密接口，每个适配器实现 |
| 签名算法不同 | 抽象签名接口，支持SM3/MD5 |
| 字段不同 | 扩展字段配置化 |
| 支付模式不同 | 策略模式处理 |
| 业务流程不同 | 适配器内部处理 |

## 六、设计方案要点

### 6.1 必须支持的能力

| 能力 | 说明 |
|------|------|
| 多签名算法 | SM3、MD5 |
| 多加密方案 | 数字信封、平台密钥、无加密 |
| 多支付模式 | 见费、后费 |
| 多字段扩展 | 扩展字段配置化 |
| 两阶段保函 | 密文+明文 |

### 6.2 配置化要求

| 配置项 | 说明 |
|--------|------|
| channelCode | 通道编码 |
| signAlgorithm | 签名算法 |
| encryptAlgorithm | 加密算法 |
| encryptMode | 加密模式 |
| signParams | 参与签名参数 |
| specialFields | 特殊字段映射 |

### 6.3 接口标准化

统一接口模型：

```json
// 请求
{
  "head": {
    "channelCode": "通道编码",
    "timestamp": "时间戳",
    "sign": "签名"
  },
  "body": {}
}

// 响应
{
  "code": "0000",
  "message": "success",
  "data": {}
}
```

适配器内部转换：

```
平台标准格式 → 适配器转换 → 平台特定格式
平台特定格式 ← 适配器转换 ← 平台标准格式
```

## 七、结论

### 7.1 核心差异

| 差异类型 | 影响程度 | 处理方式 |
|----------|----------|----------|
| 加密方案 | 高 | 抽象接口+多实现 |
| 签名算法 | 中 | 配置化切换 |
| 特殊字段 | 低 | 扩展字段配置 |
| 支付模式 | 中 | 策略模式 |
| 业务流程 | 低 | 适配器内部处理 |

### 7.2 设计建议

1. **抽象加密接口**：支持数字信封、平台密钥、无加密三种模式
2. **配置化通道管理**：新增通道只需配置，无需代码改动
3. **扩展字段支持**：特殊字段通过配置映射
4. **两阶段保函支持**：密文保函+明文保函流程内置
5. **支付模式策略**：见费/后费通过策略切换
