## 第一部分：关键业务点

### 一、业务类型

#### 1. 按出函机构分类

| 类型 | 说明 | 特点 |
|------|------|------|
| **银行电子保函** | 银行直接开具 | 信用度高，费率低 |
| **分离式银行保函** | 担保公司担保，银行出具 | 门槛低，覆盖广 |
| **保证保险保单** | 保险公司出具 | 审批快，覆盖广 |
| **担保保函** | 担保公司出具 | 灵活，门槛低 |

#### 2. 按使用场景分类

| 场景 | 说明 | 金额比例 |
|------|------|----------|
| **投标保证金保函** | 投标阶段替代保证金 | 不超过80万 |
| **履约保证金保函** | 中标后履约担保 | 合同价5-10% |
| **工程质量保证金保函** | 工程质量担保 | 合同价3% |
| **农民工工资支付保函** | 工资支付担保 | 约定 |
| **差额保证金保函** | 最低价中标差额担保 | 差额部分 |

### 四、电子签章与加密

#### 1. 信息加密场景

| 阶段 | 内容 | 说明 |
|------|------|------|
| 开标前 | 敏感信息加密 | 投标人名称、保证金金额等 |
| 开标后 | 自动解密 | 开标时间到达自动触发 |

#### 2. 加密方案：数字信封

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      数字信封加密原理                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   发送方：                                                          │
│   1. 生成随机对称密钥 (SM4)                                           │
│   2. 用对称密钥加密保函文件                                           │
│   3. 用接收方公钥 (SM2) 加密对称密钥                                 │
│   4. 合并 = 加密文件 + 加密密钥(数字信封)                            │
│                                                                      │
│   接收方：                                                          │
│   1. 用私钥 (SM2) 解密对称密钥                                       │
│   2. 用对称密钥 (SM4) 解密文件                                        │
│                                                                      │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 3. 上游接口安全机制

##### 3.1 接口防篡改（签名验签）

| 平台类型 | 签名算法 | 密钥 | 典型平台 |
|----------|----------|------|----------|
| **国密SM3** | SM3 | appkey + appsecret | 数字泸州、眉山4.0、接口规范 |
| **MD5** | MD5 | appkey + appsecret | 蔓延-见费模式 |
| **SM2签名** | SM2 | CA证书 | 部分标准规范 |

**签名流程**（以SM3为例）：
```
1. 参数按ASCII升序排序
2. 拼接成 "key=value&key=value" 格式
3. 末尾拼接 appsecret
4. 计算SM3值作为sign
```

##### 3.2 业务数据加密/解密（两种模式对比）

| 方案 | 加密内容 | 密钥来源 | 典型平台 |
|------|----------|----------|----------|
| **数字信封模式** | applydata (SM4) + digitalenvelope (SM2加密密钥) | 投标人CA证书公钥 | 数字泸州、接口规范 |
| **平台密钥模式** | 整个请求体用appSecret SM4加密 | 平台分配appSecret | 江西省、眉山 |

**关键字段说明**：

| 字段 | 用途 |
|------|------|
| `applydata` | 投保申请数据（SM4加密后的业务数据） |
| `digitalenvelope` | 数字信封 - 用投标人CA公钥加密SM4密钥 |
| `key` | 开标后推送，用于解密项目信息（SM2加密） |

##### 3.3 applydata 解密后字段

| 字段 | 说明 |
|------|------|
| biddername | 投标人名称（开标前密文） |
| biddercode | 统一社会信用代码 |
| biaoduanno | 标段编号 |
| biaoduanname | 标段名称 |
| bzjamount | 保证金金额 |
| projectname | 项目名称 |
| projectno | 项目编号 |
| tenderno | 招标编号 |

### 五、待深入讨论事项

#### 1. 高并发痛点

| 痛点 | 场景 | 影响 |
|------|------|------|
| **投标截止集中** | 开标前1小时大量投保 | 系统响应慢、超时 |
| **批量出函** | 多家金融机构同时响应 | 通道拥堵 |
| **高峰叠加** | 多项目同时开标 | 系统崩溃 |

**解决方案**：
- 异步处理队列
- 限流熔断机制
- 弹性扩容

#### 2. 流程复杂痛点

| 痛点 | 场景 | 影响 |
|------|------|------|
| **多系统交互** | 交易系统→平台→金融机构 | 流程中断难追溯 |
| **人工环节** | 审核、盖章、确认 | 效率低下 |
| **异常处理** | 超时、退保、拒赔 | 人工干预成本高 |

**解决方案**：
- 分布式事务管理
- 自动化流程引擎
- 补偿机制设计

#### 3. 数据安全痛点

| 痛点 | 场景 | 影响 |
|------|------|------|
| **敏感信息泄露** | 投标人信息、保证金金额 | 串标风险 |
| **身份冒用** | 伪造投保申请 | 欺诈风险 |
| **保函篡改** | 修改保函金额、日期 | 资损风险 |

**解决方案**：
- 全流程加密
- CA身份认证
- 电子签章防篡改

#### 4. 对接成本痛点

| 痛点 | 场景 | 影响 |
|------|------|------|
| **标准不统一** | 各地接口差异 | 重复开发 |
| **机构对接多** | 10+银行/保险/担保 | 维护困难 |
| **协议差异** | JSON/XML/SDK | 适配复杂 |

**解决方案**：
- 适配器模式
- 统一接口网关
- 配置化管理

#### 2. 分布式事务处理

```
┌─────────────────────────────────────────────────────────────────────┐
│                      分布式事务处理方案                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  适用性分析：                                                        │
│  - TCC适用于：有账户余额体系、需要资源预占的场景                      │
│  - 电子保函场景：无账户余额、实时支付（微信/支付宝/网银）           │
│  - 结论：不适用TCC，采用消息队列 + 本地消息表                        │
│                                                                      │
│  方案一: 可靠消息 + 本地消息表 (推荐)                               │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. 投保申请 → 本地事务写入业务表 + 消息表                 │   │
│  │  2. 发送消息到MQ                                           │   │
│  │  3. 消费消息处理下游业务（保函开具/通知）                  │   │
│  │  4. 定时扫描消息表，失败则补偿重试                         │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  方案二: Saga模式                                                   │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. 投保申请 ──> 风控审核 ──> 保函开具 ──> 通知         │   │
│  │  2. 任一步骤失败 ──> 逆向补偿                              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│  适用：长流程、多步骤、允许逆向补偿                                 │
│                                                                      │
│  方案三: 简单状态机                                                 │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  PENDING → PROCESSING → ISSUED / FAILED                   │   │
│  │       (可重试)     (可回退)                                │   │
│  └─────────────────────────────────────────────────────────────┘   │
│  适用：流程短、步骤少、失败直接重试                                 │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

// PDF生成 + 签章示例
public class PdfSignatureService {
    
    // 1. 使用iText生成PDF
    public byte[] generatePdf(GuaranteeInfo info) {
        Document document = new Document();
        ByteArrayOutputStream baos = new ByteArrayOutputStream();
        PdfWriter.getInstance(document, baos);
        document.open();
        
        // 添加保函内容
        document.add(new Paragraph("保函编号: " + info.getGuaranteeNo()));
        document.add(new Paragraph("投标人: " + info.getBidderName()));
        document.add(new Paragraph("金额: " + info.getAmount()));
        // ... 更多内容
        
        document.close();
        return baos.toByteArray();
    }
    
    // 2. 使用iText进行电子签名
    public byte[] signPdf(byte[] pdfContent, PrivateKey privateKey, 
                           Certificate certificate) {
        PdfReader reader = new PdfReader(pdfContent);
        ByteArrayOutputStream baos = new ByteArrayOutputStream();
        PdfStamper stamper = PdfStamper.createSignature(
            reader, baos, '\0', null, true
        );
        
        // 设置签名外观
        PdfSignatureAppearance appearance = stamper.getSignatureAppearance();
        appearance.setVisibleSignature(
            new Rectangle(100, 100, 200, 200), 1, null
        );
        appearance.setReason("电子保函签章");
        appearance.setLocation("电子保函平台");
        
        // 创建签名
        ExternalSignature signature = new PrivateKeySignature(
            privateKey, DigestAlgorithms.SHA256, "BC"
        );
        MakeSignature.signDetached(
            appearance, signature, 
            new Certificate[]{certificate},
            null, null, null, 0, CryptoStandard.CMS
        );
        
        return baos.toByteArray();
    }
}

        page.addText(50, 80, "投标人: " + info.getBidderName());
        // ... 更多内容
        
        return doc.saveToBytes();
    }
    
    // OFD签章
    public byte[] signOfd(byte[] ofdContent, PrivateKey privateKey) {
        OfdDocument doc = OfdDocument.loadFromBytes(ofdContent);
        
        // 添加电子印章
        doc.addSeal(new SealInfo(
            privateKey, 
            SealPosition.BOTTOM_RIGHT
        ));
        
        // 添加时间戳
        doc.addTimestamp(new TimestampInfo(
            "http://tsa.cn/time",
            privateKey
        ));
        
        return doc.saveToBytes();
    }
