## 一、需求分析

### 1.1 业务类型

#### 按出函机构分类

| 类型 | 说明 | 特点 |
|------|------|------|
| **银行电子保函** | 银行直接开具 | 信用度高，费率低 |
| **分离式银行保函** | 担保公司担保，银行出具 | 门槛低，覆盖广 |
| **保证保险保单** | 保险公司出具 | 审批快，覆盖广 |
| **担保保函** | 担保公司出具 | 灵活，门槛低 |

#### 按使用场景分类

| 场景 | 说明 | 金额比例 |
|------|------|----------|
| **投标保证金保函** | 投标阶段替代保证金 | 不超过80万 |
| **履约保证金保函** | 中标后履约担保 | 合同价5-10% |
| **工程质量保证金保函** | 工程质量担保 | 合同价3% |
| **农民工工资支付保函** | 工资支付担保 | 约定 |
| **差额保证金保函** | 最低价中标差额担保 | 差额部分 |

### 3.2 分布式事务（去TCC化）

```
┌─────────────────────────────────────────────────────────────────────┐
│                      分布式事务处理方案                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  适用性分析：                                                        │
│  - TCC适用于：有账户余额体系、需要资源预占的场景                      │
│  - 电子保函场景：无账户余额、实时支付（微信/支付宝/网银）           │
│  - 结论：不适用TCC，采用消息队列 + 本地消息表                        │
│                                                                      │
│  方案一: 可靠消息 + 本地消息表 (推荐)                               │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. 投保申请 → 本地事务写入业务表 + 消息表                 │   │
│  │  2. 发送消息到MQ                                           │   │
│  │  3. 消费消息处理下游业务（保函开具/通知）                  │   │
│  │  4. 定时扫描消息表，失败则补偿重试                         │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  方案二: Saga模式                                                   │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. 投保申请 ──> 风控审核 ──> 保函开具 ──> 通知         │   │
│  │  2. 任一步骤失败 ──> 逆向补偿                              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│  适用：长流程、多步骤、允许逆向补偿                                 │
│                                                                      │
│  方案三: 简单状态机                                                 │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  PENDING → PROCESSING → ISSUED / FAILED                   │   │
│  │       (可重试)     (可回退)                                │   │
│  └─────────────────────────────────────────────────────────────┘   │
│  适用：流程短、步骤少、失败直接重试                                 │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

> **重要决策**：电子保函系统无账户余额体系，不采用TCC模式

### 3.4 接口安全机制

#### 接口防篡改（签名验签）

| 平台类型 | 签名算法 | 密钥 | 典型平台 |
|----------|----------|------|----------|
| **国密SM3** | SM3 | appkey + appsecret | 数字泸州、眉山4.0 |
| **MD5** | MD5 | appkey + appsecret | 蔓延-见费模式 |
| **SM2签名** | SM2 | CA证书 | 部分标准规范 |

#### 业务数据加密/解密

| 方案 | 加密内容 | 密钥来源 | 典型平台 |
|------|----------|----------|----------|
| **数字信封模式** | applydata (SM4) + digitalenvelope (SM2加密密钥) | 投标人CA证书公钥 | 数字泸州 |
| **平台密钥模式** | 整个请求体用appSecret SM4加密 | 平台分配appSecret | 江西省、眉山 |

#### 平台配置设计

```
每个上游平台需独立配置：
- platformCode: 平台编码
- signAlgorithm: 签名算法 (SM3/MD5/SM2)
- signParams: 参与签名的参数列表
- encryptAlgorithm: 加密算法
- secretKey: appSecret
- caCertPath: CA证书路径（数字信封模式）
```

## 四、业务规则

```
┌─────────────────────────────────────────────────────────────┐
│                     保函开具校验规则                          │
├─────────────────────────────────────────────────────────────┤
│  ✓ 投标人资格校验                                           │
│    - 营业执照有效性                                          │
│    - 未被列入失信名单                                        │
│    - 投标资格符合                                           │
│                                                              │
│  ✓ 项目信息校验                                             │
│    - 项目状态为有效                                         │
│    - 标段可投                                               │
│    - 保证金金额符合                                         │
│                                                              │
│  ✓ 额度校验                                                 │
│    - 授信额度充足                                           │
│    - 保额在产品限额内                                       │
│    - 保费计算正确                                           │
│                                                              │
│  ✓ 有效期校验                                              │
│    - 保函生效日期 >= 投标截止日期                            │
│    - 保函到期日期 >= 投标有效期 + 30天                       │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 风控规则

| 规则类型 | 规则内容 | 处理方式 |
|----------|----------|----------|
| **黑名单校验** | 投标人/企业是否在黑名单 | 拒绝投保 |
| **关联校验** | 是否存在关联企业重复投保 | 预警 |
| **额度校验** | 是否超过授信额度 | 拒绝/降额 |
| **历史违约** | 历史是否有违约记录 | 拒绝/提高保费 |
| **反洗钱** | 是否符合反洗钱要求 | 名单筛查 |

## 五、总结

|------|---------|
