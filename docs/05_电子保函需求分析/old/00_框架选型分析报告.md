## 二、工作流引擎选型

### 2.1 业务需求分析

电子保函系统的审核审批流程具有以下特点：

| 流程环节 | 复杂度 | 工作流需求 |
|---------|--------|-----------|
| 保函申请 | 中 | 提交 → 补正 → 受理 |
| 审核审批 | 高 | 多级审批（风控→法务→领导）、条件分支、协办 |
| 出函签发 | 中 | 签发→盖章→生成→推送 |
| 归档管理 | 低 | 可不用工作流 |

### 2.2 工作流引擎对比

| 引擎 | 优点 | 缺点 | 适用场景 |
|-----|------|------|---------|
| **Flowable** | 功能最全、BPMN 2.0标准、社区活跃 | 较重 | 复杂审批（多级、会签） |
| **Camunda** | 轻量、性能好、云原生 | 中文资料少 | 高并发、简单流程 |
| **Activiti** | 历史悠久、生态丰富 | 已停止维护 | 不推荐新项目 |

### 2.3 推荐：Flowable

理由：
1. **功能完整**: 支持BPMN 2.0、多实例（会签/或签）、定时器
2. **国内适配**: 有中文社区、钉钉/企微集成方案
3. **集成方便**: Spring Boot starter完善
4. **维护活跃**: 持续更新
5. **支持流程设计器**: 提供流程设计可视化界面

## 三、分布式事务策略

### 3.1 方案对比

| 方案 | 模式 | 复杂度 | 性能 | 适用场景 |
|-----|------|--------|------|---------|
| **Seata AT** | 最终一致 | 中 | 高 | 电商、库存、支付 |
| **Seata TCC** | 最终一致 | 高 | 极高 | 高并发、金融 |
| **Seata Saga** | 最终一致 | 中 | 高 | 长流程、编排 |
| **Seata XA** | 强一致 | 低 | 低 | 金融、银行 |

### 3.2 推荐方案：Seata AT 模式

Seata AT 是 Spring Cloud Alibaba 生态的分布式事务解决方案：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Seata AT 模式原理                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   TM (事务管理器)                                                      │
│        │                                                                │
│        ▼                                                                │
│   ┌─────────┐    1. 注册全局事务    ┌─────────┐                     │
│   │  TC     │ ◀──────────────────▶ │  RM     │                     │
│   │ Server  │    2. 执行分支事务    │ (业务DB) │                     │
│   │         │ ◀──────────────────▶ │         │                     │
│   └─────────┘    3. 提交/回滚      └─────────┘                     │
│        │                                                                │
│        │                                                                │
│        ▼                                                                │
│   ┌─────────────────────────────────────────┐                          │
│   │  Undo Log 表（自动记录回滚日志）        │                          │
│   └─────────────────────────────────────────┘                          │
│                                                                      │
└─────────────────────────────────────────────────────────────────────────────┘
```

**使用方式**：
```yaml
seata:
  enabled: true
  data-source-proxy-mode: AT
```

**优点**：
- **低侵入**: 自动代理数据源，代码无需改动
- **自动回滚**: 通过 Undo Log 实现自动回滚
- **生态集成**: Spring Cloud Alibaba 原生支持
- **运维简单**: 可复用业务数据库，无需额外部署

### 3.3 电子保函业务流程适配

```
申请提交 ──▶ Seata AT ──▶ 流程引擎创建流程实例
                            │
                            ▼
审核通过 ──▶ Seata AT ──▶ 出函服务生成保函文件
                            │
                            ▼
签发成功 ──▶ Seata AT ──▶ 通知服务推送结果
```

### 3.4 注意事项

| 场景 | 处理方式 |
|-----|---------|
| Undo Log 膨胀 | 定时清理（建议保留7天） |
| 大表操作 | 开启全局锁，可能影响性能 |
| 异常处理 | 自动回滚，可通过 Seata Console 查看状态 |

## 四、推荐方案

### 4.1 推荐: Pig + 定制开发

```
原因:
1. 电子保函系统需要微服务架构
2. 需要对接多通道(银行/保险/担保)
3. 需要分布式事务保证数据一致性 (Seata AT)
4. 需要高并发处理
5. 需要工作流引擎支持灵活审批
6. Pig的Spring Cloud Alibaba生态完美匹配

## 八、结论

### 推荐: 基于 Pig 框架进行电子保函系统开发

1. **生态匹配**: Spring Cloud Alibaba 完美支持分布式架构需求
2. **权限完善**: RBAC + 数据权限可满足多租户多通道需求
3. **工作流支持**: Flowable 支持企业自定义审批流程
4. **事务一致性**: Seata AT 模式保证数据一致性
5. **部署灵活**: 支持微服务/模块/单体多种部署模式
6. **社区活跃**: 问题解决方案多
7. **快速开发**: 代码生成器 + 业务模块复用

### 核心改造点:
1. 权限模块增强 (通道/项目/金额数据权限)
2. 新增通道适配模块 (多通道接入)
3. 新增保函业务模块 (核心业务)
4. 新增签章模块 (PDF/OFD签章)
5. 新增工作流模块 (Flowable可视化流程)
6. 引入分布式事务 (Seata AT模式)

### 9.2 面临的问题

### 9.3 解决方案

**设计原则**：
1. **API 先行**：先定义接口，调用方与实现方解耦
2. **内部调用优先**：同进程用 Spring Bean 调用，跨进程用 Feign
3. **配置开关**：通过 `@ConditionalOnProperty` 控制启用

#### 9.3.2 服务注册发现适配

```java
// 方案：通过配置开关控制
@Configuration
public class RegistryConfig {
    
    @Bean
    @ConditionalOnProperty(name = "registry.enabled", havingValue = "true")
    public NacosServiceRegistry nacosServiceRegistry() {
        return new NacosServiceRegistry();
    }
    
    @Bean
    @ConditionalOnProperty(name = "registry.enabled", havingValue = "false")
    public LocalServiceRegistry localServiceRegistry() {
        return new LocalServiceRegistry();  // 单体模式：直接注册到本地缓存
    }
}
```

#### 9.3.3 数据库架构适配

```java
// 方案：分库分表 vs 单库多表
// application.yml
spring:
  datasource:
    # 单体部署模式
    url: jdbc:mysql://localhost:3306/guarantee
    # 微服务部署模式  
    # url: jdbc:mysql://localhost:3306/guarantee_${service-name}
```

**策略**：
| 部署模式 | 数据库策略 |
|---------|-----------|
| 单体 | 单库多表（按业务模块前缀区分） |
| 模块化 | 按模块分库，或同一库不同表 |
| 微服务 | 每服务独立库，或分库分表 |

#### 9.3.4 分布式事务适配

```java
// 方案：按部署模式选择事务策略
@Configuration
public class TransactionConfig {
    
    @Bean
    @ConditionalOnProperty(name = "deploy.mode", havingValue = "single")
    public LocalTransactionManager localTransactionManager() {
        return new LocalTransactionManager();  // 单体：本地事务
    }
    
    @Bean
    @ConditionalOnProperty(name = "deploy.mode", havingValue = "micro")
    public SeataTransactionManager seataTransactionManager() {
        return new SeataTransactionManager();  // 微服务：Seata AT
    }
}
```

#### 9.3.5 服务调用适配

```java
// 方案：统一调用入口，内部调用走 Bean，远程调用走 Feign
@Service
public class GuaranteeService {
    
    @Autowired
    private ChannelAdapterService channelAdapter;  // 内部服务
    
    @Autowired
    private RemoteProductService remoteProductService;  // 远程服务
    
    public void process() {
        // 同一进程：直接调用
        channelAdapter.apply();
        
        // 跨服务：Feign 调用（自动切换）
        remoteProductService.getProduct();
    }
}

// Feign 客户端（自动降级）
@FeignClient(name = "product-service", fallback = ProductServiceFallback.class)
public interface RemoteProductService {
    @GetMapping("/product/{id}")
    Product getProduct(@PathVariable("id") Long id);
}
```

#### 9.3.6 配置中心适配

```yaml
# 单体部署配置
deploy:
  mode: single
  registry:
    enabled: false
  datasource:
    type: single  # 单库
  
# 微服务部署配置
deploy:
  mode: micro
  registry:
    enabled: true
    type: nacos
  datasource:
    type: distributed  # 分库
```

### 9.4 部署模式对比

| 维度 | 单体部署 | 模块化部署 | 微服务部署 |
|------|---------|-----------|-----------|
| **代码结构** | 多模块 Maven | 多模块 Maven | 多模块 Maven + 多项目 |
| **数据库** | 单库多表 | 分库或分表 | 每服务独立库 |
| **服务发现** | 无 | 可选 | Nacos |
| **分布式事务** | 本地事务 | Seata AT | Seata AT/TCC |
| **网关** | 无 | 可选 | Spring Cloud Gateway |
| **配置中心** | 本地/Nacos | Nacos | Nacos |
| **部署复杂度** | 低 | 中 | 高 |
| **运维成本** | 低 | 中 | 高 |
| **扩展性** | 差 | 中 | 好 |
| **适用规模** | <10人团队 | 10-50人 | 50+人 |

| 数据库架构 | MyBatis-Plus 多数据源切换 |
| 分布式事务 | 本地事务 / Seata AT 切换 |
| 服务调用 | Feign + 内部调用统一入口 |
