### 3.2 平台配置设计

```
每个上游平台需独立配置：
- platformCode: 平台编码
- signAlgorithm: 签名算法 (SM3/MD5/SM2)
- signParams: 参与签名的参数列表
- encryptAlgorithm: 加密算法
- secretKey: appSecret
- caCertPath: CA证书路径（数字信封模式）
- ipWhitelist: IP白名单
- protocolType: 协议类型 (HTTP/SDK/MQ/FILE)
```

## 4. 分布式事务方案

### 4.1 方案选择

> **重要决策**：电子保函系统无账户余额体系，**不采用TCC模式**

### 4.2 推荐方案

```
┌─────────────────────────────────────────────────────────────────────┐
│                  分布式事务处理方案                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  适用性分析：                                                        │
│  - TCC适用于：有账户余额体系、需要资源预占的场景                      │
│  - 电子保函场景：无账户余额、实时支付（微信/支付宝/网银）           │
│  - 结论：不适用TCC，采用可靠消息 + 本地消息表                        │
│                                                                      │
│  方案一: 可靠消息 + 本地消息表 (推荐)                               │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. 投保申请 → 本地事务写入业务表 + 消息表                 │   │
│  │  2. 发送消息到MQ                                           │   │
│  │  3. 消费消息处理下游业务（保函开具/通知）                  │   │
│  │  4. 定时扫描消息表，失败则补偿重试                         │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  方案二: Saga模式                                                   │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  1. 投保申请 ──> 风控审核 ──> 保函开具 ──> 通知         │   │
│  │  2. 任一步骤失败 ──> 逆向补偿                              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│  适用：长流程、多步骤、允许逆向补偿                                 │
│                                                                      │
│  方案三: 简单状态机                                                 │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  PENDING → PROCESSING → ISSUED / FAILED                   │   │
│  └─────────────────────────────────────────────────────────────┘   │
│  适用：流程短、步骤少、失败直接重试                                 │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## 6. 接口安全机制

### 6.1 接口防篡改

| 平台类型 | 签名算法 | 密钥 | 典型平台 |
|----------|----------|------|----------|
| **国密SM3** | SM3 | appkey + appsecret | 数字泸州、眉山4.0 |
| **MD5** | MD5 | appkey + appsecret | 蔓延-见费模式 |
| **SM2签名** | SM2 | CA证书 | 部分标准规范 |

### 6.2 业务数据加密

| 方案 | 加密内容 | 密钥来源 | 典型平台 |
|------|----------|----------|----------|
| **数字信封模式** | applydata (SM4) + digitalenvelope (SM2加密密钥) | 投标人CA证书公钥 | 数字泸州 |
| **平台密钥模式** | 整个请求体用appSecret SM4加密 | 平台分配appSecret | 江西省、眉山 |

### 6.3 签名流程（SM3示例）

```
1. 参数按ASCII升序排序
2. 拼接成 "key=value&key=value" 格式
3. 末尾拼接 appsecret
4. 计算SM3值作为sign
```

## 12. 等保三级合规

### 12.1 等保三级要求

| 安全层面 | 要求项 | 实现方案 |
|----------|--------|----------|
| **物理安全** | 机房访问控制、监控 | 政务云机房/私有机房 |
| **网络安全** | 防火墙、入侵检测、隔离 | VPC/安全组/硬件防火墙 |
| **主机安全** | 身份鉴别、访问控制、审计 | 系统加固、堡垒机 |
| **应用安全** | 安全审计、访问控制、通信保密 | 安全框架、TLS |
| **数据安全** | 备份恢复、加密存储 | 异地备份、AES/SM4 |

### 12.2 评测项清单

| 序号 | 评测项 | 优先级 | 实现方式 |
|------|--------|--------|----------|
| 1 | 身份鉴别 | 高 | 双因子认证、堡垒机 |

### 13.5 多地部署架构

class MultiRegionDeployment:
    # 方案一：物理隔离
    def physical_isolation(self):
        """各地市独立部署，数据完全隔离"""
        # 优点：安全可靠
        # 缺点：运维成本高

    # 方案二：逻辑隔离
    def logical_isolation(self):

        # 优点：统一运维
        # 缺点：需严格权限控制

    # 推荐：方案一 + 方案二结合
    # 敏感数据物理隔离，普通数据逻辑隔离
