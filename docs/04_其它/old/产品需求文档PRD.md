## 一、产品概述

### 1.1 产品背景

电子保函系统是服务于公共资源交易领域的数字化担保凭证管理平台，由保险公司、银行或担保公司应申请人请求，向受益人开具电子担保文件。系统支持投标保证、合同履约、预付款支付、质量保证等多种场景的全流程线上化管理。

### 1.2 产品定位

打造覆盖"投保申请-审核出函-保函管理-退保理赔"全流程的电子保函服务平台，实现与交易平台、金融机构（银行/保险/担保公司）的无缝对接。

### 1.3 产品目标

1. **业务目标**
   - 支持4种主流保函类型：投标保函、履约保函、预付款保函、质量保函
   - 实现与多个交易平台、银行、保险机构、担保公司的标准化对接
   - 提供见费模式和见保模式两种出函方式

2. **用户目标**
   - 投标人：全程线上办理保函，便捷高效
   - 金融机构：自动化审核出函，降低运营成本
   - 交易中心：统一保函管理，提升监管效率

## 二、用户角色与权限

### 2.1 系统定位

本系统为**电子保函SaaS平台**，采用多租户架构，支持多方对接：

| 角色 | 说明 | 数量 |
|------|------|------|
| 交易平台 | 上游渠道（对接保函业务） | 多个 |
| 出函机构 | 银行/保险/担保公司（出具保函） | 多个 |
| 担保公司 | 平台入驻企业（保函业务主体） | 多个 |

**担保公司配置**：每个担保公司可自主选择：
- 接入的交易平台（可选一个或多个）
- 接入的出函机构（可选一个或多个）

### 2.2 系统端点划分

| 系统 | 说明 | 部署方式 | 主要用户 |
|------|------|---------|---------|
| **Web Portal** | 业务办理门户 | Web端 | 投标人 |
| **后台管理系统** | 机构内部管理 | Web端 | 金融机构、系统管理员 |
| **移动端App** | 移动办理 | iOS/Android | 投标人 |
| **API接口** | 对接交易平台 | API | 交易平台 |

### 2.3 用户角色

| 角色 | 说明 | 适用端点 | 主要功能 |
|------|------|---------|---------|
| 投标人 | 保函申请方 | Portal/App | 申请投保、查看进度、下载保函、申请退保 |
| 担保公司 | 平台入驻企业（业务运营方）| 后台系统 | 投保管理、选择出函机构、查看保函、统计报表 |
| 出函机构 | 银行/保险/担保公司（出函方）| 后台系统 | 审核投保、出具保函、保函管理、退保处理 |
| 系统管理员 | 平台运维 | 后台系统 | 系统配置、机构管理、用户管理 |

> **注**：招标代理、交易中心在上游交易平台操作，不在本系统直接操作。

### 2.4 移动端功能范围

| 角色 | 支持功能 |
|------|---------|
| 投标人 | 投保申请、进度查询、保函下载、消息通知、个人中心 |

### 2.5 权限体系

- **角色级权限**：基于用户角色分配功能权限
- **机构级权限**：基于所属机构限制数据访问范围
- **操作级权限**：具体到单个功能的增删改查权限
- **租户级权限**：基于对接的交易平台区分数据隔离

## 四、业务流程

### 4.1 投保出函流程（见费模式）

```
┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
│  投保申请   │───▶│  保费缴纳   │───▶│  审核出函   │───▶│  保函送达   │───▶│ 开标/公示  │
└────────────┘    └────────────┘    └────────────┘    └────────────┘    └────────────┘
     ↓                                    ↓
┌────────────┐                     ┌────────────┐
│  投保撤销   │                     │  审核拒绝   │
└────────────┘                     └────────────┘
```

### 4.1.1 投保出函流程（见保模式）

```
┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
│  投保申请   │───▶│  审核出函   │───▶│  保费缴纳   │───▶│  保函送达   │───▶│ 开标/公示  │
└────────────┘    └────────────┘    └────────────┘    └────────────┘    └────────────┘
     ↓                                    ↓
┌────────────┐                     ┌────────────┐
│  投保撤销   │                     │  审核拒绝   │
└────────────┘                     └────────────┘
```

> **说明**：
> - **见费模式**：保费到账后出具保函，风险较低
> - **见保模式**：出具保函后收取保费，需评估投保人信用

| 接口名称 | 功能 | 方向 |
|---------|------|------|
| 投保申请接口 | 提交投保申请 | 系统→金融机构 |
| 投保结果查询 | 查询投保审核结果 | 系统→金融机构 |

### 5.3 接口规范

- **传输协议**: HTTPS + JSON
- **加密算法**: 国密SM3（报文摘要）、SM4（文件加密）
- **数字证书**: CA数字证书认证
- **电子签章**: 支持电子签章功能
- **文档格式**: OFD版式文档
- **编码格式**: UTF-8

### 5.4 接口报文格式

#### 5.4.1 通用报文头

| 字段 | 类型 | 说明 |
|------|------|------|
| `transCode` | String | 交易代码 |
| `transDate` | String | 交易日期(YYYYMMDD) |
| `transTime` | String | 交易时间(HHmmss) |
| `reqSeqNo` | String | 请求流水号 |
| `signData` | String | 签名数据(Base64) |

#### 5.4.2 通用响应码

| 响应码 | 说明 |
|--------|------|
| 0000 | 交易成功 |
| 1001 | 参数校验失败 |
| 1002 | 签名验证失败 |
| 1003 | 报文格式错误 |
| 2001 | 业务处理中 |
| 2002 | 业务处理失败 |
| 3001 | 数据不存在 |
| 3002 | 数据已存在 |
| 4001 | 权限不足 |
| 9999 | 系统异常 |

#### 5.5 异常流程处理

| 异常场景 | 处理机制 |
|----------|----------|
| 接口超时 | 启用重试机制（最多3次，间隔5s/10s/30s） |
| 金融机构系统故障 | 记录异常，自动切换备用渠道 |
| 保函生成失败 | 人工介入处理，发送告警通知 |
| 退保失败 | 原路退回，自动重试 |
| 签名验证失败 | 记录日志，拦截请求，通知排查 |
| 报文解密失败 | 记录原始报文，告警通知，人工处理 |

## 七、保函类型详细需求

> **保函格式说明**：
> - **独立格式**：保函独立于基础合同，银行/担保公司独自承担责任，无需追偿
> - **连带格式**：保函与基础合同连带，享有先诉抗辩权

| 投标金额 | 投标总价 | 是 |
| 保函金额 | 担保金额 | 是 |

## 九、风险与依赖

### 9.1 项目风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 金融机构接口标准不统一 | 高 | 制定标准适配层，支持灵活配置 |
| 国密算法兼容性 | 中 | 充分测试，选择成熟方案 |
| 交易平台对接复杂 | 高 | 提前调研，预留充足对接时间 |
| 等保认证周期长 | 中 | 提前启动认证流程 |

### 9.2 业务风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 恶意索赔 | 高 | 严格审核索赔材料，建立黑名单机制 |
| 欺诈投保 | 中 | 接入信用评估，加强实名认证 |
| 保费逾期 | 中 | 建立逾期提醒，设置保函失效机制 |
| 机构跑路 | 高 | 保证金制度，多机构备选 |

### 9.3 合规风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 数据跨境 | 高 | 本地化存储，遵守数据安全法 |
| 电子签章合规 | 中 | 选择有资质的CA机构 |
| 反洗钱 | 中 | 客户身份识别，大额交易报告 |
| 隐私保护 | 中 | 脱敏处理，授权机制 |
