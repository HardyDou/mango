## 一、模块概述

保函出具模块是电子保函系统的核心功能模块，负责在投保申请通过审核后，生成具有法律效力的电子保函文档。本模块是连接投保申请与保函管理的关键环节。

---

## 二、功能需求

### 2.1 保函生成

| 功能 | 描述 | 优先级 | 负责人 |
|------|------|--------|--------|
| 保函模板管理 | 支持配置不同类型的保函模板 | P0 | 后端开发工程师 |
| 保函内容生成 | 根据投保申请信息填充保函模板 | P0 | 后端开发工程师 |
| 保函编号生成 | 生成唯一的保函编号（格式：BG+机构代码+年月日+序号） | P0 | 后端开发工程师 |
| 保函状态管理 | 管理保函状态（待出具、已出具、已推送、已下载） | P0 | 后端开发工程师 |
| 保函有效期 | 支持配置保函有效期（默认1年，最长不超过招标文件要求） | P0 | 后端开发工程师 |

**保函编号规则**：
- 格式：`BG{机构代码}{YYYYMMDD}{序号6位}`
- 示例：`BGICBC20260314000001`

**保函基本信息**：
| 字段 | 类型 | 说明 |
|------|------|------|
| guaranteeNo | String | 保函编号 |
| guaranteeType | String | 保函类型（投标/履约/预付款/质量） |
| projectName | String | 项目名称 |
| projectNo | String | 项目编号 |
| applicantName | String | 申请人名称 |
| applicantDocType | String | 申请人证件类型（身份证/营业执照/其他） |
| applicantDocNo | String | 申请人证件号码 |
| beneficiaryName | String | 受益人名称 |
| beneficiaryDocType | String | 受益人证件类型（身份证/营业执照/其他） |
| beneficiaryDocNo | String | 受益人证件号码 |
| guaranteeAmount | BigDecimal | 保函金额 |
| currency | String | 币种（默认CNY） |
| startDate | Date | 生效日期 |
| expireDate | Date | 到期日期 |
| issueDate | Date | 出具日期 |
| guaranteeContent | String | 保函内容 |

### 2.2 电子签章

| 功能 | 描述 | 优先级 | 负责人 |
|------|------|--------|--------|
| CA签章申请 | 向CA机构申请数字签名 | P0 | 后端开发工程师 |
| 签章验证 | 验证签章有效性 | P0 | 后端开发工程师 |
| 批量签章 | 支持批量签章处理 | P2（第二阶段） | 后端开发工程师 |
| 签章记录 | 记录所有签章操作 | P0 | 后端开发工程师 |

**签章流程**：
1. 生成保函OFD文档
2. 计算文档Hash（SM3算法）
3. 调用CA签章服务进行数字签名
4. 将签名信息嵌入OFD文档
5. 验证签章有效性

**签章请求参数**：
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| documentHash | String | 是 | 文档Hash值 |
| signerName | String | 是 | 签章人姓名 |
| signerCert | String | 是 | 签章人证书 |
| pageNo | Integer | 是 | 签章页码 |
| posX | Double | 是 | 签章X坐标 |
| posY | Double | 是 | 签章Y坐标 |

### 2.3 OFD文件生成

| 功能 | 描述 | 优先级 | 负责人 |
|------|------|--------|--------|
| OFD模板渲染 | 将保函内容渲染为OFD格式 | P0 | 后端开发工程师 |
| 版式设置 | 设置页面大小、边距、字体等 | P0 | 后端开发工程师 |
| 条码生成 | 生成保函防伪条码（使用保函编号作为条码内容） | P0 | 后端开发工程师 |
| 文件存储 | 存储生成的OFD文件 | P0 | 后端开发工程师 |

**OFD文档规范**：
- 页面尺寸：A4 (210mm × 297mm)
- 边距：上20mm，下20mm，左25mm，右25mm
- 字体：仿宋_GB2312
- 字号：正文12pt，标题16pt，加粗
- 条码位置：页面底部居中

## 三、状态流转

```
待出具 → 已生成 → 已签章 → 已推送/已下载
                    ↓
              签章失败 → 待重试
                    ↓
              推送失败 → 待重试
```

**状态说明**：
| 状态 | 说明 |
|------|------|
| PENDING_ISSUE | 待出具 |
| GENERATED | 已生成 |
| SIGNED | 已签章 |
| SIGN_FAILED | 签章失败 |
| PUSHED | 已推送 |
| PUSH_FAILED | 推送失败 |
| DOWNLOADED | 已下载 |

## 六、技术要求

### 6.1 后端技术要求

- 使用OFD生成库生成标准OFD文档
- 使用国密SM3算法计算文档Hash
- 集成CA签章服务进行数字签名
- 文件存储使用MinIO或阿里云OSS
- 推送服务支持异步处理

### 6.2 安全要求

- 传输加密：HTTPS + 国密SSL
- 文件签名：SM3 + SM2数字签名
- 访问控制：JWT Token + 权限验证
- 审计日志：记录所有操作

### 6.3 重试机制

**签章重试规则**：
- 最大重试次数：3次
- 重试间隔：指数退避（1分钟 → 2分钟 → 4分钟）
- 最终失败处理：状态变更为"签章失败"，需人工处理

**推送重试规则**：
- 最大重试次数：3次
- 重试间隔：指数退避（1分钟 → 2分钟 → 4分钟）
- 最终失败处理：状态变更为"推送失败"，需人工处理

### 6.4 CA签章服务商要求

- 必须具有《电子签名法》规定的合法资质
- 需提供有效期内的数字证书
- 签章服务响应时间需满足业务需求
