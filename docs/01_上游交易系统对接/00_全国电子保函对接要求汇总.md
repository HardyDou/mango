# 全国公共资源交易系统电子保函对接要求汇总

## 一、各地电子保函接口规范汇总

### 1. 已发布地方标准的省份

| 省份/城市 | 标准号 | 标准名称 | 状态 |

| **安徽(合肥)** | DB3401/T 308-2023 | 公共资源交易投标保证金电子保函平台数据规范 | 现行 |
| **辽宁** | DB21/T 4088-2025 | 建设工程领域电子保函基础公共服务平台数据和服务规范 | 现行 |

### 2. 常见对接模式

#### 模式一：省级统建模式（主流）
- **特点**：省建市用，一次接入全省通用
- **代表**：湖南、湖北、四川、安徽
- **优势**：统一接口标准，减少重复对接

#### 模式二：地市独立模式
- **特点**：各地市独立对接
- **代表**：早期各地市
- **问题**：对接成本高，标准不统一

#### 模式三：开放式架构（创新）
- **特点**：出函机构无需与交易系统对接，直接开函
- **代表**：苏州工业园区
- **优势**：降低对接成本，灵活度高

## 二、接口规范关键要素

### 1. 数据交换格式

| 要素 | 常见要求 |
|------|----------|
| **报文体格式** | JSON / XML |
| **字符编码** | UTF-8 |
| **签名算法** | SM2/SM3/SM4 (国密)、RSA、MD5 |
| **加密方式** | AES、RSA |
| **传输协议** | HTTPS + JSON |

### 2. 核心接口清单

| 接口名称 | 方向 | 说明 |
|----------|------|------|
| 投标申请 | 交易系统→金融机构 | 获取保函申请信息 |
| 保函开具 | 金融机构→交易系统 | 返回电子保函 |
| 保函验真 | 交易系统→金融机构 | 验证保函真伪 |
| 退保申请 | 交易系统→金融机构 | 申请解除保函 |
| 保函状态查询 | 交易系统→金融机构 | 查询保函当前状态 |
| 项目信息同步 | 交易系统→金融机构 | 项目基本信息推送 |

### 3. 电子保函文件格式

| 格式 | 说明 |
|------|------|
| PDF | 主流格式，需含电子签章 |
| OFD | 国产化替代方案 |
| XML | 结构化数据交换 |
| JSON | API接口传输 |

## 三、技术架构建议

### 1. 系统架构设计原则

```
┌─────────────────────────────────────────────────────────────┐
│                      交易系统 (上游)                        │

└─────────────────────────┬───────────────────────────────────┘
                          │ HTTP JSON API
                          ▼
┌─────────────────────────────────────────────────────────────┐

│   ┌──────────────┬──────────────┬──────────────┐          │
│   │ 接入管理      │ 业务处理      │ 运营管理      │          │
```

### 2. 多通道适配器设计

为适应各地不同接口规范，建议采用适配器模式：

```python
# 适配器接口
class GuaranteeChannelAdapter:
    def apply_for_guarantee(self, apply_info): ...
    def issue_guarantee(self, issue_info): ...
    def verify_guarantee(self, guarantee_no): ...
    def cancel_guarantee(self, cancel_info): ...

# 具体适配器
class HunanAdapter(GuaranteeChannelAdapter): ...
class AnhuiAdapter(GuaranteeChannelAdapter): ...
class XiamenAdapter(GuaranteeChannelAdapter): ...
class OpenArchitectureAdapter(GuaranteeChannelAdapter): ...
```

### 3. 数据库设计要点

| 表名 | 说明 |
|------|------|

## 五、实施建议

### 1. 优先级排序
1. **第一梯队**：本省/本市 - 深度定制对接
2. **第二梯队**：周边省份 - 标准接口适配
3. **第三梯队**：全国扩展 - 开放式架构

### 2. 快速接入方案
- 预置主流通道适配器（湖南、安徽、福建、辽宁标准）
- 配置化通道管理，支持新通道快速添加
- 支持"一次对接，全网通用"模式

### 3. 合规要求
- 等保三级认证（福建福州市要求）
- 国密算法支持（SM2/SM3/SM4）
- 电子签章兼容性（PDF签章、OFD签章）

## 六、SaaS模式与部署要求

### 1. SaaS模式支持情况

#### 市场需求
- **一套服务对接多个交易系统**：目前市场需求强烈，尤其适合担保公司/保险公司
- **多通道整合**：一个平台对接多家银行/担保公司/保险公司
- **降本增效**：避免重复对接，降低运维成本

#### 现有SaaS/多租户方案
| 厂商 | 部署模式 | 特点 |
|------|----------|------|
| **融都科技** | SaaS/私有化 | 恒生电子投资，一次对接多家金融机构 |
| **保函通** | 私有化部署 | 标准化接口体系，支持多通道 |
| **契约锁** | 公有云/私有化 | 多租户支持，API集成 |
| **开放签** | 开源/SaaS | 支持私有化部署、多租户、SaaS化 |

#### SaaS模式优势
- **快速上线**：开箱即用，缩短对接周期
- **低成本**：无需自建 infrastructure
- **弹性扩展**：按需扩容
- **统一管理**：多通道统一监控、运营

#### SaaS模式挑战
- **数据安全**：敏感金融数据存储第三方
- **定制化**：标准化与个性化需求冲突
- **合规性**：政务行业对数据本地化要求

### 2. 独立部署要求

#### 政务行业强制要求

| 要求类型 | 具体内容 | 适用范围 |
|----------|----------|----------|
| **数据本地化** | 数据必须存储在指定机房/政务云 | 全国政务系统 |
| **等保三级** | 必须通过等保三级认证 | 福建福州市等 |
| **国密算法** | 必须支持SM2/SM3/SM4 | 政务、金融 |
| **物理隔离** | 与互联网隔离或专线连接 | 涉密/高安全系统 |
| **国产化** | 支持国产服务器、操作系统、数据库 | 政务信创项目 |

### 3. 部署架构建议

#### 多租户架构设计

```python
# 多租户隔离方案
class MultiTenantArchitecture:
    # 方案一：数据库隔离
    def database_isolation(self):
        """每个租户独立数据库"""
        # 优点：数据完全隔离
        # 缺点：资源消耗大
        
    # 方案二：Schema隔离
    def schema_isolation(self):
        """同一数据库，Schema隔离"""
        # 优点：资源共享
        # 缺点：需要权限控制
        
    # 方案三：字段隔离
    def field_isolation(self):
        """同一表，通过tenant_id隔离"""
        # 优点：资源利用率高
        # 缺点：逻辑隔离需谨慎
```

#### 推荐部署模式

| 场景 | 推荐模式 | 说明 |
|------|----------|------|
| 政务系统 | 私有化部署 | 数据不出域，必须独立部署 |
| 银行/保险 | 私有化/SaaS | 视客户要求 |
| 担保公司 | SaaS为主 | 降低成本，快速接入 |
| 信创项目 | 国产化私有化 | 适配国产化环境 |

### 4. 特殊要求汇总

| 地区/场景 | 特殊要求 |
|-----------|----------|
| **福州市** | 等保三级认证、信息安全管理体系认证 |
| **政务系统** | 数据本地化、政务云部署 |
| **信创项目** | 国产化（国产CPU/OS/DB） |
| **金融监管** | 数据脱敏、日志留存 |
| **高安全** | 物理隔离、专线连接 |
