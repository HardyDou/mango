1. 概述 
1.1. 编写目的 
本文档定义了江西省公共资源交易平台金融服务子系统（以下简称“金融
服务子系统” ）与金融机构业务系统的接口交互规范，主要适用于投标保函，规
定了数据交互时所采用的数据传输协议、接口报文格式、数据加密方式、数据
接口定义等内容，金融机构业务系统可依据本文档与金融服务子系统开展系统
对接工作。 
1.2. 术语 
序号 名称 说明 
1 API API 应用程序编程接口(Application 
Programming Interface) 
2 HTTP HTTP 超文本传输协议(HyperText Transfer 
Protocol) 
3 电子保函 电子保函是保证人以使用 CA 证书进行电子签
名的数据电文为介质通过计算机网络向受益方
开立的具有法律效力的担保凭证。 
4 appKey 用于标识请求来源系统的唯一性 
5 appSecret 接口提供方给出的16 位密钥，用于加密。 
6 国密 SM3 国产散列函数标准，用于数字签名及验证。其
安全性及效率与SHA-256 相当。 
7 国密 SM4 国产分组密码标准，主要用于数据加密，属对
称加密。 
8 订单编号(orderNo) 定义为当用户提交开函意向申请时 金融服务子
系统自动生成的编号，用于关联该笔订单后续
的业务数据。当出现开函意向受理不通过后重
新提交意向申请的、撤单成功后重新提交意向
申请的、退函成功后重新提交意向申请的情
况，该订单编号将会重新产生。 
9 申请编号(applyNo) 定义为当用户在某个业务节点办理并提交由 金
融服务子 系统自动生成的编号。不同的业务 节
点办理，该申请编号是唯一且相对独立的。

2.3.2. 加密算法 
步骤如下： 
① 接口提供方给出长度为 16 位的 appSecret 密钥。 
② 请求方通过密钥对业务参数进行国密 SM4 对称加密算法，加密模式选择 CBC，
填充方式 PKCS5Padding。 
③ 经过 SM4 加密后的数据通过 BASE64 编码转换为明文的字符串。 
2.3.3. 签名方法 
步骤如下： 
① data 依照 2.3.2 进行加密。 
② 将 appKey，timestamp，requestId，data 参数按照 ASCII 升序排序，按照
“参数=参数值”的模式，用“&”字符拼接成字符串。 
③ 通过 appSecret 密钥，对上面字符串做 SM3 信息摘要，转换为 BASE64 字符
串作为签名传输。 
 
注：接收方接收到请求，将接收到的报文进行 SM3 摘要，比对接收到的摘要信
息结果，若相同说明报文传输安全，若不同说明报文经过篡改，传输不安全。
