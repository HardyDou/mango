1.前言
1.1 文档说明
本文档定义了交易系统与金融服务平台的接口交互方式、数据格式等。请相
关技术人员详细阅读本文档。
1.2 阅读对象
业务系统设计人员、开发人员及测试人员。
1.3 业务术语
无。
2.名词解释
2.1 交易系统公钥
从交易系统获取的，用于金融服务平台验证交易系统签名的 SHA256-RSA 算
法公钥，推送交易系统业务数据的加密。
2.2 金融服务平台公钥
金融服务平台在交易系统配置的，用于交易系统验证金融服务平台签名的
SHA256-RSA 算法公钥，推送平台业务数据的加密。
2.3 金融服务平台接口地址
金融服务平台配置在交易系统的用于接口交互的通信地址。
2.4 交易系统接口地址
交易系统对外提供的用于接口交互的通信地址。

2.5 时间戳
请求方提交数据时产生的时间标识，即格林威治时间 1970 年 01 月 01 日 00
时 00 分 00 秒(北京时间 1970 年 01 月 01 日 08 时 00 分 00 秒)起至现在的总秒数。
2.6 金融服务平台跳转地址
从交易系统选择平台后，跳转至金融服务平台的网址，交易系统将在跳转地
址的后面拼接参数，金融服务平台跳转地址?sectionId=1&code=fdsafd，如果选
择项目后跳转 sectionId 字段必填，没有选择项目直接跳转只传 code 即可。
3.接入方式
3.1 通信协议
所有接口基于 HTTP 协议进行通讯，通讯报文采用 JSON 格式，编码为 UTF-8，
双方的请求报文通过加密签名后传输给对方，提交请求使用 POST 方式。
3.2 加密算法逻辑
JSON 业务数据使用 RSA 算法，交易中心推送金融服务平台数据时，需要使
用金融服务平台的公钥加密业务数据，使用交易中心的私钥进行签名，金融服务
平台接到数据后使用交易中心的公钥进行验签，使用自己的私钥进行解密业务数
据，金融服务平台推送数据采用相同的逻辑。
3.3 签名算法
使用基于非对称加密秘钥的 SHA256-RSA 的数字签名算法。
3.4 签名
3.4.1 请求签名
请求发起方需要使用自身的私钥对消息体进行 SHA-256 with RSA 签名，请
求接收方必须使用请求发起方的公钥验证签名来保证请求的真实性和数据的完
整性。

3.4.2 签名生成
签名串交易中心推送数据一共有 3 行。金融服务平台推送数据一共有 2 行，
每一行为一个参数。行位以\n 结束，包括最后一行。如果参数本身以\n 结束，
也需要附加一个\n。金融服务平台签名串和交易系统的签名串有一个参数上的不
同，如下：
交易系统请求金融服务平台的签名串组成如下：
请求类型\n
时间戳\n
请求报文主体（加密前的 json）\n
金融服务平台请求交易系统签名串组成如下：
时间戳\n
请求报文主体（加密前的 json）\n
以金融服务平台请求交易系统为例，如请求业务参数{"acd":"测试"}，则签名串组
成如下：
1554208460\n
{"acd":"测试"}\n
得到待签名串后，使用私钥对代签名串进行 SHA256 with RSA 签名，并对签名结
果进行 Base64 编码得到签名值。
