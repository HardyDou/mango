# 下游出函机构接入对比分析报告

本文档对比分析各下游出函机构（银行、保险公司、担保公司）接入的相同与不同之处，为后续适配器设计方案提供参考。

---

## 一、机构类型总览

| 机构类型 | 产品形式 | 特点 |
|----------|----------|------|
| **银行** | 银行保函 | 直开式/分离式，需银行授信 |
| **保险公司** | 保证保险保单 | 银保监会批复，费率灵活 |
| **担保公司** | 担保保函 | 融资性/非融资性资质，需银行授信 |

---

## 二、相同点分析

### 2.1 业务流程相同

所有机构的**核心业务流程**是一致的：

```
投保申请 → 保费支付 → 保函/保单开具 → 保函验真 → 退保/注销
```

### 2.2 通讯协议相同

| 协议 | 说明 |
|------|------|
| HTTPS | 必须支持 |
| JSON | 主流格式 |
| UTF-8 | 编码要求 |
| POST | 请求方式 |

### 2.3 核心接口相同

| 接口 | 说明 |
|------|------|
| 投保申请 | 平台→机构 |
| 支付接口 | 平台→机构 |
| 保函/保单查询 | 平台→机构 |
| 退保申请 | 平台→机构 |
| 发票申请 | 平台→机构 |

### 2.4 电子签章要求相同

| 要求 | 说明 |
|------|------|
| 文档签名 | 必须覆盖整个文档 |
| 签章位置 | 页面右下角或指定位置 |
| CA证书 | 使用权威CA机构颁发 |
| 时间戳 | 包含可信时间戳 |

---

## 三、不同点分析

### 3.1 接口协议差异

#### 银行：交易码模式

光大银行采用**交易码+固定格式**：

```
┌─────────────────────────────────────────────────────────────────┐
│                      银行接口模式                                  │
├─────────────────────────────────────────────────────────────────┤
│  特点：                                                          │
│  - 固定交易码（LGA003开立, LGA008查询, LGA010注销）           │
│  - 字段相对固定                                                │
│  - 支持MFT文件传输                                             │
│  - 需银行分配CustomerNo、BchCod                                │
└─────────────────────────────────────────────────────────────────┘
```

#### 保险公司：保险业务模式

保险公司采用**保险业务术语**：

```
┌─────────────────────────────────────────────────────────────────┐
│                      保险公司接口模式                              │
├─────────────────────────────────────────────────────────────────┤
│  特点：                                                          │
│  - 投保人/被保人/受益人概念                                     │
│  - 保单号代替保函编号                                           │
│  - 保费计算因子（份数、人数、保额因子）                         │
│  - 支持见费出单/后费出单                                        │
└─────────────────────────────────────────────────────────────────┘
```

#### 担保公司：混合模式

担保公司通常复用银行或保险接口：

```
┌─────────────────────────────────────────────────────────────────┐
│                      担保公司接口模式                              │
├─────────────────────────────────────────────────────────────────┤
│  特点：                                                          │
│  - 分离式保函：担保公司担保+银行出函                             │
│  - 直开保函：担保公司直接出函                                    │
│  - 需对接银行授信额度                                            │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 安全机制差异

| 机构类型 | 加密方式 | 签名方式 | 密钥管理 |
|----------|----------|----------|----------|
| 光大银行 | 可选 | MD5/SHA1 | 线下分配 |
| 汉唐保险 | RSA | SHA1withRSA | 线下交换公私钥 |
| 其他机构 | SSL | MD5 | appKey+appSecret |

#### 光大银行安全机制

```json
// 公共参数
{
  "ChnlNo": "业务渠道",
  "CustomerNo": "客户号",
  "TransCode": "交易码",
  "BatchID": "流水号"
}
```

#### 汉唐保险安全机制

```json
// 网关公共参数
{
  "appKey": "应用授权key",
  "bizContent": "RSA加密业务数据",
  "signValue": "SHA1withRSA签名",
  "encryptType": "RSA"
}
```

### 3.3 产品类型差异

#### 保函类型代码

| 类型 | 光大银行代码 | 说明 |
|------|--------------|------|
| 投标保函 | A | BID BOND |
| 履约保函 | B | PERFORMANCE GUARANTEE |
| 预付款保函 | C | ADVANCE PAYMENT GUARANTEE |
| 质量保函 | D | QUALITY GUARANTEE |
| 农民工工资保函 | 1 | 特殊类型 |

#### 保险产品要素

| 要素 | 说明 |
|------|------|
| rationType | 产品代码 |
| premium | 保费 |
| amount | 保额 |
| insurPeriod | 保险期间 |
| insurYearFlag | 期间单位(D/M/Y) |

### 3.4 支付模式差异

| 模式 | 机构 | 说明 |
|------|------|------|
| 见费出单 | 大部分保险公司 | 支付保费后才出保单 |
| 后费出单 | 部分银行 | 先出函，后支付 |
| 授信模式 | 银行直开 | 银行授信，无需支付 |

### 3.5 业务流程差异

#### 银行：预审+开立两阶段

```
┌─────────────────────────────────────────────────────────────────┐
│                    银行两阶段流程                                 │
├─────────────────────────────────────────────────────────────────┤
│  阶段一：预审                                                    │
│  LGA001: 保函开立预审申请                                       │
│  LGA002: 预审结果查询                                           │
│                                                                  │
│  阶段二：开立                                                    │
│  LGA003: 保函开立申请                                           │
│  LGA006: 保函开立回执                                           │
└─────────────────────────────────────────────────────────────────┘
```

#### 保险公司：投保+支付分离

```
┌─────────────────────────────────────────────────────────────────┐
│                    保险业务流程                                    │
├─────────────────────────────────────────────────────────────────┤
│  1. 投保申请 (INPUT)                                           │
│  2. 返回支付链接 (payUrl)                                       │
│  3. 客户支付保费                                                │
│  4. 支付回调                                                    │
│  5. 查询电子保单 (QUERYEPOLICY)                                │
└─────────────────────────────────────────────────────────────────┘
```

### 3.7 资质要求差异

| 机构类型 | 准入资质 | 特殊要求 |
|----------|----------|----------|
| 银行 | 金融许可证、保函业务资质 | 需申请接入、分配机构代码 |
| 保险公司 | 保险许可证、保证保险产品备案 | 产品需银保监会批复 |
| 担保公司 | 融资性/非融资性担保许可证 | 需银行授信额度 |

## 七、结论

### 7.1 核心差异

| 差异类型 | 影响程度 | 处理方式 |
|----------|----------|----------|
| 机构类型 | 高 | 抽象接口+多实现 |
| 字段命名 | 高 | 统一领域模型+适配转换 |
| 安全机制 | 中 | 配置化切换 |
| 支付模式 | 中 | 策略模式 |
| 业务流程 | 中 | 适配器内部处理 |
| 产品代码 | 低 | 配置化映射 |

### 7.2 设计建议

1. **抽象机构接口**：银行、保险、担保分别实现
2. **统一领域模型**：申请人/受益人/金额/日期统一命名
3. **配置化机构管理**：新增机构只需配置，无需代码改动
4. **产品代码映射**：支持不同机构的产品代码转换
5. **两阶段流程支持**：预审+开立、投保+支付流程内置
6. **支付模式策略**：见费/后费/授信通过策略切换
