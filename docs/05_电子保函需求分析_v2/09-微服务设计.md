# 09-微服务设计

本文档说明电子保函系统的DDD领域划分与微服务设计。

---

## 9.1 DDD领域划分

### 9.1.1 领域划分原则

| 领域类型 | 定义 | 包含 |
|----------|------|------|
| **核心域** | 业务核心价值 | 保函业务 |
| **通用域** | 可复用能力 | 用户权限、机构、通道 |
| **支撑域** | 支持核心域 | 支付、发票、通知、风控、文件 |
| **基础设施** | 基础技术能力 | 网关、注册、配置 |

### 9.1.2 领域映射

```
┌─────────────────────────────────────────────────────────────────────────────┐

│   │                        核心域                                     │     │
│   │                     baohan-guarantee                            │     │

│   └─────────────────────────────────────────────────────────────────┘     │

│   │   通用域      │   通用域      │   通用域      │   基础设施   │          │
│   │ baohan-upmsp │baohan-institu│baohan-channel│ baohan-gate  │          │
```

## 9.3 核心域：baohan-guarantee

### 9.3.1 服务职责

| 模块 | 说明 |
|------|------|
| 投保申请 | 接收投保、处理投保逻辑 |

## 9.4 通用域

### 9.4.1 baohan-upmsp（用户权限服务）

| 功能 | 说明 |

| 协议管理 | 接口协议版本 |

| 额度控制 | 机构额度管理 |

### 9.5.5 baohan-file（文件服务）

| 功能 | 说明 |
|------|------|
| 文件存储 | 保函文件存储 |

## 9.8 分布式事务设计

### 9.8.1 方案选择

> 重要决策：电子保函系统无账户余额体系，不采用TCC模式

### 9.8.2 推荐方案

```
┌─────────────────────────────────────────────────────────────────┐
│                  分布式事务处理方案                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  方案一: 可靠消息 + 本地消息表 (推荐)                            │
│  ┌───────────────────────────────────────────────────────────┐   │
│  │  1. 投保申请 → 本地事务写入业务表 + 消息表               │   │
│  │  2. 发送消息到MQ                                          │   │
│  │  3. 消费消息处理下游业务（保函开具/通知）                 │   │
│  │  4. 定时扫描消息表，失败则补偿重试                        │   │
│  └───────────────────────────────────────────────────────────┘   │
│                                                                  │
│  方案二: Saga模式                                               │
│  适用：长流程、多步骤、允许逆向补偿                               │
│                                                                  │
│  方案三: 简单状态机                                             │
│  适用：流程短、步骤少、失败直接重试                              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```
