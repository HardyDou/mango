## Problem Statement

## Architecture Principles

**核心理念：业务代码与部署拓扑彻底分离。**

同一份业务代码，不改一行，可以：
- 部署为多个微服务（每个模块独立进程，通过 Feign 通信）
- 部署为一个单体（所有模块在同一个进程，本地调用）
- 部署为任意组合（把 A+B 合成一个服务，C 独立）

### 五个核心原则

```
原则 1: 业务代码不知道部署拓扑
       ↓
       业务模块的 *-core 只写业务逻辑
       *-starter 是"如何调用"的适配器
       *-starter-remote 是"跨进程调用"的实现

原则 2: gateway 与业务无关
       ↓
       同一个 gateway，可以接 WebFlux（响应式）
       也可以接 Servlet Filter（阻塞式）
       业务感知不到网关的存在

原则 3: 避免重复逻辑
       ↓
       单体和微服务的差异只在 wiring 层
       业务逻辑只写一次，在 *-core 中

原则 4: SPI 注入替代条件分支
       ↓
       不要 if (deployment == microservices) {...}
       要 @Autowired IXxxService — SPI 自动注入

原则 5: 基础设施按需切换，不绑死实现
       ↓
       Redis / Memory / DB 三种 KV 实现
       业务层只依赖抽象接口
       部署时根据环境（有没有 Redis / 有没有 DB / 是不是集群）自动决定
```

### 核心抽象：数据访问层（DAL）

```
业务层（*-core）
    ↓ 只依赖接口
当前 `mango-infra-kv` 只有基础的 KV 实现，需要扩展为完整的 IUseCase 体系。

**根抽象 IDataStore**（底层数据结构）：
```
├── KV        → get / set / setnx(key,ttl) / del / exists / expire(key,ttl)
├── Hash      → hset / hget / hgetall / hdel / hincrby
├── ZSet      → zadd / zrange / zrangebyscore / zrem / zcard
├── List      → lpush / rpush / lrange / lpop / llen
├── HyperLogLog → pfadd / pfcount
└── Geo      → geoadd / geopos / geodist
```

**业务场景接口 IUseCase**（依赖 IDataStore）：
```
├── ICache       → getCached / setCache(key,value,ttl) / evict / exists
│                   用途：会话缓存、查询缓存、热点数据加速
├── ILocker      → lock(key,ttl) / tryLock(key,ttl,timeout) / unlock
│                   用途：分布式锁（保障幂等、防止重复处理）
├── ITokenStore  → store(token,payload,ttl) / get(token) / revoke(token) / isRevoked
│                   用途：JWT 黑名单、刷新 token 存储
├── IIdempotent  → checkAndMark(key,ttl) → boolean
│                   用途：防重复提交（表单幂等、接口幂等）
├── ITimeWindow   → addAndCount(key,maxCount,windowSeconds) → count
│                   用途：时间窗口限流（登录频率、API 调用频率）
├── IRateLimiter → tryAcquire(key,permits,ttl) → boolean
│                   用途：令牌桶限流
├── IBloomFilter → add(key,value) / mightContain(key,value) → boolean
│                   用途：数据去重、重复检测
├── IDistributedCounter → incr(key,delta) / decr(key,delta) / get(key)
│                   用途：统计计数、排行榜
└── ISession     → setSession(userId,attrs,ttl) / getSession(userId) / invalidate(userId)
                    用途：用户会话管理
```

**SPI 注入 + 环境自动决策**：

| IUseCase | 有 Redis | 无 Redis 有 DB | 什么都无 |
|---------|---------|--------------|---------|
| ICache | RedisCache | DbCache | MemoryCache |
| ILocker | RedisLocker | DbLocker | MemoryLocker（单机） |
| ITokenStore | RedisToken | DbToken | MemoryToken |
| IIdempotent | RedisIdempotent | DbIdempotent | MemoryIdempotent |
| ITimeWindow | RedisTimeWindow | DbTimeWindow | MemoryTimeWindow |
| IRateLimiter | RedisRateLimiter | 令牌桶算法 | MemoryRateLimiter |
| IBloomFilter | RedisBloomFilter | - | MemoryBloomFilter |
| IDistributedCounter | RedisCounter | DbCounter | AtomicLong（单机） |

**TTL 是所有抽象的第一等公民**：每个接口方法都强制携带 ttl 参数，Memory/DB 实现用时间戳对比，Redis 实现用原生 EXPIRE。

**强制约束：模块代码必须严格遵守规范，否则部署无关性失效。**

`mvn mango:check` 必须检测以下违规，违规则构建失败：
1. 禁止 `new RedisLockerImpl()` — 必须 `@Autowired ILocker`
2. 禁止 `redisTemplate.opsForZSet()` — 必须注入 IZSet
3. 禁止硬编码 TTL — 必须从配置读取或传入 ttl 参数
4. 禁止 `if (hasRedis())` 条件判断 — 一切由 SPI 决定

## Premises

1. **业务与部署分离** — 同一份代码支持单体/微服务/任意聚合，不改业务代码
2. **Gateway 协议无关** — WebFlux 和 Servlet Filter 共存，业务不感知
3. **SPI 注入** — 部署拓扑通过配置注入实现，不用 if-else 条件分支
4. **业务代码先行** — 先写业务代码，后决定部署拓扑
5. **脚手架核心价值** — 让 AI Agent 高效率实现业务需求

## Approaches Considered

### Approach A: 脚手架优先（当前方向）
  Summary: 继续完善 mango 脚手架，补齐规范、CLI、代码生成器，然后基于脚手架开发电子保函业务
  Effort: M
  Risk: Low
  Pros: 技术债务少、AI 友好、可复用
  Cons: 业务价值显现慢
  Reuses: 已有 mango 脚手架基础

### Approach B: 业务优先
  Summary: 直接开发电子保函业务功能，脚手架在业务开发中逐步完善
  Effort: L
  Risk: Medium
  Pros: 业务价值显现快
  Cons: 技术债务积累、AI 协作效率低
  Reuses: 无

### Approach C: 并行推进
  Summary: 脚手架和业务功能并行开发，通过 Sprint 机制交替交付
  Effort: XL
  Risk: Medium
  Pros: 脚手架不成为瓶颈
  Cons: 团队认知负荷大、上下文切换成本高

## Recommended Approach

**Approach A: 脚手架优先**，理由：
1. mango 脚手架已有良好基础（D23-D29），继续完善成本低
2. 电子保函业务的核心流程（保函生命周期）需要稳定的底层支持
3. AI Agent 高效协作依赖于脚手架的规范和 CLI 工具
4. 当前阶段目标清晰：完善脚手架 + 落地用户/权限等通用模块
