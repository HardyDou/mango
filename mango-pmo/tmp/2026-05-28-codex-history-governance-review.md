## 3. AI 经常出现的问题

1. 没有把用户连续补充的约束收敛成硬规则，导致“记录一下，别忘记”类要求反复出现。
2. 对 PMO 触发范围判断不够精确：简单查看和正式交付容易混淆；正式任务又可能漏读必需规范。
3. preflight 规则加载偏粗，一命中前端或后端就加载整套规范，增加执行噪声和上下文负担。
4. 复杂需求未先拆解业务对象、字典/枚举、数据模型、接口模型、设计模式和验收项，就进入实现。
5. 前端页面容易按字段顺序或复制页面推进，未稳定遵守搜索区、功能区、列表区、分页区、表单分组和状态标签规则。
6. 公共组件容易按当前仓库可用实现，未从 npm 独立消费、单体部署、微前端部署和业务复用角度校验。
7. 后端容易在 API、core、starter、starter-remote、support 边界上摇摆，出现契约污染、远程适配和模块信息职责不清。
8. 发布任务容易漏账号口径、制品范围、版本验证、仓库检查和发布后回查。
9. 收尾动作容易漏：PR 合并后 worktree 清理、数据库清理、服务停止、分支状态确认、最终风险说明。
10. 验证口径容易过重或过轻：前端小改反复 build，或真实接口、数据库、权限链路未接通却表达成完成。

## 4. 用户侧协作错误

1. 指令经常分多条追加，关键约束被埋在聊天后半段，容易造成 AI 只执行前半段。
2. 范围词有时过短，例如“可以”“更新代码”“发布组件”，缺少对象、分支、版本、仓库和验收口径。
3. 中途会混入无关文本或错别字，增加解析噪声。
4. 对合并、发布、清理这类高风险操作，有时未一次性说明目标 worktree、数据库、远端分支和是否需要 PR。
5. 需求会边做边细化，但未及时转成文档或规范，导致后续会话重复解释同一约束。
6. UI 需求常以局部问题提出，未先给出字段关系、联动关系、数据来源和状态枚举。
7. 后端架构判断常以片段问题出现，缺少本次要固化的边界规则和例外条件。

## 5. PMO 优化结论

1. `mango-pmo` 继续作为唯一长期规范源；本文件只作为临时复盘，不作为长期规则源。
2. preflight 必须从“全局 always + 大包 bundle”改为“最小基线 + 精确专题命中”。
3. `rules/00-dev-flow.md` 只保留全局流程、worktree、提交和文档归属，不复制领域规范。
4. `rules/01-delivery-contract.md` 只在正式交付、设计驱动、台账驱动或用户明确要求完整交付时加载。
5. 前端规范按任务精准加载：页面/UI 加载 Element Plus；公共组件加载组件规范和 Monorepo；测试只在测试任务加载。
6. 后端规范按影响面加载：API、DB、security、persistence、module、test、versioning 分别由路径和关键词触发。
7. 规则正文保留全部信息，但优先去重、合并重复语义，减少“同一规则多处维护”。
8. 发布规则必须区分 Maven 后端发布和 npm 前端发布；npm 发布继续复用 Maven/Nexus 账号。
9. 纯规范、设计文档、复盘文档等不影响服务代码的治理任务，可以在 `main` 直接修改提交。

## 6. 后续检查建议

- 为 PMO preflight 增加代表性样例验证，覆盖前端页面、前端组件、后端 API、数据库、发布、治理。
- 将可自动判断的规则逐步沉淀为 lint、脚本或 review checklist。
- 每次用户纠正“不是这个意思”后，若属于长期约束，必须立即归并到 `mango-pmo` 对应规范。
