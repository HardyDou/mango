# 详细设计模板规范

## 1. 定位

- 本规范适用于 Mango 框架、平台能力、模板、CLI、前端包等 Mango 开发任务。
- 本规范适用于基于 Mango 的企业业务项目、业务模块和业务后台开发任务。
- 详细设计首先是 `For AI`：让 AI 能把 PRD 转成可开发、可验证、可交付的实现路径。
- 详细设计其次是 `For Human`：帮助人审查 AI 的理解是否完整、设计是否无歧义、落地是否会偏离需求。
- 详细设计承接 PRD，说明系统如何落地业务需求。
- 详细设计必须从关键对象和关键业务流程展开，不能只写页面或接口。
- 详细设计必须写清状态机、数据流、关键技术、异常边界和验收映射。
- 详细设计可以包含技术内容；PRD 中禁止出现的技术细节应在本规范约束下进入详细设计。
- 详细设计必须保留 PRD 编号追踪关系，不得改写或丢弃 `BO/BF/BR/PG/AC` 编号。
- 详细设计的质量标准不是“技术说明完整”，而是 AI 能否按设计拆解任务、实现功能、验证结果，并把每个交付物追溯回 PRD。

## 2. AI 设计前置规则

- 详细设计开始前必须检查 PRD 是否包含目标用户、业务用户类型、不处理范围、业务术语、关键对象、关键业务流程、业务规则、页面原型、字典展示项、追踪矩阵和验收标准。
- PRD 缺失上述任一项时，必须先输出“PRD 缺口清单”，不得自行补齐后继续设计。
- 缺口检查必须检查内容，不只检查章节标题；`BO/BF/BR/PG/AC` 为空、无引用、无用户可见断言或追踪断裂时，必须 `STOP` 或 `ASK`。
- 详细设计动作必须标明 `STOP`、`ASK`、`WRITE` 或 `NEXT`：阻断缺口用 `STOP`，需要业务/设计确认用 `ASK`，可写设计正文用 `WRITE`，自检全通过后进入开发用 `NEXT`。
- 详细设计中新增的接口、数据、权限、租户、配置、状态或组件，必须能追溯到 PRD、用户确认或现有系统约束。
- 无法追溯的内容必须写入“待确认设计问题”，不得写入正式设计结论。
- 新增技术结论必须登记来源证据：来源类型、来源路径/ID、是否推断、推断依据、是否阻断。
- 详细设计进入开发前，必须明确影响模块、接口变化、数据变化、权限/租户边界、初始化数据来源、异常边界和验证方式；缺任一项视为设计阻断。
- 生成详细设计时使用 `mango-pmo/templates/detailed-design.md`；本规范中的“章节说明、编写要求、关联规范”是写作约束，不复制到最终详细设计正文。

- 来源证据
- 影响模块与改动边界
- 关键对象
- 关键对象关系
- 关键业务流程
- 状态机设计
- 数据流设计
- 页面与功能映射
- 菜单、页面与权限资源设计
- 字典与配置设计
- 前端复用判断
- 关键技术
- 开发任务映射
- 异常与边界
- 验收映射
- 交付台账候选项
- 风险与取舍

## 5. PRD 与详细设计对应关系

| PRD 内容 | 详细设计内容 | 要求 |
|---|---|---|
| 关键对象 | 关键对象、关键对象关系 | 需求侧写业务含义，设计侧写模型关系和实现边界。 |
| 关键业务流程 | 关键业务流程、状态机设计、数据流设计 | 需求侧写用户流程和业务规则，设计侧写系统流程、状态流转和异常分支。 |
| 菜单与页面原型 | 页面与功能映射、通用前端组件 | 需求侧写用户看到的页面，设计侧写功能映射和复用组件。 |
| 字典与展示项 | 字典与配置设计 | 需求侧写业务选项和文案，设计侧写编码、来源、初始化和兼容。 |
| 状态与反馈 | 状态机设计、异常与边界 | 需求侧写用户可见状态，设计侧写完整状态图和处理规则。 |
| 验收标准 | 验收映射 | 需求侧写用户可见结果，设计侧写交付物和验证方式。 |
| 无 | 关键技术 | 技术设计专属。 |

## 6. 禁止事项

- 只有页面设计，没有关键对象。
- 只有接口设计，没有关键业务流程。
- 只有状态枚举，没有完整状态图。
- 只有数据表，没有数据流。
- 只有正常流程，没有异常和边界。
- 只有技术方案，没有 PRD 验收映射。
- 用详细设计替代 PRD，或把 PRD 的页面原型全文复制到详细设计。
- 静默遗漏 PRD 编号项，或用一个“主流程通过”覆盖多个独立验收项。
