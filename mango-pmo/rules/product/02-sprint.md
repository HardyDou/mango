# Sprint 规范

## Sprint 定义

- 每个 Sprint = 小且完整的可交付功能
- 不能交付一半就停止
- Sprint 周期：建议 1-2 周

## 用户故事格式

```
作为 [角色]
我想要 [功能]
以便 [价值]
```

## 分解规则

| 规则 | 说明 |
|------|------|
| 只描述 WHAT | 不描述怎么实现 |
| 完整功能 | 每个 Sprint 可独立交付 |
| MVP 优先 | 先做最小可用版本 |
| 渐进式开发 | 每次迭代完善功能 |

## 任务粒度

| 类型 | 限制 |
|------|------|
| 后端任务 | ≤ 50 行/任务 |
| 前端任务 | ≤ 100 行/任务 |

## Sprint 验收标准

每个 Sprint 交付必须包含：

- 功能代码
- 单元测试（覆盖率 ≥ 80%）
- E2E 截图验证
- 代码规范检查通过

## 数据库迁移规范

DDL 变更必须通过 Flyway migration 文件，禁止直接修改生产数据库。

### 迁移文件命名

```
db/migration/{module}/V{version}__{description}.sql
```

| 示例 | 说明 |
|------|------|
| V1__init.sql | 初始化建表 |
| V2__add_column.sql | 新增字段 |
| V3__seed_data.sql | 种子数据 |

### 创建规则

1. 新增不修改：每次 DDL 变更新增一个 migration 文件
2. 版本递增：V1 → V2 → V3
3. 模块隔离：每个领域模块独立子目录
