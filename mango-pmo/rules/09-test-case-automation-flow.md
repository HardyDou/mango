# 测试用例与自动化工程化流程

## 1. 定位

- 本文件约束测试用例从需求验收项登记、设计映射、自动化实现、执行分层到结果基线沉淀的全流程。
- PRD 只定义用户可见验收项；测试用例、自动化层级、测试命令、脚本路径和结果基线在设计、交付台账和验收证据中登记。
- 后端测试细节遵循 `rules/backend/08-test.md`；前端测试细节遵循 `rules/frontend/04-test.md`；测试资产目录遵循 `rules/04-test-assets.md`。

## 2. 生命周期门禁

1. 需求阶段必须为关键对象、关键流程、业务规则、页面和主要动作建立 `AC-xxx` 验收项。
2. 设计阶段必须把每个 `AC-xxx` 映射到测试用例候选项，明确测试层级、优先级、自动化判断、测试数据和证据类型。
3. 开发阶段必须同步实现自动化所需稳定测试契约；前端页面至少覆盖核心页面、业务动作、业务字段、业务记录和业务状态锚点。
4. 验证阶段必须按优先级执行自动化测试或手工验收，并把结果写入验收证据和测试结果基线。
5. 发布或 PR 前必须说明新增、修改、废弃的测试用例，以及 E2E 脚本和测试结果基线是否同步。

## 3. 用例登记字段

测试用例候选项至少登记：

| 字段 | 要求 |
|---|---|
| 用例 ID | 使用 `TC-xxx`，长期自动化用例必须稳定 |
| 来源 | 引用 `AC-xxx`，必要时同时引用 `BO/BF/BR/PG` |
| 场景 | 一个用户目标或一个业务规则结果 |
| 优先级 | `P0`、`P1`、`P2` 或 `P3` |
| 测试层级 | 单元、API、组件、E2E、截图、手工 |
| 自动化判断 | `AUTO`、`MANUAL` 或 `EXCEPTION` |
| 测试数据 | 数据来源、唯一标识、清理策略 |
| 稳定契约 | 语义锚点、API fixture、测试账号或环境要求 |
| 执行入口 | 测试文件、标签、命令或手工步骤 |
| 证据 | 报告、截图、trace、日志或人工记录路径 |
| 状态 | `CANDIDATE`、`AUTOMATED`、`MANUAL`、`FLAKY`、`DEPRECATED` |

## 4. 用例颗粒度

- 一个长期 E2E 用例只能覆盖一个用户目标和一个可验证业务结果。
- 一个用例可以包含必要的前置、操作和清理步骤，但不得把多个独立能力合并成“主流程通过”。
- 表单校验、局部组件状态、接口参数校验和纯业务规则优先放到组件、API 或单元测试；不得为了覆盖率堆 E2E。
- 只有排序、排名、分页位置、拖拽顺序本身是业务目标时，才允许按第几个断言。

优先级定义：

| 优先级 | 定义 | 执行要求 |
|---|---|---|
| `P0` | 登录、权限、租户、核心入口和核心业务可用性 | 每次 PR 或 AI 交付优先执行 |
| `P1` | 关键业务闭环和高风险回归 | PR 按影响范围执行，主干定时全量执行 |
| `P2` | 边界、历史缺陷、复杂组合和低频流程 | 夜间或发布前执行 |
| `P3` | 探索、临时诊断或一次性验证 | 不作为长期资产，任务结束后删除或升级 |

## 5. 自动化层级选择

- E2E 只验证用户目标、路由、登录态、接口联动、权限/租户边界和关键业务闭环。
- API 测试优先验证数据规则、权限规则、状态流转、异常分支和稳定接口契约。
- 组件测试优先验证表单、弹窗、表格、空态、错误态、加载态和交互状态。
- 截图或视觉检查用于页面布局、遮挡、溢出、空白、错误页和明显样式异常。
- 手工验收只用于无法稳定自动化或成本高于收益的场景，必须写明不可自动化原因和证据要求。

## 6. 前端自动化稳定契约

前端 E2E 必须优先使用业务语义锚点：

| 属性 | 用途 | 示例 |
|---|---|---|
| `data-page` | 页面能力 | `menu.management` |
| `data-surface` | 业务区域、表单、弹窗或抽屉 | `menu.form` |
| `data-action` | 业务动作 | `menu.create` |
| `data-field` | 业务字段 | `menu.name` |
| `data-record-key` | 业务记录、行、卡片或树节点 | `menu:e2e-menu-001` |
| `data-state` | 业务状态 | `ready`、`empty`、`error` |

规则：

- 锚点按业务语义命名，禁止按布局、顺序、组件结构或样式命名。
- 页面布局变化、按钮位置变化、弹窗改抽屉、表格改卡片时，业务语义锚点不应改名。
- 删除或改名长期锚点视为测试契约变化，必须同步测试用例、脚本和结果基线。
- E2E `specs` 中禁止直接依赖 `.el-*`、布局 class、深层 DOM、`nth()`、`first()`、`last()`、`waitForTimeout()` 和 `force: true`。
- 无法避免的 DOM 细节只能封装在 `support`、`fixtures`、`tasks` 或 `components` 中，并写明原因。

## 7. 复用结构

长期自动化测试按以下层次复用：

| 层次 | 责任 |
|---|---|
| `fixtures` | 登录态、租户、测试账号、测试数据、浏览器上下文和清理 |
| `tasks` | 业务动作，例如创建菜单、发布流程、发起审批 |
| `selectors` | 语义定位器和少量兼容封装 |
| `assertions` | 业务结果、页面状态和错误检查 |
| `specs` | 只表达用例目标、步骤和断言，不写复杂 DOM 查找 |

禁止把临时 workaround、具体测试断言或一次性数据构造过度抽成公共复用能力。

## 8. 测试数据和隔离

- 自动化用例必须创建或声明自己的测试数据，不依赖人工预置数据、其它用例残留状态或执行顺序。
- 写入数据必须使用可识别唯一前缀，例如 `E2E_`、`IT_`、`TC_<用例ID>_`。
- 用例必须提供前置清理或后置清理；清理只能处理本用例创建的数据。
- 涉及真实后端、租户、权限、支付、工作流、文件或消息时，必须登记账号/租户标识、数据集和环境边界。

## 9. 执行策略

| 场景 | 最低执行要求 |
|---|---|
| 本地开发 | 受影响模块单元/API/组件测试，加受影响 `P0/P1` 单条或标签用例 |
| AI 交付 | 静态检查、受影响测试、受影响页面截图检查、相关 `P0` E2E |
| PR | `P0` 全量，受影响模块 `P1`，必要时截图检查 |
| 主干定时 | `P0` 和 `P1` 全量 |
| 夜间 | `P0`、`P1`、`P2` 全量，可包含多浏览器 |
| 发布前 | 发布影响范围内 `P0/P1` 必须通过，`P2` 按风险执行或登记例外 |

默认日常 E2E 使用 Chromium；Firefox/WebKit 放到夜间、发布前或兼容性专项。CI 重试通过仍必须记录 flaky 风险，不能视为稳定资产。

## 10. 用例资产维护

- 自动化用例必须支持按单用例、标签/能力组合和全量套件执行。
- Playwright 用例标题必须带优先级和模块标签，例如 `@p0 @menu 平台管理员可维护菜单`。
- 后端 E2E 使用 JUnit 5 `@Tag`，至少包含 `e2e` 和能力标签。
- `FLAKY` 用例必须登记原因、证据和修复计划；连续影响交付时必须降级、隔离或删除。
- 业务废弃时删除用例；业务路径变化时更新用例；只是布局变化时应优先维护锚点或 task 封装。
- 新增回归缺陷时，必须补充对应用例候选项，并判断自动化层级。

## 11. 禁止事项

- 禁止需求没有验收项就进入详细设计。
- 禁止设计没有测试层级映射就进入开发。
- 禁止把所有验收项默认自动化成 E2E。
- 禁止把只打开页面、接口 200、截图有画面或无 console error 当作完整验收。
- 禁止用不稳定脚本冒充长期测试资产。
- 禁止自动化测试向共享业务库写入不可清理脏数据。
