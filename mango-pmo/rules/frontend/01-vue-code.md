# Vue 代码规范

## 1. 基本规则

- 使用 Vue 3。
- 使用 `script setup`。
- 使用 TypeScript。
- 不使用 `any`。
- 新增页面和组件优先复用现有能力。

## 2. 组件规则

- 组件名使用 `PascalCase`。
- 页面组件放 `views`。
- 只在当前页面使用的组件放页面目录。
- 可跨模块复用的组件放对应 `packages/*`。
- 不在 `apps` 和 `packages` 里维护两份相同组件。

## 3. 组件使用规则

- 优先直接使用 Element Plus 原生组件。
- 只有存在稳定复用价值时才封装组件。
- 公共组件禁止直接依赖基座 store、router、菜单、启动逻辑。
- 公共组件需要的数据和行为通过 `props`、`emits`、回调传入。

## 3.1 页面布局规则

- 页面布局必须优先使用 Element Plus 官方标准布局和组件用法，参考 `https://element-plus.org/zh-CN/`。
- 栅格布局使用 `el-row`、`el-col` 和 24 分栏体系；需要间距、对齐或响应式时使用 `gutter`、`justify`、`align`、`xs/sm/md/lg/xl` 等标准属性。
- 表单页面使用 `el-form`、`el-form-item`、`label-width`、`el-row`、`el-col` 组织字段；禁止用零散 `div` 拼接表单布局。
- 列表、详情、弹窗、抽屉等后台页面优先使用 Element Plus 的 `el-table`、`el-descriptions`、`el-dialog`、`el-drawer`、`el-card` 等标准结构。
- 禁止在业务页面自创非标准布局体系、随意嵌套卡片、用大段临时 CSS 修正本可由 Element Plus 标准布局解决的问题。

## 3.2 组件示例规则

- 组件库示例页必须使用 Element Plus 文档风格。
- 每个常见用法都必须是独立示例，示例下方提供可展开/收起的代码。
- 示例覆盖基础用法、常用变体、禁用/只读、业务集成和边界场景。
- 示例页底部必须提供属性、slot、方法/事件、对象字段或接口返回字段明细。
- 示例文案只说明真实用法和约束，不写与组件无关的营销描述。
- 同一类能力优先由一个组件通过 `props`、`slots`、`events` 支撑，禁止为了展示多个场景拆出多个前端组件作为长期实现。

## 4. 类型规则

- 对象结构优先使用明确类型。
- API 类型与页面状态类型分开定义。
- 公共类型放公共位置。
- 页面私有类型不要放到公共包。
- 后端 `Long`、雪花 ID、数据库主键、业务主键等标识字段在前端统一使用字符串语义，公共类型使用 `ApiId`。
- 禁止对 ID 调用 `Number(id)`、`parseInt(id)` 后再用于接口请求、路由参数、表格 key、组件 model 或文件 token。
- 只有页码、数量、金额、排序等真实数值字段可以建模为 `number`。

## 4.1 文件上传与回显规则

- 业务表单提交给业务 API 时，只能提交文件中心标识，例如 `fileId`、`fileIds`、`attachmentIds`、`mango-file:{id}` token。
- 上传组件返回的 `url`、`previewUrl`、`downloadUrl`、`directPreviewUrl`、`directDownloadUrl` 只允许用于当前页面即时展示，禁止写入业务表单 payload。
- 业务页面回显图片、附件、图标时，应按文件 ID 通过 `@mango/file` 或 `mango-file` API 获取临时预览/下载地址。
- 禁止把 MinIO/S3/OSS/COS/Kodo 直连地址、预签名 URL、文件服务下载 URL 存入业务字段、前端缓存或配置 JSON 作为长期数据。

## 4.2 真实数据规则

- 页面数据必须来自真实 API、配置、字典服务、枚举契约或可替换 provider。
- 字典、枚举、菜单、权限、租户默认数据、状态映射和业务配置不得在页面中写成最终静态数据。
- Mock 只能用于本地开发或测试，不能作为交付完成依据。
- 后端接口已具备能力时，前端必须列出 mock 替换清单并完成真实联调。
- 页面交付必须验证新增、编辑、搜索、回显、空结果、接口错误和权限不足场景。

## 5. 样式规则

- 优先使用现有设计变量和主题变量。
- 不直接覆盖 Element Plus 全局样式。
- 不为一个页面临时定义新的全局设计规则。
- 第三方组件样式必须显式引入。

## 6. Store 规则

- 基座 store 只服务基座。
- 公共包禁止直接依赖基座 store。
- 跨包状态通过契约、参数、事件传递。

## 7. 路由规则

- 路由按模块组织。
- 动态路由逻辑集中管理。
- 页面组件不直接写权限装配逻辑。

## 8. 禁止事项

- 为简单按钮、输入框做无意义二次封装
- 在公共组件里引用基座代码
- 同一能力在多个目录重复实现
- 直接把页面临时样式扩散成全局规则
- 提交临时代码、止血代码、绕过式兼容代码作为最终实现
- 为了规避构建或运行错误，给 import、类型、接口参数增加没有架构含义的特殊写法
- 在没有定位根因和完成验证前，把“能跑起来”的改动交付
- 业务表单把文件访问地址当作业务数据提交
- 用页面静态数组、mock 数据或硬编码状态替代真实 API、配置、字典服务或枚举契约

## 9. 临时代码处理规则

- 临时代码只能用于本地定位问题，提交前必须删除或重构为正式方案。
- 正式方案必须说明能力归属、依赖方向和复用边界。
- 单体和微前端共用的菜单、权限、图标、运行时能力必须沉淀到 `packages/*`，禁止通过 app 私有文件互相中转。
- 对运行态问题的修复必须同时验证开发态和生产构建态。
- 如果确实需要过渡兼容，必须有明确删除条件、关联任务和注释说明，不允许无期限保留。
