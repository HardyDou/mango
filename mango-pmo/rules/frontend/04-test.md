# 前端测试规范

## 1. 基本要求

- 有前端改动，必须有对应验证。
- 新增交互，必须覆盖主要流程。
- 修复缺陷，必须补回归验证。

## 2. 测试层次

- 单元测试：工具函数、组合式函数、独立逻辑。
- 组件测试：组件渲染、输入输出、事件。
- E2E 测试：页面主流程、路由、接口联动。
- 是否需要视觉截图，按改动影响面决定。

## 3. 页面回归走查

涉及页面、菜单、路由、样式、组件集成、微前端、运行时配置或打包方式的改动，必须做页面细节走查。

必须检查：

- 截图留存：每个关键页面至少保留一张截图，截图路径写入交付记录。
- 页面内容：不能只验证 URL 打开；必须确认主内容区域有业务内容，不是空白、404、加载失败或无限 loading。
- 框架区域：左侧菜单、顶部栏、面包屑、用户区、通知入口和当前菜单激活状态符合预期。
- 样式细节：按钮、图标、表格、表单、弹窗、卡片、工具栏位置和形状不能明显跑偏。
- 交互细节：搜索、新增、编辑、提交、弹窗打开关闭、页签或菜单切换等关键动作至少覆盖主路径。
- 数据状态：正常数据、空状态、接口错误或权限不足必须有明确页面反馈。
- 控制台与网络：记录 console error、Vue error、资源 404、接口 4xx/5xx；存在错误时不得声明页面通过。
- 部署形态：单体和微前端共用页面、组件或样式时，必须分别验证对应形态。

禁止把“服务已启动”“路由可访问”“截图有画面”当作页面验收通过。交付结论必须写明具体页面、关键元素、异常记录和截图路径。

## 4. 测试范围

至少覆盖：

- 正常场景
- 空状态
- 错误场景
- 关键交互
- 权限或路由限制

## 5. 命名规则

- 测试文件使用 `*.spec.ts` 或 `*.test.ts`。
- 用例名称直接描述场景和结果。

## 6. 提交前检查

至少执行与改动范围对应的检查：

- `pnpm test`
- `pnpm build`
- `pnpm playwright test`

## 7. 禁止事项

- 页面改动没有任何验证
- 只测 happy path
- 发现回归后不补测试
- 没有截图和页面细节结论就声明页面通过
