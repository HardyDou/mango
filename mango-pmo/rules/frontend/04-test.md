# 前端测试规范

## 1. 基本要求

- 有前端改动，必须有对应验证。
- 新增交互，必须覆盖主要流程。
- 修复缺陷，必须补回归验证。

## 2. 测试层次

- 单元测试：工具函数、组合式函数、独立逻辑。
- 组件测试：组件渲染、输入输出、事件。
- E2E 测试：页面主流程、路由、接口联动。
- 涉及页面、菜单、布局、主题、能力组合或 Admin Shell 的阶段性交付，必须执行 E2E 截图验收。

## 2.1 阶段验收强制规则

每个阶段完成后，必须同时完成回归测试和新特性测试。未完成以下验收，不得声明阶段完成：

- 回归测试：覆盖本阶段影响范围内的既有页面、菜单、权限、接口联动、路由跳转和核心业务流程。
- 新特性测试：按本阶段设计目标逐项验证新增能力，不得只验证 happy path 或只验证页面可打开。
- E2E 截图识别：涉及前端页面时必须保存截图，并识别页面菜单、UI 布局、颜色、间距、遮挡、横向溢出、弹层、抽屉、下拉、空/加载/错误态是否符合预期。
- 数据和功能：必须验证真实接口、真实菜单、真实权限、真实数据回显和关键交互；未接真实接口或真实数据时必须标记为未验证或 `EXCEPTION`。
- 一级菜单抽查：每个受影响或可见的一级菜单，必须随机抽查 1-3 个子页面；不足 3 个时全部抽查。
- 一级功能抽查：每个受影响的一级功能，必须随机抽查 1-3 个子功能或关键操作；不足 3 个时全部抽查。
- 证据留存：必须保留 E2E 命令输出、截图、layout/report、抽查清单、失败接口/console error 记录和人工截图识别结论。
- 交付台账：阶段台账中必须写明回归测试、新特性测试、菜单/页面抽查、功能抽查和截图识别的证据路径。

抽查不能替代阶段目标验收。阶段目标列明的页面、能力、接口、数据、权限或模式矩阵必须逐项验证。

## 3. 测试范围

至少覆盖：

- 正常场景
- 空状态
- 错误场景
- 关键交互
- 权限或路由限制
- 菜单显示、菜单跳转和页面挂载
- UI 布局、主题颜色、弹层/抽屉/下拉和响应式边界

## 4. 命名规则

- 测试文件使用 `*.spec.ts` 或 `*.test.ts`。
- 用例名称直接描述场景和结果。

## 5. 提交前检查

至少执行与改动范围对应的检查：

- `pnpm test`
- `pnpm build`
- `pnpm playwright test`

## 6. 禁止事项

- 页面改动没有任何验证
- 只测 happy path
- 发现回归后不补测试
- 阶段完成后不做回归测试
- 阶段完成后不做新特性测试
- 涉及页面却不保留 E2E 截图和截图识别结论
- 用单个首页截图替代一级菜单和一级功能抽查
