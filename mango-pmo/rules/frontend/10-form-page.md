# 表单页 UI 规范

## 1. 页面结构

1. 表单页是下级页面，必须有返回栏。
2. 返回栏只放返回图标和当前标题。
3. 返回栏不放面包屑、状态、页面操作按钮。
4. 页面级操作放到底部悬浮操作栏。
5. 底部操作栏与内容区同宽。

## 2. 分组原则

表单不是接口字段容器，表单布局必须服务业务判断路径。

必须先分组，再排字段。每个分组都要回答一个明确业务问题，例如：

- 业务识别：当前处理对象是谁。
- 核心判断：本次操作是否合理。
- 金额信息：原始金额、操作金额、剩余金额如何对比。
- 账户信息：收款、付款、退款、结算账户是否正确。
- 凭证与说明：上传依据、填写原因、补充备注。
- 状态与结果：状态、结果、失败原因。
- 时间与审计：谁在什么时候做了什么。

## 3. 字段顺序

1. 表单不能按接口字段、数据库字段、返回字段顺序机械排列。
2. 组内字段必须有业务顺序。
3. 后面的字段要能参考前面的字段。
4. 典型顺序为：基础值 -> 操作值 -> 计算结果 -> 状态/结论。
5. 有对比、级联、依赖、计算关系的字段必须放在同一组、相邻行。

示例：

```text
可退金额 -> 退款金额 -> 剩余金额 -> 退款类型
```

## 4. Label 与布局

1. 后台表单默认使用左 label、右内容。
2. 宽度足够时，不默认使用上下结构。
3. 长文本、上传、JSON、富文本、复杂控件、移动端窄屏可以使用上下结构。
4. 多列布局不追求填满，要看字段长度和阅读重量。
5. 短字段可以并排，例如状态、类型、金额、日期、开关、短编码。
6. 长字段独占整行，例如名称、地址、开户行、URL、备注、原因、说明、上传、JSON。
7. 禁止两个长字段强行左右并排。
8. 禁止为了凑列拆散强关联字段。

## 5. 技术字段

内部 ID、规则 ID、JSON、扩展参数、报文、低频审计信息必须后置、折叠或放入独立技术信息分组。

## 6. 操作区

1. 表单主操作固定在底部操作栏。
2. 按钮顺序为：取消 -> 保存/确认。
3. 一个操作区只能有一个 `primary`。
4. 危险操作使用 `danger`，并带二次确认。
5. 提交中必须 loading，禁止重复提交。
6. 未保存离开或关闭时必须提醒确认。

## 7. 检查清单

1. 是否按业务判断分组。
2. 每组是否有明确业务依据。
3. 字段顺序是否符合业务先后。
4. 对比、级联、计算字段是否相邻。
5. 长字段是否独占整行。
6. label 是否默认左右结构。
7. 技术字段是否后置。
8. 操作区是否稳定、提交态是否完整。
