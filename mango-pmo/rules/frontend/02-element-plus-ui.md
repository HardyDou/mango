# Element Plus UI 规范

## 1. 适用范围

- 适用于 Mango 前端管理后台页面。
- 适用于基于 Mango 业务 starter 创建的业务后台页面。
- 默认基于 Vue 3、TypeScript 和 Element Plus。
- Vue 代码、Monorepo 边界和测试要求仍以对应前端规范为准。

## 2. 需求拆解

拿到页面需求后，开发前必须先提取：

- 页面类型：列表、详情、表单、弹窗、抽屉、步骤流、看板或配置页。
- 查询条件：关键字、状态、类型、时间范围、所属对象、层级对象。
- 字典和枚举：状态、类型、来源、级别、开关、审批动作等有限选项。
- 辅助数据：组织、用户、角色、租户、部门、标签、分类、上级对象等关联数据。
- 操作集合：新增、编辑、删除、启停、导入、导出、批量操作、审批动作。
- 状态集合：启用、停用、草稿、发布、成功、失败、处理中、过期、异常等业务状态。

拆解结果必须决定哪些内容使用已有组件，哪些内容需要新增页面私有组件，哪些内容需要下沉为公共组件。

## 3. 选择组件

- 字典、枚举、辅助数据和关联对象优先做成选择组件。
- 选择组件必须优先支持单选；业务存在批量筛选或多对象绑定时必须支持多选。
- 选择组件涉及远程数据时必须支持输入检索、清空、加载中、空结果和错误提示。
- 选择组件允许手输时，必须明确手输值的校验、保存语义和回显方式。
- 页面内出现两处及以上同类选择逻辑时，必须抽为组件或 composable。
- 公共选择组件不得直接依赖页面 store、路由、菜单或权限装配逻辑。
- 选择组件的 value 类型必须与后端契约一致；ID 字段按前端 ID 规则使用字符串语义。

## 4. 列表页布局

列表页必须按以下顺序组织：

1. 搜索区域。
2. 功能区域。
3. 列表区域。
4. 分页区域。
5. 弹窗、抽屉或二次确认。

搜索区域规则：

- 使用 `ElForm` 承载查询条件。
- 查询项超过一行时必须支持展开和收起。
- 主按钮顺序固定为查询、重置、展开或收起。
- 查询按钮使用 `primary`，重置按钮使用默认按钮。
- 时间范围使用 `ElDatePicker` 的 range 类型，不拆成两个无关输入框。

功能区域规则：

- 位于列表上方。
- 左侧放主操作，例如新增、导入、导出、批量删除。
- 右侧放刷新、列设置、密度切换等视图操作。
- 主操作按钮最多一个 `primary`。
- 危险操作必须使用 `danger` 并带二次确认。

列表区域规则：

- 使用 `ElTable` 或项目既有表格组件。
- 表格列必须有稳定宽度、最小宽度或溢出策略。
- ID、编码、名称、状态、时间、操作列顺序必须稳定。
- 操作列固定在右侧；按钮文案必须短且动作明确。
- 行内按钮超过三个时，低频操作必须收进 `ElDropdown`。

分页区域规则：

- 分页位于列表下方右侧。
- 页码、每页条数和总数必须由真实接口结果驱动。
- 查询条件变化后必须回到第一页。

## 5. 表单和弹窗

- 新增和编辑表单优先复用同一份表单组件。
- 复杂表单、详情核对区、确认弹窗和抽屉的信息组织必须遵守 [表单 UI 布局规范](./07-form-layout.md)。
- 表单项使用 `ElFormItem`，label 必须是业务名词，不写实现词。
- 必填、长度、格式、范围、唯一性等约束必须有前端校验；后端错误必须能回显到页面。
- 新增、修改、搜索表单必须先按业务聚集性分组，再按字段关联性排序。
- 同一业务对象、同一审批链路、同一金额/期限/文件信息必须放在同一组内。
- 级联、互相影响、前后置依赖字段必须相邻；强联动字段优先水平并列，字段较长或依赖说明较多时垂直连续。
- 搜索表单只保留高频条件；低频条件超过一行时必须放入展开区域。
- 20 个以内固定选项使用下拉选择，组件必须支持单选和多选配置。
- 超过 20 个选项、远程选项或增长型选项必须支持输入检索。
- 下拉选择必须支持清空、禁用、加载中、空结果和错误提示。
- 表单项顺序禁止按接口字段、数据库字段或开发便利性直接排列。
- 弹窗用于短表单和确认操作；抽屉用于字段多、需要上下文参照或分组编辑的场景。
- 保存按钮必须显示提交中状态，提交中禁止重复提交。
- 关闭未保存表单前必须提示用户确认。

## 6. 字体和文案层级

页面文案必须按语义分层，不按局部视觉临时改字号：

- 页面标题：18px，字重 600，用于页面主标题。
- 区块标题：16px，字重 600，用于搜索区、列表区、表单分组。
- 正文：14px，字重 400，用于表格、表单、普通说明。
- 辅助说明：13px，字重 400，用于提示、补充说明、空状态说明。
- 注释和次要信息：12px，字重 400，用于时间、来源、弱提示、表格补充字段。
- 按钮、表单控件、表格内容默认跟随 Element Plus 基础字号，不单独改小。

禁止事项：

- 禁止页面内随意使用 11px 以下字号。
- 禁止使用负字距。
- 禁止为了挤内容压缩字号；应调整布局、换行、折叠或使用提示浮层。
- 禁止用颜色替代文字说明表达关键状态。

## 7. 状态展示

- 业务状态统一使用 `ElTag` 或项目公共状态标签组件。
- 状态颜色必须来自业务语义映射，不允许页面内手写裸色值。
- 成功、启用、正常类状态使用 `success`。
- 处理中、运行中、待审批类状态使用 `primary` 或 `warning`，按业务风险选择。
- 草稿、未开始、停用、关闭类状态使用 `info`。
- 失败、异常、拒绝、删除类状态使用 `danger`。
- 同一业务状态在列表、详情、表单回显中的文案和颜色必须一致。
- 未知状态必须显示为可识别的兜底文案，禁止空白。

## 8. 空、加载和错误状态

- 页面首次加载必须有加载状态。
- 列表无数据必须使用统一空状态，不显示空白表格。
- 查询无结果和系统无数据的空状态文案必须区分。
- 接口错误必须显示可理解的错误文案，并保留重试入口。
- 删除、禁用、批量操作等高风险动作必须显示影响范围。

## 9. UI 质量门禁

页面交付前必须完成 UI 和交互走查：

- 搜索区、功能区、列表区、分页区、弹窗或抽屉边界必须清晰，间距、对齐和密度一致。
- 表单字段必须按业务聚集性分组和关联性排序；联动字段必须相邻，不按接口字段顺序堆叠。
- 主操作、次操作、危险操作和行内操作必须位置稳定、语义清晰。
- 表格列宽、溢出、固定列、空状态、加载态、错误态和分页必须完整可见。
- 弹窗和抽屉必须有明确标题、提交中状态、重复提交保护和未保存关闭确认。
- 页面不得出现文字重叠、控件挤压、按钮换行失控、表格操作列遮挡或移动端/窄屏不可用。
- 页面必须使用真实接口、配置或可替换 provider 回显数据；禁止用静态假数据证明布局完成。

## 10. 样式边界

- 优先使用 Element Plus 组件能力、项目设计变量和局部 class。
- 禁止覆盖 Element Plus 全局 class 作为页面局部方案。
- 禁止在页面中硬编码品牌色、状态色和大段间距常量。
- 新增公共样式必须说明复用场景，并放入对应公共包或主题文件。
- 卡片、搜索区、表格区和弹窗内部不得套多层视觉容器。

## 11. 可检查规则

能用 lint、脚本或 review checklist 检查的规则必须落到检查项：

- 检查 PMO preflight 是否加载本规范。
- 检查业务 starter 和 mango-cli 模板是否携带本规范。
- 检查页面新增选择组件时是否支持单选、多选或输入检索中的必要能力。
- 检查新增、修改、搜索表单是否按业务聚集性分组、关联性排序和联动相邻布局。
- 检查页面是否存在重复的字典、枚举、状态映射。
- 检查状态标签是否通过统一映射渲染，禁止页面内散写状态色。
- 检查列表页是否包含搜索区域、功能区域、列表区域和分页区域。
- 检查页面是否完成 UI 走查，并保留真实接口、截图、E2E 或手工验收证据。
- 检查页面是否仍存在静态假数据、硬编码字典、硬编码菜单、硬编码权限或 mock 接口依赖。

当前无法稳定自动判断的视觉一致性，必须在 PR review 中按本规范人工确认。

## 12. 禁止事项

- 禁止复制已有页面后长期保留重复选择逻辑、状态映射和表格操作布局。
- 禁止为单个页面创建与 Element Plus 语义冲突的控件样式。
- 禁止让页面私有样式成为事实全局规范。
- 禁止用静态选项替代必须来自接口或配置的字典、枚举和辅助数据。
- 禁止表单字段按接口返回顺序、数据库字段顺序或随手追加顺序布局。
- 禁止把“页面能打开”当作页面完成；页面必须通过真实数据、真实交互和 UI 走查验收。
