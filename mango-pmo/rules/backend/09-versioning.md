# 版本化发布规范

## 1. 适用范围

适用于存在发布态和历史版本的领域对象，例如编号规则、流程定义、模板配置、计费规则、审批规则等。

只要已发布内容会影响业务记录、外部单据、审计追踪或运行时行为，就必须按本规范处理历史版本。

## 2. 基本模型

- 版本号必须单调递增，不复用、不回退。
- 历史版本是不可变快照，发布后不得原地修改。
- 当前生效版本只能有一个。
- 草稿版本可以编辑，发布后进入生效或历史状态。

推荐状态：

| 状态 | 含义 |
|------|------|
| `DRAFT` | 草稿，可编辑，未生效 |
| `ACTIVE` | 当前生效版本 |
| `HISTORY` | 历史版本，不可编辑 |

## 3. 默认回滚策略

历史版本切换或回滚，默认采用：

1. 选择一个历史版本。
2. 复制该历史版本的完整配置，生成一个新的草稿版本。
3. 新版本号取当前最大版本号加一。
4. 按正常发布流程发布新版本。
5. 原生效版本转为历史版本，新版本成为生效版本。

示例：

```text
当前：V1 HISTORY，V2 ACTIVE
切换到 V1
结果：V1 HISTORY，V2 HISTORY，V3 ACTIVE
说明：V3 的内容复制自 V1
```

这样可以保证回滚本身也是一次可审计的发布动作。

## 4. 禁止做法

- 禁止直接把当前版本指针改回历史版本作为默认回滚方案。
- 禁止修改历史版本内容来模拟回滚。
- 禁止回退版本号。
- 禁止回滚已经生成的业务数据、流水值、单据号、外部记录。
- 禁止把发布态、历史态和草稿态混在同一条可变记录里。

## 5. 直接切指针例外

只有满足以下低风险场景时，才允许直接切换指针：

- 非业务关键配置。
- 运行环境部署指针，例如灰度、蓝绿、镜像版本。
- A/B 策略或 Feature Flag。
- 不要求严格审计。
- 紧急恢复场景，且必须补充审计记录和原因说明。

涉及业务单据、金额、权限、审批、编号、合同、外部集成结果的场景，不使用直接切指针。

## 6. 数据要求

版本主表至少需要表达：

- 业务对象标识。
- 版本号。
- 版本状态。
- 发布时间。
- 发布人。

按需要增加：

- 来源版本号，例如 `source_version`。
- 来源记录 ID，例如 `source_id`。
- 发布原因，例如 `publish_reason`。
- 当前生效标识或唯一约束，保证同一业务对象只有一个生效版本。

版本明细表必须跟随版本快照保存，不能读取可变的当前配置作为历史内容。

## 7. 测试要求

涉及版本化发布时，必须覆盖：

- 新增草稿。
- 编辑草稿。
- 发布后旧版本变历史、新版本变生效。
- 从历史版本复制生成新版本。
- 历史版本不可原地修改。
- 同一业务对象不能同时存在多个生效版本。
- 业务生成结果不因版本回滚而被回滚。
