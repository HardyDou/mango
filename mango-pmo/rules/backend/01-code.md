# 后端代码规范

## 1. 基本规则

- 代码必须可读、可测、可维护。
- 同一逻辑只保留一份实现。
- 不写无意义包装层。
- 不把临时方案当正式方案提交。

## 2. 复杂度控制

- 方法保持简短。
- 类职责保持单一。
- 一个类只解决一类问题。
- 超过当前抽象层职责的逻辑必须下沉或拆分。

## 3. 异常处理

- 只捕获明确异常。
- 捕获后必须处理或继续抛出。
- 禁止吞异常。
- 禁止直接捕获 `Exception` 或 `Throwable` 作为常规写法。

## 4. 安全要求

- 禁止硬编码密钥、口令、地址、账号。
- 禁止 SQL 拼接。
- 输入必须校验。
- 日志禁止输出敏感信息。

## 5. 质量要求

- 提交前执行 `mvn mango:check`。
- 明显重复代码必须抽取。
- 明显无用代码必须删除。

## 6. 禁止事项

- 复制代码后长期保留重复实现
- 为了通过检查而绕过规则
- 把基础设施细节直接扩散到业务层
