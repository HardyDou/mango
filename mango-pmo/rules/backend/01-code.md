# 后端代码规范

## 1. 基本规则

- 代码必须可读、可测、可维护。
- 同一逻辑只保留一份实现。
- 不写无意义包装层。
- 不把临时方案当正式方案提交。

## 2. 复杂度控制

- 方法保持简短。
- 类职责保持单一。
- 一个类只解决一类问题。
- 超过当前抽象层职责的逻辑必须下沉或拆分。

## 3. 异常处理

- 只捕获明确异常。
- 捕获后必须处理或继续抛出。
- 禁止吞异常。
- 禁止直接捕获 `Exception` 或 `Throwable` 作为常规写法。

## 4. 前置条件校验

- 业务逻辑入口必须先做前置条件校验。
- 业务前置条件统一使用 `Require`。
- 禁止在业务逻辑中直接写 `if (...) throw ...` 做前置条件校验。
- 禁止用 `if (...) return ...` 跳过非法参数处理。
- `Require` 失败统一抛业务异常。

## 5. 安全要求

- 禁止硬编码密钥、口令、地址、账号。
- 禁止 SQL 拼接。
- 输入必须校验。
- 日志禁止输出敏感信息。

## 6. 质量要求

- 提交前执行 `mvn pmd:check`、`mvn checkstyle:check` 和 `mvn mango:check`。
- 通用代码质量规则由 P3C/PMD/Checkstyle 负责，例如方法长度、类长度、复杂度、异常吞掉、空 catch、硬编码等。
- `mango:check` 只负责 Mango 项目特有规则，例如模块命名、模块边界、`api/core/starter` 职责、KV key、测试物料一致性、`module.properties`、远程适配。
- 命名规则按工具分工执行：Java 类、方法、字段命名交给 P3C/Checkstyle；P3C/Checkstyle 覆盖不到的 Mango 模块命名保留在 `mvn mango:check -Drule=naming`。
- 明显重复代码必须抽取。
- 明显无用代码必须删除。

## 7. 禁止事项

- 复制代码后长期保留重复实现
- 为了通过检查而绕过规则
- 把基础设施细节直接扩散到业务层
- 在业务逻辑里到处散写参数 `if/throw`
