# 后端代码规范

## 1. 基本规则

- 代码必须可读、可测、可维护。
- 同一逻辑只保留一份实现。
- 不写无意义包装层。
- 不把临时方案当正式方案提交。

## 2. 复杂度控制

- 方法保持简短。
- 类职责保持单一。
- 一个类只解决一类问题。
- 超过当前抽象层职责的逻辑必须下沉或拆分。

## 3. 异常处理

- 只在能补充上下文、转换协议、释放资源、降级兜底或恢复业务状态的边界捕获异常。
- 捕获后必须处理、记录并继续抛出，或转换为语义等价的业务异常；禁止 catch 后吞异常。
- 禁止直接捕获 `Exception` 或 `Throwable` 作为常规写法；框架兜底处理器除外。
- 日志必须保留原始异常堆栈。使用 `log.warn("...", ex)` 或 `log.error("...", ex)`，禁止只记录 `ex.getMessage()` 后丢失堆栈。
- 业务异常必须表达真实业务原因，例如参数非法、状态不允许、资源不存在、权限不足；禁止把 SQL、IO、网络、序列化、签名算法等系统异常伪装成无关业务异常。
- 异常包装必须保留原始 cause，例如 `new BizException(code, message, ex)` 或对应支持 cause 的异常类型；当前异常类型不支持 cause 时，必须先记录带堆栈日志，再抛出语义准确的异常。
- 禁止将 A 异常包装成误导性的 B 异常。例如数据库字段不存在不能包装成订单不存在，验签算法异常不能包装成金额不一致。
- 公共异常处理器必须对未知异常和基础设施异常打印完整堆栈，对外返回可以脱敏，但内部日志必须可定位根因。
- 第三方回调、消息消费、定时任务、异步任务等入口必须记录业务定位信息，例如通道、订单号、租户、任务编码、消息 ID；敏感字段仍按安全规则脱敏。
- 对外接口禁止暴露堆栈、SQL、密钥、token、证书、签名原文等敏感细节；内部日志禁止输出密钥、口令、完整证书和 token。

### 3.1 外部入口可诊断性

- 第三方回调、开放接口、支付查单、退款查单、对账拉取、通知投递和定时任务属于外部或异步入口，失败时必须能从日志或失败记录定位根因。
- 外部入口必须记录稳定定位字段，包括入口类型、通道编码、租户 ID、应用 ID、签约 ID、业务订单号、支付订单号、退款订单号、通道交易号、请求摘要、处理结果和耗时；没有的字段不伪造。
- 外部入口禁止只依赖控制台日志作为唯一失败证据；回调、对账拉取、通知投递等需要运营排查的链路必须有可查询的失败记录、批次记录或处理摘要。
- 外部入口捕获异常后，日志必须包含原始异常堆栈和定位字段；重新抛出的业务异常必须保留 cause，或在抛出前已记录带堆栈日志。
- 禁止把协议解析、验签、网络 IO、数据库更新、租户上下文、权限拦截等异常包装成误导性的资源不存在、金额不一致或状态不允许。
- 对外 ACK 或错误响应必须按协议脱敏；内部失败记录和日志必须保留足够定位信息，但不得保存密钥、证书、完整 token、完整签名原文和敏感报文。
- 新增外部入口时必须补充异常路径测试，覆盖解析失败、验签失败、资源不存在、状态不允许、幂等重复、下游网络失败和数据库更新失败中的适用场景。

## 4. 前置条件校验

- 业务逻辑入口必须先做前置条件校验。
- 业务前置条件统一使用 `Require`。
- 禁止在业务逻辑中直接写 `if (...) throw ...` 做前置条件校验。
- 禁止用 `if (...) return ...` 跳过非法参数处理。
- `Require` 失败统一抛业务异常。

## 5. 安全要求

- 禁止硬编码密钥、口令、地址、账号。
- 禁止 SQL 拼接。
- 输入必须校验。
- 日志禁止输出敏感信息。

## 5.1 文件引用规则

- 业务模块禁止持久化文件访问地址，包括 `url`、`previewUrl`、`downloadUrl`、对象存储直连地址、预签名 URL、临时预览地址。
- 业务表只能保存文件中心标识或引用关系，例如 `file_id`、`file_ids`、`mango-file:{id}` token、业务附件关联表。
- 业务模块需要预览、下载、归档、权限校验、派生文件处理时，必须通过 `mango-file` 按文件 ID 操作。
- 文件访问地址属于运行时派生数据，不能写入业务实体、业务扩展字段、JSON 配置或缓存作为长期数据。

## 6. 质量要求

- 提交前统一执行 `mvn mango:check -Drule=all`。
- 涉及测试设计、测试归属、有效测试判断，遵循 `mango-pmo/rules/backend/08-test.md`。
- `mango:check` 对内聚合 `mango-check`、`pmd`、`checkstyle`、`spotbugs`，是项目质量验收的唯一入口。
- 通用代码质量规则由 P3C/PMD/Checkstyle 负责，例如方法长度、类长度、复杂度、异常吞掉、空 catch、硬编码等。
- `mango:check` 是项目统一检查入口，但插件实现只负责 Mango 项目特有规则，例如模块命名、模块边界、`api/core/starter` 职责、KV key、测试物料一致性、`module.properties`、远程适配。
- `mango:check` 报告必须标识规则来源，例如 `mango-check`、`pmd`、`checkstyle`、`spotbugs`。
- 命名规则按工具分工执行：Java 类、方法、字段命名交给 P3C/Checkstyle；P3C/Checkstyle 覆盖不到的 Mango 模块命名保留在 `mvn mango:check -Drule=naming`。
- Javadoc 采用契约优先策略，不追求形式完整：
  - API、SPI、注解、配置类必须写有语义的 Javadoc。
  - public 方法暴露契约时，说明参数约束、返回语义、异常或边界。
  - 枚举值必须说明业务含义。
  - getter/setter、简单字段、显然方法不强制写 Javadoc。
  - `@author`、Javadoc 首句句号等形式规则不作为 Mango 质量规则。
- 明显重复代码必须抽取。
- 明显无用代码必须删除。

## 7. 禁止事项

- 复制代码后长期保留重复实现
- 为了通过检查而绕过规则
- 把基础设施细节直接扩散到业务层
- 在业务逻辑里到处散写参数 `if/throw`
- 在业务数据中保存文件下载、预览或对象存储访问地址
