# Mango PMO 总流程

## 1. 定位

- `mango-pmo` 是 Mango 的唯一规范源。
- `mango-docs` 只放设计文档、Sprint 计划、交付记录。
- 规范、流程、角色、Skill 一律放 `mango-pmo`。

## 2. 目标

- AI 能基于 Mango 完成一句话需求。
- 产出可开发、可验证、可发布的结果。
- 降低歧义、返工、越权、遗漏。

## 3. 标准流程

### 3.0 业务消费 Mango 问题处理

业务开发使用 Mango 时发现疑似 Mango 框架、starter、CLI、模板、前端包或发布物料问题，必须先按 [Mango Issue 登记 Runbook](./07-mango-issue-runbook.md) 归因、登记和回写任务记录。

本节只保留流程入口；Issue 触发条件、登记字段、阻塞处理和任务结束提醒统一维护在 `rules/07-mango-issue-runbook.md`。

### 3.0.1 PMO 加载原则

- preflight 只加载本次任务必须遵守的最小规范集。
- `rules/00-dev-flow.md` 和 `rules/03-ai-coding-redlines.md` 是正式任务基线。
- `rules/01-delivery-contract.md` 只在正式需求、设计、开发、测试、验收、发布、PR 或交付台账任务中加载。
- 领域规范只在路径或任务关键词命中时加载；禁止因为“前端”“后端”泛词一次性加载整套规范。
- 简单问答、只读定位、状态查看、日志查看、目录查看、worktree 列表不触发正式流程。
- 规范、设计文档、交付记录等不影响服务代码的治理改动，可在 `main` 直接修改提交；仍需执行 PMO preflight。

PMO preflight 的首要目的不是审批命令，而是在正式交付前刷新 Agent 对本项目规范的记忆。Agent 必须先识别用户意图和规范遗漏风险，再决定是否进入 PMO。

Agent 在长会话和复杂任务中容易出现以下问题：

- 忘记本项目已有规范，按通用经验或旧习惯执行。
- 想不到本次任务对应的前端、后端、数据库、测试、发布、文档或工作区规则。
- 只关注眼前报错，忽略权限、租户、真实数据、验证证据和发布影响。
- 顺手扩大范围、修历史问题，或把非本次问题混入当前交付。
- 用“能编译、能启动、能打开页面”替代真实验收。

判定原则：

- 先判断用户意图：是交付变更、方案/验收、发布/提交，还是查看、同步、定位、本地运维。
- 再判断风险：是否存在规范遗漏风险，或会影响受版本控制文件、服务行为、接口/数据库/测试/构建、发布物料、长期规范或交付结论。
- 用户意图是“交付一个变更”，或风险会影响可交付结果时执行。
- 用户意图只是“查看状态、同步仓库、整理本地环境”，且风险只限本地状态时不执行。
- 如果不触发的操作过程中发现需要人工修改文件、解决冲突、修复问题或形成交付结论，先停止当前操作，再按实际任务执行 preflight。

意图与风险判定：

| 用户意图 | 风险特征 | 是否执行 |
|----------|----------|----------|
| 查看、解释、定位、列举 | 只读，不改变仓库和运行态 | 不执行 |
| 纯同步仓库 | 只执行 fast-forward/fetch，不解决冲突、不形成验证结论 | 不执行 |
| 本地运维 | 只影响本机进程、缓存、端口、未跟踪临时文件 | 不执行 |
| 编码、修复、重构、配置变更 | 需要召回开发、测试、边界、工作区规范 | 执行 |
| 设计、测试方案、验收、发布、提交、PR | 需要召回交付、证据、发布、PR 规范 | 执行 |
| 规范、流程、Agent 入口治理 | 需要召回规范唯一源和文档归档规则 | 执行 |
| 冲突处理、失败修复、回滚选择 | 需要人工判断并改变代码或历史 | 执行 |

以下情况必须执行：

- 按设计文档、plan、Sprint 计划或交付台账进行开发、验证、发布或治理。
- 人工新增或修改代码、接口、数据库、测试、前端页面、构建配置。
- 新增或修改设计文档、Sprint 计划、交付记录、规范或 Agent 入口。
- 做架构设计、技术方案、测试方案、验收、发布、提交或 PR。
- 处理 `git pull`、`rebase`、`merge` 产生的冲突，或为同步后的失败检查做修复。
- 用户明确要求“按 PMO / 按计划 / 按设计文档 / 按交付台账”推进。

以下情况不需要执行：

- 简单问答、概念解释、使用说明。
- 只读定位文件、类、接口、配置或快速查看模块现状。
- 查看 git 状态、目录结构、日志、报错片段，但不进入修复或交付。
- 执行纯仓库同步命令，例如 `git fetch`、`git pull --ff-only`、`git remote update`，且不人工编辑文件、不解决冲突、不做交付验证。
- 查看或刷新 GitHub Issue、PR、分支、tag、release 列表，但不创建、修改或发布。
- 安装依赖、清理本地缓存、停止本地进程、查看端口占用等本地运维操作，且不修改受版本控制文件。
- 用户明确要求“先快速看看 / 暂不进入 PMO 流程”。

### 3.0.2 工作区策略

PMO preflight 必须输出本次任务的工作区策略：

- `worktree-required`：必须使用任务专用 Git worktree 和任务分支。
- `main-direct-allowed`：可在主工作区直接修改并提交。
- `needs-human-check`：影响范围不足，必须先确认路径或任务边界。

以下改动必须使用任务 worktree：

- 代码、接口、数据库、测试、前端页面、构建配置。
- `mango/**`、`mango-ui/**`、`mango-business-starter/**`、`scripts/**`、`.github/**`。
- `package.json`、`pnpm-lock.yaml`、`pom.xml`、migration、`src/**`。
- 发布脚本、启动脚本、CLI、starter、模板和发布物料。

以下改动可在主工作区直接提交：

- `mango-pmo/**` 规范、流程、Agent 定义和 PMO 工具治理。
- `mango-docs/**` 设计文档、Sprint 计划、交付记录和历史材料。
- `AGENTS.md`、`CLAUDE.md`、`GEMINI.md` 入口路由。

只要同一任务同时触及必须 worktree 的文件，整项任务必须使用 worktree。

同一任务或同一 PR 在本地只能对应一个开发 worktree。验收返工、Review 修改、CI 修复、PR 门禁修复和交付台账未完成项属于当前任务返工时，必须复用原任务 worktree 和原任务分支。

### 3.1 需求

必须完成：

1. 明确目标
2. 明确范围
3. 明确验收标准
4. 为关键对象、关键流程、业务规则、页面和主要动作建立可追踪验收项
5. 按 [交付契约与设计说明规则](./01-delivery-contract.md) 明确交付契约

产出：

- 需求说明

### 3.2 设计

必须完成：

1. 明确影响模块
2. 明确接口变化
3. 明确数据变化
4. 明确测试范围
5. 按 [测试用例与自动化工程化流程](./09-test-case-automation-flow.md) 建立测试用例候选、自动化层级和执行策略
6. 按 [交付契约与设计说明规则](./01-delivery-contract.md) 建立交付台账

产出：

- 设计说明

### 3.3 开发

必须完成：

1. 确认交付契约、设计说明和交付台账已建立
2. 从 `main` 新建独立 Git worktree 和任务分支
3. 在任务 worktree 内按规范编码
4. 新增代码遵守最新规则
5. 不扩大历史债务

禁止：

- 直接在主工作区或 `main` 分支上做代码、接口、数据库、测试、前端页面或构建配置改动。
- 多个任务共用同一个开发 worktree。

例外：

- 只修改规范、流程、设计文档、交付记录、Agent 入口或 PMO 临时复盘，且不影响服务代码、接口、数据库、测试、前端页面或构建配置时，可在 `main` 直接提交。

产出：

- 代码
- 必要测试

### 3.4 验证

必须完成：

1. 运行对应检查命令
2. 验证改动范围
3. 记录未完成项和风险
4. 按交付台账逐项验证
5. 按测试用例优先级执行自动化或手工验收，并沉淀测试结果基线

产出：

- 验证结果

### 3.4.1 任务中发现新问题

适用于 Mango 开发者、Mango 维护者和业务开发者在开发、联调、回归测试、验收或发布验证中发现的新问题。

必须先判断问题归属：

1. 如果问题由本次任务改动引入，必须纳入当前任务直接修复，并补充对应回归验证。
2. 如果问题不是本次任务改动引入，必须登记 Issue，记录复现路径、影响范围、证据和建议优先级，然后回到当前任务主线。
3. 如果无法判断是否由本次任务引入，必须对比主分支、发布物料或任务前基线；仍无法判断时，按风险记录 Issue，并在交付报告中说明未归因风险。

禁止：

- 把非本次任务引入的问题顺手扩展成当前任务范围。
- 因为发现历史问题而中断当前任务主线。
- 未登记 Issue 就把已确认的非本次问题留作口头记录。
- 用 Issue 逃避本次任务引入的缺陷修复。

### 3.4.2 任务返工与验收缺陷修复

验收、Review、CI 或 PR 检查中发现的问题，必须先判断是否属于当前任务或当前 PR。

必须执行：

1. 如果问题由当前任务或当前 PR 引入，属于同一任务返工，必须回到原任务 worktree 和原任务分支修复。
2. 同一任务返工包括验收发现的 bug、Review 修改意见、CI 失败、PR 门禁失败和交付台账未完成项。
3. 修复前必须先执行 `git worktree list`，确认是否已有当前任务分支或 PR 分支对应的 worktree。
4. 原 worktree 不可用时，必须优先基于原任务分支重建 worktree，并在交付报告中说明原因。
5. 如果问题不是当前任务引入，必须登记 Issue 或 PR 评论，并等待用户确认是否作为新任务处理。

禁止：

- 同一任务返工时新建无关联 worktree。
- 同一任务返工时新建无关联修复分支。
- 将当前任务引入的缺陷拆成新任务来规避原任务验收。

### 3.5 提交

必须完成：

1. 在任务 worktree 内完成验证
2. 提交前先将最新 `main` 合并到任务分支，并处理冲突
3. 合并 `main` 后重新执行受影响验证
4. 提交代码
5. Push 任务分支
6. 使用 `gh` 创建 PR
7. 说明改动范围
8. 说明验证结果
9. 说明遗留问题
10. 说明交付台账总项、完成项和例外项

PR 要求：

- 禁止绕过 PR 直接把任务分支合并到 `main`。
- PR 创建后必须保留任务 worktree，直到 PR 已合并。
- PR 合并后，必须使用开发环境规范中的 worktree 删除脚本清理任务 worktree。

产出：

- Commit
- 已创建 PR

## 4. 文档放置规则

### 4.1 `mango-pmo`

只放：

- 规范
- 流程
- Agent 定义
- Skill 约束

### 4.2 `mango-docs`

只放：

- 架构设计
- Sprint 计划
- 交付记录
- 历史设计

禁止：

- 在 `mango-docs` 新增规范文件
- 在 `mango-docs` 维护规则入口
- 在 `mango-docs` 写“必须遵守”的长期规则

## 5. 角色分工

### 5.1 使用者定义

| 使用者 | 定义 | 主要边界 |
|--------|------|----------|
| Mango 维护者 | 维护 Mango 主仓、发布物料、PMO 规范、CLI、starter、框架能力和公共模块的人或 Agent | 可以修改 Mango 框架、模板、规范和发布流程；必须保证发布物料彼此兼容 |
| Mango 开发者 | 在 Mango 主仓内开发框架能力、平台模块、公共组件、CLI 和 starter 的人或 Agent | 面向可复用能力开发；不得把单一业务项目逻辑写入 Mango 通用模块 |
| 业务开发者 | 在企业业务项目内基于 Mango 发布物料开发业务模块的人或 Agent | 依赖 Mango CLI、starter、前后端包和业务 PMO 基线开发；不得直接修改 Mango 主仓来交付单一业务需求 |

Mango 维护者和 Mango 开发者面向框架通用性负责；业务开发者面向企业业务交付负责。

### 5.2 Agent 角色

| 角色 | 负责 |
|------|------|
| PM Agent | 需求说明、范围、验收标准 |
| Tech Lead Agent | 设计、模块边界、接口、任务拆分 |
| Dev Agent | 开发、测试、提交 |
| QA Agent | 测试范围、执行结果 |
| PMO Agent | 规范、流程、资产治理 |

## 6. 执行原则

- 规则优先于习惯
- 新规则只写一次，只放 `mango-pmo`
- 新代码遵守新规则
- 老代码按模块渐进重构
- 设计文档不作为规范源

## 7. 禁止事项

- 没有范围和验收标准就直接开发
- 没有交付契约和设计说明就直接开发
- 在设计文档里定义长期规则
- 同一规则在多个目录维护
- 明知已有规范仍写第二份
