# Mango PMO 总流程

## 1. 定位

- `mango-pmo` 是 Mango 的唯一规范源。
- `mango-docs` 只放设计文档、Sprint 计划、交付记录。
- 规范、流程、角色、Skill 一律放 `mango-pmo`。

## 2. 目标

- AI 能基于 Mango 完成一句话需求。
- 产出可开发、可验证、可发布的结果。
- 降低歧义、返工、越权、遗漏。

## 3. 标准流程

### 3.1 需求

必须完成：

1. 明确目标
2. 明确范围
3. 明确验收标准

产出：

- 需求说明

### 3.2 设计

必须完成：

1. 明确影响模块
2. 明确接口变化
3. 明确数据变化
4. 明确测试范围

产出：

- 设计说明

### 3.3 开发

必须完成：

1. 按规范编码
2. 新增代码遵守最新规则
3. 不扩大历史债务

产出：

- 代码
- 必要测试

### 3.4 验证

必须完成：

1. 运行对应检查命令
2. 验证改动范围
3. 记录未完成项和风险

产出：

- 验证结果

### 3.5 提交

必须完成：

1. 提交代码
2. 说明改动范围
3. 说明验证结果
4. 说明遗留问题

产出：

- Commit 或 PR

## 4. 文档放置规则

### 4.1 `mango-pmo`

只放：

- 规范
- 流程
- Agent 定义
- Skill 约束

### 4.2 `mango-docs`

只放：

- 架构设计
- Sprint 计划
- 交付记录
- 历史设计

禁止：

- 在 `mango-docs` 新增规范文件
- 在 `mango-docs` 维护规则入口
- 在 `mango-docs` 写“必须遵守”的长期规则

## 5. 角色分工

| 角色 | 负责 |
|------|------|
| PM Agent | 需求说明、范围、验收标准 |
| Tech Lead Agent | 设计、模块边界、接口、任务拆分 |
| Dev Agent | 开发、测试、提交 |
| QA Agent | 测试范围、执行结果 |
| PMO Agent | 规范、流程、资产治理 |

## 6. 执行原则

- 规则优先于习惯
- 新规则只写一次，只放 `mango-pmo`
- 新代码遵守新规则
- 老代码按模块渐进重构
- 设计文档不作为规范源

## 7. 禁止事项

- 没有范围和验收标准就直接开发
- 在设计文档里定义长期规则
- 同一规则在多个目录维护
- 明知已有规范仍写第二份
