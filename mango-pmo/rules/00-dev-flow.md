# Mango PMO 总流程

## 1. 定位

- `mango-pmo` 是 Mango 的唯一规范源。
- `mango-docs` 只放设计文档、Sprint 计划、交付记录。
- 规范、流程、角色、Skill 一律放 `mango-pmo`。

## 2. 目标

- AI 能基于 Mango 完成一句话需求。
- 产出可开发、可验证、可发布的结果。
- 降低歧义、返工、越权、遗漏。

## 3. 标准流程

### 3.0 业务消费 Mango 问题处理

业务开发者使用 Mango 时发现疑似 Mango 框架问题，必须先归因，不得直接在业务项目中修改 Mango 框架、starter、CLI、模板或前端包源码。

确认由 Mango 框架缺陷引起时：

1. 必须向 Mango 提交 Issue，记录 Issue 地址、复现路径、影响范围、证据、当前 Mango 版本和建议优先级。
2. 必须在当前业务任务记录中登记 Issue 地址和阻塞状态。
3. 业务侧不得私自修复 Mango 缺陷；只允许采用不破坏业务架构且不掩盖缺陷的业务侧替代方案。
4. 如果 Mango 缺陷阻塞当前业务模块，且没有同等质量的业务侧替代方案，必须告知开发者具体原因，暂停当前模块开发。
5. 当前任务全部结束时，必须再次提醒开发者根据 Issue 结论决定升级 Mango、更换方案或调整业务计划。

非缺陷类建议也必须通过 Mango Issue 提交，并标记为建议，不得伪装成业务缺陷。

### 3.0.1 PMO 加载原则

- preflight 只加载本次任务必须遵守的最小规范集。
- `rules/00-dev-flow.md` 和 `rules/03-ai-coding-redlines.md` 是正式任务基线。
- `rules/01-delivery-contract.md` 只在正式需求、设计、开发、测试、验收、发布、PR 或交付台账任务中加载。
- 领域规范只在路径或任务关键词命中时加载；禁止因为“前端”“后端”泛词一次性加载整套规范。
- 简单问答、只读定位、状态查看、日志查看、目录查看、worktree 列表不触发正式流程。
- 规范、设计文档、交付记录等不影响服务代码的治理改动，可在 `main` 直接修改提交；仍需执行 PMO preflight。

### 3.1 需求

必须完成：

1. 明确目标
2. 明确范围
3. 明确验收标准
4. 按 [交付契约与设计说明规则](./01-delivery-contract.md) 明确交付契约

产出：

- 需求说明

### 3.2 设计

必须完成：

1. 明确影响模块
2. 明确接口变化
3. 明确数据变化
4. 明确测试范围
5. 按 [交付契约与设计说明规则](./01-delivery-contract.md) 建立交付台账

产出：

- 设计说明

### 3.3 开发

必须完成：

1. 确认交付契约、设计说明和交付台账已建立
2. 从 `main` 新建独立 Git worktree 和任务分支
3. 在任务 worktree 内按规范编码
4. 新增代码遵守最新规则
5. 不扩大历史债务

禁止：

- 直接在主工作区或 `main` 分支上做代码、接口、数据库、测试、前端页面或构建配置改动。
- 多个任务共用同一个开发 worktree。

例外：

- 只修改规范、流程、设计文档、交付记录、Agent 入口或 PMO 临时复盘，且不影响服务代码、接口、数据库、测试、前端页面或构建配置时，可在 `main` 直接提交。

产出：

- 代码
- 必要测试

### 3.4 验证

必须完成：

1. 运行对应检查命令
2. 验证改动范围
3. 记录未完成项和风险
4. 按交付台账逐项验证

产出：

- 验证结果

### 3.4.1 任务中发现新问题

适用于 Mango 开发者、Mango 维护者和业务开发者在开发、联调、回归测试、验收或发布验证中发现的新问题。

必须先判断问题归属：

1. 如果问题由本次任务改动引入，必须纳入当前任务直接修复，并补充对应回归验证。
2. 如果问题不是本次任务改动引入，必须登记 Issue，记录复现路径、影响范围、证据和建议优先级，然后回到当前任务主线。
3. 如果无法判断是否由本次任务引入，必须对比主分支、发布物料或任务前基线；仍无法判断时，按风险记录 Issue，并在交付报告中说明未归因风险。

禁止：

- 把非本次任务引入的问题顺手扩展成当前任务范围。
- 因为发现历史问题而中断当前任务主线。
- 未登记 Issue 就把已确认的非本次问题留作口头记录。
- 用 Issue 逃避本次任务引入的缺陷修复。

### 3.5 提交

必须完成：

1. 在任务 worktree 内完成验证
2. 提交前先将最新 `main` 合并到任务分支，并处理冲突
3. 合并 `main` 后重新执行受影响验证
4. 提交代码
5. Push 任务分支
6. 使用 `gh` 创建 PR
7. 说明改动范围
8. 说明验证结果
9. 说明遗留问题
10. 说明交付台账总项、完成项和例外项

PR 要求：

- 禁止绕过 PR 直接把任务分支合并到 `main`。
- PR 创建后必须保留任务 worktree，直到 PR 已合并。
- PR 合并后，必须使用开发环境规范中的 worktree 删除脚本清理任务 worktree。

产出：

- Commit
- 已创建 PR

## 4. 文档放置规则

### 4.1 `mango-pmo`

只放：

- 规范
- 流程
- Agent 定义
- Skill 约束

### 4.2 `mango-docs`

只放：

- 架构设计
- Sprint 计划
- 交付记录
- 历史设计

禁止：

- 在 `mango-docs` 新增规范文件
- 在 `mango-docs` 维护规则入口
- 在 `mango-docs` 写“必须遵守”的长期规则

## 5. 角色分工

### 5.1 使用者定义

| 使用者 | 定义 | 主要边界 |
|--------|------|----------|
| Mango 维护者 | 维护 Mango 主仓、发布物料、PMO 规范、CLI、starter、框架能力和公共模块的人或 Agent | 可以修改 Mango 框架、模板、规范和发布流程；必须保证发布物料彼此兼容 |
| Mango 开发者 | 在 Mango 主仓内开发框架能力、平台模块、公共组件、CLI 和 starter 的人或 Agent | 面向可复用能力开发；不得把单一业务项目逻辑写入 Mango 通用模块 |
| 业务开发者 | 在企业业务项目内基于 Mango 发布物料开发业务模块的人或 Agent | 依赖 Mango CLI、starter、前后端包和业务 PMO 基线开发；不得直接修改 Mango 主仓来交付单一业务需求 |

Mango 维护者和 Mango 开发者面向框架通用性负责；业务开发者面向企业业务交付负责。

### 5.2 Agent 角色

| 角色 | 负责 |
|------|------|
| PM Agent | 需求说明、范围、验收标准 |
| Tech Lead Agent | 设计、模块边界、接口、任务拆分 |
| Dev Agent | 开发、测试、提交 |
| QA Agent | 测试范围、执行结果 |
| PMO Agent | 规范、流程、资产治理 |

## 6. 执行原则

- 规则优先于习惯
- 新规则只写一次，只放 `mango-pmo`
- 新代码遵守新规则
- 老代码按模块渐进重构
- 设计文档不作为规范源

## 7. 禁止事项

- 没有范围和验收标准就直接开发
- 没有交付契约和设计说明就直接开发
- 在设计文档里定义长期规则
- 同一规则在多个目录维护
- 明知已有规范仍写第二份
